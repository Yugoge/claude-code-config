  5 + ## Bug #60
  6 + 
  7 + **日期**: 2026-03-25
  8 + **组件**: happy-cli (sessionProtocolMapper.ts)
  9 + **严重性**: High
 10 + **状态**: 未修复
 11 + **平台**: Web, 所有 session
 12 + 
 13 + ### 问题
 14 + 
 15 + 使用 slash command（如 `/dev`、`/review`）后，折叠块（WrappedEventBlock）完全不显示。Command prompt 内容泄露为普通 user message 气泡。每次 session 重启都会重复出现。
 16 + 
 17 + ### 根因分析
 18 + 
 19 + **数据流**：Claude SDK → JSONL → sessionProtocolMapper → WebSocket → happy-app
 20 + 
 21 + 问题在 `sessionProtocolMapper.ts` 的 `isSkillPrompt` 检测逻辑。该逻辑依赖 `pendingSkillCommandUuid` 在 skill prompt 到达时已被设置，但存在 **竞态条件**：
 22 + 
 23 + 1. `<command-message>` 被 SDK stream 发出 → mapper 处理，设置 `pendingSkillCommandUuid` + `pendingSkillCommandName`，然后调用 `closeTurn()` 关闭当前 turn
 24 + 2. Skill prompt（`isMeta=true`）不经 SDK stream 发出，由 JSONL meta-scanner 异步读取转发
 25 + 3. 如果 meta-scanner 读取 skill prompt 的时机早于 SDK stream 发出 command-message，`pendingSkillCommandUuid` 尚未设置 → `isSkillPrompt = false` → skill prompt 泄露为普通 user text
 26 + 
 27 + **关键竞态**：`onSessionFound` 启动 meta-scanner（`sendExisting: false`），但 meta-scanner 的 `createSessionScanner` 是异步的（`readFile`）。在 `for await` 循环中，command-message 和 meta-scanner 的读取在同一 event loop 上竞争。
 28 + 
 29 + **JSONL 结构**（实际观察）：
 30 + - Command-message：`type=user`, string content with `<command-message>` tags, **无 isMeta**, `uuid=X`
 31 + - Skill prompt：`type=user`, array content, `isMeta=true`, `parentUuid=X`
 32 + 
 33 + SDK stream 只发出 command-message（无 isMeta），不发出 skill prompt（isMeta=true）。meta-scanner 只转发 isMeta=true 的消息。
 34 + 
 35 + ### 错误修复记录（minimax commit `db00e4b0`，已 revert `cae912a9`）
 36 + 
 37 + minimax 的修复添加了 `isMetaOnlyPath` 条件和 `processedSkillPrompts` 去重，但存在致命缺陷：
 38 + 
 39 + 1. **在 `closeTurn()` 中清空 `pendingSkillCommandUuid` 和 `pendingSkillCommandName`** — `<command-message>` 处理时先设置 pending 再调用 `closeTurn()`，导致 pending 状态被立即清空，skill prompt 到达时永远看不到
 40 + 2. **`isMetaOnlyPath` 条件逻辑矛盾** — 要求 `pendingSkillCommandName !== undefined && pendingSkillCommandUuid === undefined`，但这两个字段总是同时设置/清除，此条件永远为 false
 41 + 3. **引入回归** — 破坏了原本在 Scenario A（SDK stream 先发 command-message）下正常工作的路径
 42 + 
 43 + ### 正确修复方向
 44 + 
 45 + 需要解决的核心问题是 `closeTurn()` 调用位置。在 `<command-message>` 处理中（line ~613），`closeTurn()` 在设置 pending 字段之后调用。修复选项：
 46 + 
 47 + 1. **调换顺序**：先 `closeTurn()` 再设置 pending 字段（最简单）
 48 + 2. **不在 `closeTurn()` 中清空 pending 字段**（保持现有行为，pending 字段在被消费时清空）
 49 + 3. **对 Scenario B（竞态）额外处理**：当 skill prompt 先于 command-message 到达时，需要缓存或延迟处理
