## Edge Cases & Risks

1. **safe_add removal breaks sync_changes()**: If any code path in sync_changes calls
   safe_add for a purpose other than pre-staging for safe_commit, removing safe_add will
   silently skip staging. Dev must trace all callers of safe_add (found: only line 435)
   before deletion.

2. **session-git-init.sh guard race**: `git rev-list --count HEAD` fails on an empty repo
   (no commits yet) with exit code 128. The guard must treat exit code 128 as "zero commits"
   (allow the Initial commit), not as an error that aborts the hook.

3. **copytruncate logrotate on append-only logs**: checkpoint-core.sh appends via
   `>>`. `copytruncate` is safe for append-mode writers; no postrotate SIGHUP needed.

4. **clean.md rollback documentation**: The current rollback says `git reset --hard <sha>`.
   After the fix, the checkpoint is a detached ref, not a HEAD ancestor. Rollback is
   `git checkout refs/checkpoints/<branch> -- <path>` for file-level or
   `git reset --hard refs/checkpoints/<branch>` for full tree (destructive). Dev must
   update both the rollback docs and the context-recording block.

5. **checkpoint-prune.sh ref rewrite safety**: `git update-ref refs/checkpoints/<branch> <new-tip>`
   must be guarded to confirm new-tip is an ancestor of the current ref tip to avoid
   accidentally losing commits.

6. **install-checkpoint-refspec.sh duplicate guard**: `git config --get-all` may show multiple
   entries; the script must grep for the exact refspec string before adding.
