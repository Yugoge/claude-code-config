# BA Specification: Auto-Commit/Checkpoint System v2 — refs/checkpoints Migration

**Request ID**: dev-20260416-063500
**Created**: 2026-04-16T06:35:00Z

## Goal

Refactor the auto-commit/checkpoint system so that all automated snapshot commits are written
to `refs/checkpoints/<sanitized-branch>` instead of branch HEAD, ensuring `git blame` on every
line points to a real semantic commit and branch history is never polluted by auto-snapshots.

## Context

The current system uses 7 hooks/scripts that each call `git commit` directly onto the working
branch HEAD. Every automated snapshot (PostToolUse threshold, Stop hook, fswatch daemon, manual
/checkpoint, /push pre-flight) creates a commit indistinguishable from real work, poisoning
`git log`, `git blame`, and PR history. The v2 design extracts a shared `checkpoint-core.sh`
library that uses `git commit-tree` + CAS `git update-ref` to write commits only to a dedicated
`refs/checkpoints/<branch>` ref, never touching HEAD.

## Edge Cases and Risks

1. **Empty repo**: `git rev-parse HEAD` fails; must use `git read-tree --empty` and omit
   `-p <parent>` from `git commit-tree`. CAS `update-ref` must use empty-string old-value.
2. **Detached HEAD**: `git branch --show-current` returns empty; fall back to
   `refs/checkpoints/detached-<short-sha>`.
3. **Branch name with `/`**: `tr '/' '-'` applied before constructing ref; avoids
   git D/F conflict between ref and directory (e.g., `feature/x` → `refs/checkpoints/feature-x`).
4. **Concurrent hook invocations**: Two Stop hooks may fire simultaneously; CAS retry loop
   handles this; one will win, loser retries up to 5 times.
5. **SIGKILL residue**: Temp index files may persist if hook is killed; 60-minute sweep on
   each invocation prevents accumulation.
6. **`denyNonFastForwards` remote**: Push MUST NOT use `-f`; CAS guarantees the ref chain is
   always fast-forward (each new commit has prior ref as parent).
7. **Double-commit risk from stop hook chain**: `stop-git-commit.sh` runs before
   `auto-commit.sh`; both will call `write_checkpoint`. CAS + idempotency guard (tree
   unchanged = skip) ensures at most one new commit when tree is identical across two
   invocations.
8. **`smart-checkpoint.sh` deletion**: Verify it is not listed as a hook in
   `settings.json`/`settings.local.json` before deletion. If referenced, convert instead of
   deleting.

## Acceptance Criteria

### AC1: No HEAD pollution from auto-snapshot
- GIVEN a scratch git repo with 15 files written by PostToolUse tool calls
- WHEN the PostToolUse threshold fires
- THEN `git log HEAD` is unchanged AND `git log refs/checkpoints/master` has exactly 1 commit

### AC2: Full working-tree capture
- GIVEN a repo with staged, modified, deleted, and chmod-only changes
- WHEN `write_checkpoint` is called
- THEN the checkpoint tree reflects all those changes (verify via `git diff <checkpoint>..HEAD`)

### AC3: Empty repo bootstrap
- GIVEN a git repo with no commits (`git log` fails with "does not have any commits")
- WHEN `write_checkpoint` is called
- THEN the checkpoint ref is created as a root commit (no parent) and exits 0

### AC4: Detached HEAD fallback
- GIVEN `git checkout --detach && <edit files>`
- WHEN `write_checkpoint` is called
- THEN checkpoint is written to `refs/checkpoints/detached-<short-sha>`

### AC5: Branch slash sanitization
- GIVEN current branch is `feature/x`
- WHEN `write_checkpoint` is called
- THEN checkpoint is on `refs/checkpoints/feature-x` with no D/F conflict errors

### AC6: Concurrent invocations
- GIVEN 10 parallel invocations of `write_checkpoint` against the same repo
- WHEN all complete
- THEN the ref has exactly 10 linear commits AND `git fsck --unreachable` shows no new
  dangling commits

### AC7: Push behavior
- GIVEN `/push` invoked on a clean working tree
- THEN push succeeds normally
- GIVEN `/push` invoked on a dirty working tree
- THEN command exits non-zero with explicit message "commit first" (no auto-commit created)

### AC8: Stop hook idempotency
- GIVEN the stop hook chain fires (stop-git-commit.sh then auto-commit.sh both run)
- WHEN tree state is identical in both invocations
- THEN at most ONE new commit appears on the checkpoint ref (idempotency guard skips duplicate)
