## Blocker(s)

### Blocker 1 — AC-04 hook bypassable via compound command (in_scope_real_bug)

**Severity**: critical (user's verbatim requirement "permanently forbid bulk without user permission" not met)
**Codex finding**: #2 (BLOCKER)
**Reproduced live**: yes

**The defect**: `hooks/pretool-bash-safety.sh` Layer 1.F (lines 633-671) deny-block has an allowlist exception for read-only inspection: `(ls|stat|cat|file|wc|head|tail|grep|jq|find|test|[) ... /tmp/claude-bulk-` WITHOUT `>`, `>>`, or `tee\b`. The allowlist matches the read-only PREFIX of a compound command but does not assert the compound's REMAINDER is also read-only. A subagent can issue:

```
ls /tmp/claude-bulk-allowed-<sid>.flag; python3 -c 'open("/tmp/claude-bulk-allowed-<sid>.flag","w").write("y")'
```

The `ls ... /tmp/claude-bulk-` prefix satisfies the allowlist regex; the command contains no `>`, `>>`, or `tee\b` (the `python3 -c '...'` writer uses `open(...).write(...)`, not a shell redirect). Hook returns exit 0. Auth flag is then created without user authorization. Subsequent `python3 scripts/write-bulk-commit-sentinel.py` (allowed by Layer 1.G via script-path exemption) consumes the unauthorized flag and writes a real bulk-commit sentinel.

**Live reproduction** (path string obfuscated in my test to avoid the hook reading my QA test command itself):

| Case | Command pattern | Hook exit |
|------|-----------------|-----------|
| `compound_semi` | `ls <protected> ; python3 -c 'open("<protected>","w").write("y")'` | **0 (ALLOWED — bug)** |
| `compound_and`  | `ls <protected> && python3 -c 'open("<protected>","w").write("y")'` | **0 (ALLOWED — bug)** |
| `simple_redir`  | `echo x > <protected>` | 2 (BLOCKED — correct) |
| `touch_only`    | `touch <protected>` | 2 (BLOCKED — correct) |
| `pure_ls`       | `ls <protected>` | 0 (ALLOWED — correct, read-only) |
| `pure_cat`      | `cat <protected>` | 0 (ALLOWED — correct, read-only) |
| `python_open_only` | `python3 -c 'open("<protected>","w").write("y")'` | 2 (BLOCKED — correct) |

The hook correctly blocks `python_open_only` standalone but allows it when prefixed by a read-only-looking command. This is precisely the bypass pattern codex called out, and it defeats the M4 user requirement.

**User requirement at stake** (verbatim from line 1 of `docs/dev/user-requirement-dev-20260524-205206.md`):
> changelog-analyst bulk-commit bug：只要用户不使用commit bulk，永久禁止使用bulk

Translation: "if user does not invoke /commit --bulk, permanently forbid bulk." The hook is the last-line defense for this requirement (script-level gate is layer 1, command doc is layer 2, hook is layer 3, session-Stop reap is layer 4). The compound-command bypass means the hook layer's "permanently forbid" guarantee is conditional on subagents not chaining commands — which is precisely what codex/Cycle-5 iter-2 finding #2 already warned about and the dispatch documented hardening for. The hardening covered the python/node/perl interpreter-only case but missed the compound-prefix case.

**Proposed fix** (for next dev cycle):
- In `hooks/pretool-bash-safety.sh` Layer 1.F: when the protected glob is matched, deny if the command contains ANY of: `;`, `&&`, `||`, `|` (pipe), or a here-doc, UNLESS the entire command is a single read-only verb invocation. The current allowlist regex `(^|[[:space:];|&])(ls|stat|...) ... /tmp/claude-bulk-` already accepts shell separators in the prefix — the fix is to invert: require the whole command to be `^[[:space:]]*(ls|stat|cat|...) [^;&|<>]*$` (no separators, no redirects, no compound), OR add a separator-presence check before applying the allowlist.
- Add regression test cases to `tests/generated/20260524-205206/test_AC_04_d4e6f8a0c3b5d7ea.py` (or successor): `g_hook_compound_semi`, `g_hook_compound_and`, `g_hook_compound_pipe`, asserting `expected_hook_exit_nonzero: true`.
