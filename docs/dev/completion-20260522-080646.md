### A — bash-safety executable-context detection ✅ PASS (QA iter3)

**Problem**: Four rules in `hooks/pretool-bash-safety.sh` applied `grep -qE` to the full raw command string, blocking legitimate commands when prompt bodies or string arguments contained tokens like `killall`, `kill`, `rm`. Confirmed incident: `codex exec "…killall…claude…"` was blocked during cycle 20260519-211515.

**Solution**: New `hooks/lib/bash_context_strip.py` (stdlib-only, ~520 lines) implements `strip_non_executable_contexts()` with 5 passes: control-operator split, sh/bash -c recursion, heredoc consumer classification, argument stripping (with `DANGER_COMMANDS` exclusion — danger command args are preserved, not stripped), and comment stripping. A shim in `pretool-bash-safety.sh` computes `COMMAND_CONTEXT_STRIPPED` and applies it to the four affected rules; Layer 1.A-E and absolute bans remain on raw `$COMMAND`. M2 binding honored: line 743 compound rule applies stripped view to danger-token grep only, raw `$COMMAND` to workflow-path grep.

**Tests**: 30 unit tests (`hooks/tests/test_bash_safety_context.py`) + 19 AC tests (`tests/generated/20260522-080646-A/`).

**Codex bugs caught**: DANGER_COMMANDS exclusion (quoted args to `killall "happy"` were incorrectly stripped, creating exit-0 bypass).

### B — write-guard sentinel integration ✅ PASS (QA iter3)

**Problem**: The `/allow` sentinel-grant system had no Write-tool support. Attempts to use `Write` to overwrite existing files were unconditionally blocked with no grant pathway.

**Solution**: Added `match_sentinel_grant_for_write(task_id, session_id, target_path)` to `hooks/lib/allowlist.py` (mirrors Bash sentinel matcher; handles targetless grants as wildcards). Hoisted FILE_PATH parse in `hooks/pretool-write-guard.sh` and inserted sentinel check before existing-file block; CF-1 invariant fires only when file exists AND sentinel is `valid_no_match`. Extended `hooks/posttool-allowlist-consume.py` to consume Write grants via 3-candidate task_id lookup (matching `userprompt-consent-allowlist.sh` keying logic) — symmetry with pretool verified.

**Codex bugs caught**: (1) CF-1 must be gated on `[ -f FILE_PATH ]`; (2) shell glob matching triggered CF-1 on expired sentinels; (3) posttool must use same 3-candidate lookup as pretool (single-candidate allowed unlimited re-use of same sentinel).

**Tests**: 4 AC tests (`tests/generated/20260522-080646-B/`) + 40 unit tests in `test_allowlist_consolidation.py`.

### D — dev self-report provenance ✅ PASS (QA iter2, approve-with-warnings)

**Problem**: Dev agents self-reported `files_modified` without verifying against a baseline git snapshot. A dev could claim it modified files it didn't touch, or omit files it did touch, with no gate to catch it.

**Solution**: 
- `commands/dev.md`: orchestrator captures `baseline_head_sha` + `baseline_dirty_snapshot` at dispatch time.
- `agents/dev.md`: `files_modified` mandated from `git diff --name-only <baseline_head_sha>` (working-tree form); `files_created` from `git ls-files --others --exclude-standard` excluding baseline untracked paths; `observed_preexisting[]` for pre-existing dirty files.
- `agents/qa.md`: provenance check gates — `files_modified` paths absent from git diff + not in baseline_dirty_snapshot → `dev_provenance_violation`; `files_created` paths absent from git ls-files → `files_created_provenance_violation`.
- `agents/changelog-analyst.md`: Phase 2 provenance filter uses split: git diff for `files_modified`, git ls-files for `files_created`. Working-tree form (`git diff --name-only <sha>`, NOT `..HEAD`) — codex found the `..HEAD` form would return empty pre-commit.

**Warning W1**: Dev-D's own report under-counts `files_modified` (claims 2, git diff shows 4). The discrepancy is because items were modified across multiple remediation rounds; each round's report is correct for that round. The aggregate QA verified all 4 files are correctly implemented.

**Tests**: 5 AC tests (`tests/generated/20260522-080646-D/`).
