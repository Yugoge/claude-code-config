# BA Specification: Privilege Guard Security Gap — auto-bulk: Bypass in BULK=false Mode

**Request ID**: 20260526-052545
**Created**: 2026-05-26T05:25:45Z
**Tier**: STANDARD

## Goal

Prevent `changelog-analyst` from bypassing `pretool-git-privilege-guard.py` by
using the `auto-bulk:` commit message prefix in BULK=false (non-bulk) mode. The
fix requires three components: (1) the guard must verify a bulk-commit sentinel
exists when `BLESSED_BRIDGE_RE` matches; (2) `write-bulk-commit-sentinel.py`
must enforce a human-auth-flag gate before writing the sentinel; (3)
`agents/changelog-analyst.md` DO NOT section must explicitly prohibit the
`auto-bulk:` prefix in BULK=false mode.

## Context

`BLESSED_BRIDGE_RE` in `pretool-git-privilege-guard.py` (line 95) matches
commit messages of the form `auto-bulk: end-of-cycle commit for <branch>`. Prior
to commit `41063a1`, matching this regex bypassed grant verification entirely —
no sentinel was required. A `changelog-analyst` running in BULK=false could
fabricate an `auto-bulk:` prefix commit message and be granted write access
without any human authorization, committing arbitrary files (observed: another
session's `changelog-analyst` committed `close.md`, `score-update.sh`, and
`score-inject.sh` that belonged to a different task context).

Commits `41063a1` and `e443a98` added the sentinel check and deferral logic to
the guard. However, two additional gaps remain: (a) `write-bulk-commit-sentinel.py`
had its human-auth-flag gate removed in a subsequent auto-bulk commit (`9d68a38`),
allowing any subagent to write a valid sentinel, and (b) the
`changelog-analyst.md` DO NOT section has no explicit rule prohibiting
`auto-bulk:` prefix in BULK=false mode.

## Edge Cases & Risks

- **Risk 1 — stale grant files**: `_has_active_commit_grant()` is a global `/tmp` scan. Stale `.json`/`.lck` grant files from terminated sessions will cause active bulk sessions to be incorrectly deferred. The `stop-cleanup-allowlist.sh` should reap these, but if a session crashes it won't. The tests expose this via real `/tmp` pollution (2 test failures observed). The S1 fix (mock `_has_active_commit_grant` in test) resolves this for unit tests.
- **Risk 2 — auth-flag path override must be `/tmp/`-rooted**: the `CLAUDE_BULK_AUTH_FLAG_PATH_OVERRIDE` env var must be constrained to `/tmp/`-rooted paths with a whitelisted basename prefix, or `pretool-bash-safety.sh` may block the auth flag write in production. Verify against Layer 1.F pattern in `pretool-bash-safety.sh`.
- **Risk 3 — unlink race**: the auth flag is unlinked before sentinel write. If two concurrent `/commit --bulk` invocations race on the same session, the second will fail with "auth flag absent" (the first already consumed it). This is intentional — single-use consumption ensures exactly-once authorization per human action.
