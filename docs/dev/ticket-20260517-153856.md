# BA Specification: Establish Bidirectional Handoff Contracts Between Specialist Agents and BA

**Request ID**: 20260517-153856
**Created**: 2026-05-17T15:38:56Z
**Tier**: STANDARD

## Goal

Establish bidirectional handoff contracts between all specialist agents (ui-specialist, architect, product-owner, user) and the downstream BA/dev/QA pipeline by (1) enriching each specialist's output schema with BA-actionable fields and (2) adding a Specialist Findings Intake Protocol to agents/ba.md.

For ui-specialist specifically, the machine-enforced contract lives in `skills/ui-shared/report-schema.json` (JSON Schema Draft-07); the three canonical output channels are `automated_findings[]`, `contextual_findings[]`, and `aesthetic_findings[]`. The `issues[]` key at `agents/ui-specialist.md:255` is a prose example only — not a schema-validated channel. Making ui-specialist's handoff machine-enforceable requires enriching `report-schema.json`'s `finding_base` definition (structured location, selector, measured, expected, downstream_agent). For the other four files (architect, product-owner, user, ba.md) — which have no separate JSON schema enforcer — the changes are agent-prose convention additions that are binding by the same convention as all other agent spec constraints.

## Context

All four specialist agents currently report findings in output schemas designed for PM triage (severity, category for prioritization) rather than for BA root-cause analysis. BA and dev receive natural-language descriptions ("button spacing inconsistent") but lack the CSS selector, measured vs. expected values, and source file locations required to make deterministic implementation decisions. No documented consumption protocol exists on the BA side.

For ui-specialist: the machine-enforced output contract is `skills/ui-shared/report-schema.json` (JSON Schema Draft-07). It defines three canonical finding channels: `automated_findings[]` (finding_deterministic), `contextual_findings[]` (finding_contextual), `aesthetic_findings[]` (finding_aesthetic) — all inheriting from `finding_base`. The `finding_base` definition has `location` as a flat string with no selector, file, line, or url decomposition. None of the finding types have `measured`, `expected`, or `downstream_agent` fields at the base level (finding_deterministic has `computed_value`/`expected_value` but only for that channel). The `issues[]` key mentioned in the Output Format section of ui-specialist.md is a prose illustration of the overall report structure — it is not a channel in report-schema.json and carries no schema enforcement.

The fix is: (a) enrich `report-schema.json`'s `finding_base` with structured location, measured, expected, downstream_agent; (b) update ui-specialist.md prose to match; (c) add BA-actionable fields to the other three specialist agent prose schemas; (d) add a Specialist Findings Intake Protocol to ba.md. All sides must change simultaneously.

## Edge Cases & Risks

- **user agent is browser-only**: agents/user.md explicitly forbids reading source code. The `location.file` field MUST be nullable (null when source not accessible). Dev must not make location.file required for user agent.
- **ui-specialist delegates to report-schema.json**: The inline schema in ui-specialist.md is ADDITIVE. It does not replace the `~/.claude/skills/ui-shared/report-schema.json` delegation — it adds an inline illustration alongside the external schema reference. The two must stay consistent.
- **architect runtime_evidence vs measured**: architect.md already has `runtime_evidence` (console error / network failure / performance metric). The `measured` field added by this spec is the scalar extracted from that evidence (e.g., "2340ms"). Dev must ensure measured does not duplicate runtime_evidence but rather distills it.
- **BA intake protocol scope**: The new BA section documents consumption rules only for findings already in BA's possession (passed via dev.md Step 2). It does NOT change how BA receives findings or when specialists run.
- **Schema version drift**: No `schema_version` field is mandated (Won't Have), so future schema changes could silently break the contract. This is an accepted risk — the user did not request versioning.
