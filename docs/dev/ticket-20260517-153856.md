# BA Specification: Establish Bidirectional Handoff Contracts Between Specialist Agents and BA

**Request ID**: 20260517-153856
**Created**: 2026-05-17T15:38:56Z
**Tier**: STANDARD

## Goal

Establish machine-enforceable bidirectional handoff contracts between all specialist agents (ui-specialist, architect, product-owner, user) and the downstream BA/dev/QA pipeline by (1) enriching each specialist's issues[] JSON schema with BA-actionable fields and (2) adding a Specialist Findings Intake Protocol to agents/ba.md.

## Context

All four specialist agents currently report findings in issues[] arrays designed for PM triage (severity, category for prioritization) rather than for BA root-cause analysis. BA and dev receive natural-language descriptions ("button spacing inconsistent") but lack the CSS selector, measured vs. expected values, and source file locations required to make deterministic implementation decisions. No documented consumption protocol exists on the BA side: BA has no rule for which specialist fields are mandatory, how to handle vague claims, or what to do when measured/expected are absent.

The fix is additive schema enrichment on the output side plus a formal intake protocol on the consumption side. Both sides must change simultaneously for the contract to be binding.

## Edge Cases & Risks

- **user agent is browser-only**: agents/user.md explicitly forbids reading source code. The `location.file` field MUST be nullable (null when source not accessible). Dev must not make location.file required for user agent.
- **ui-specialist delegates to report-schema.json**: The inline schema in ui-specialist.md is ADDITIVE. It does not replace the `~/.claude/skills/ui-shared/report-schema.json` delegation — it adds an inline illustration alongside the external schema reference. The two must stay consistent.
- **architect runtime_evidence vs measured**: architect.md already has `runtime_evidence` (console error / network failure / performance metric). The `measured` field added by this spec is the scalar extracted from that evidence (e.g., "2340ms"). Dev must ensure measured does not duplicate runtime_evidence but rather distills it.
- **BA intake protocol scope**: The new BA section documents consumption rules only for findings already in BA's possession (passed via dev.md Step 2). It does NOT change how BA receives findings or when specialists run.
- **Schema version drift**: No `schema_version` field is mandated (Won't Have), so future schema changes could silently break the contract. This is an accepted risk — the user did not request versioning.
