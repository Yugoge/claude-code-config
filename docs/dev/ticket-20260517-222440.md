## Edge Cases & Risks

1. **Grant consumption double-counting**: `pretool-bash-safety.sh` calls `check_and_consume_allowlist()` at line 615 (which LOGS but does NOT delete — deletion is PostToolUse). `pretool-git-privilege-guard.py`'s `_check_git_allowlist()` is read-only (lines 141-177, confirmed by refactor commit aaf3c44 which explicitly made it "read-only"). Consumption is delegated to `posttool-allowlist-consume.py`. So adding the call in `_evaluate_commit()` does NOT double-consume the grant. Both hooks read the same grant file; PostToolUse fires once and deletes it. Correct.

2. **Blessed-bridge regression**: The `BLESSED_BRIDGE_RE` check (line 452) must remain as the first early-exit in `_evaluate_commit()`. If the new `_check_git_allowlist()` call is inserted BEFORE the blessed-bridge check, every normal auto-bulk commit would do an unnecessary file I/O. Insert AFTER the blessed-bridge check, same as the `/commit`-wrapper grant checks that follow it.

3. **Error message update scope**: Only `_block_default_deny_commit()` at lines 433-446 needs updating. The message says "It can't be bypassed from inside the session" — this becomes false after the fix. The update must be narrow: indicate that `/allow` works for main agent only; for subagents the block remains unconditional. Do not touch other block messages in the file.
