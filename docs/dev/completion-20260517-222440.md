## Root Cause Analysis

**Symptom**: After `/allow`, `git commit` is still blocked by `pretool-git-privilege-guard.py` even though the orchestrator gate and `pretool-bash-safety.sh` are both cleared.

**Root Cause**: `_evaluate_commit()` in `pretool-git-privilege-guard.py` did not call `_check_git_allowlist(command, data)`. Every other `evaluate-*` function in the same file (`_evaluate_merge`, `_evaluate_reset_hard`, `_evaluate_direct_ref_mutation`, `_evaluate_forbidden_plumbing`) calls it; `_evaluate_commit` was the sole exception.

**Root Cause Commit**: `c6653c0 — fix(hooks): /allow + /do behavior + english-only durable prevention` — this commit fixed `pretool-bash-safety.sh` but did NOT touch `pretool-git-privilege-guard.py`.

**Why It Happened**: The prior "永久彻底修复" cycle only updated one of two independent PreToolUse hooks. The privilege guard's commit path was intentionally kept tighter than other paths and was not updated in the same cycle.

## Implementation

**Approach**: Three minimal additive changes to `hooks/pretool-git-privilege-guard.py`:

1. In `_evaluate_commit()`: added `if _check_git_allowlist(command, data): return` after the `BLESSED_BRIDGE_RE` check and before the SID grant lookup — the exact same pattern already in `_evaluate_merge()`
2. In `_block_default_deny_commit()`: added sentence "Main agent may bypass with `/allow <pattern>` before the git commit command." (additive, nothing removed)
3. In `_check_git_allowlist()` docstring: removed `non-commit/` prefix that became factually wrong after change 1
