# BA Specification: Fix /allow Write Sentinel — Writer Path Bugs + Schema Mismatch (Privilege Escalation)

**Request ID**: 20260524-125300-B
**Created**: 2026-05-24T12:53:00Z
**Tier**: STANDARD
**Baseline**: f79fba1c1684d246928a3cfb39bf1ed7a9a5c8a5
**Severity**: critical (upgraded from major — two additional critical writer-path bugs found by QA)
**Risk level**: high

## Goal

Fix three co-required bugs in the `/allow` sentinel stack that together allow any
`/allow --tool "Write /path"` invocation to either produce no sentinel at all
(CRITICAL-2) or produce a bare wildcard Write grant instead of a scoped one
(CRITICAL-1 and ORIGINAL), resulting in unrestricted Write access rather than
the scoped access the user intended.

## Context

The `/allow` command processes user prompts and writes two grant artifacts: a legacy
pattern-string grant at `/tmp/claude-bash-allowlist-<sid>.json` and a structured
sentinel at `/tmp/claude-grants/<task_id>.json`. Three bugs together make
path-scoped Write grants non-functional:

**CRITICAL-2 (prerequisite)**: `_looks_regex()` in the Python parser at
`hooks/userprompt-consent-allowlist.sh:82-84` returns `True` for any string
containing `.` (dot is in the character class `[\^$\\.*+?\[\]\(\){}|]`). When
`--tool` is used, the `flag_pattern` is `"Write /tmp/allowed.json"` — which
contains a dot — so `is_regex = _looks_regex(flag_pattern)` returns `True`. The
sentinel write block at line 180-182 exits early when `IS_REGEX="true"`. Result:
`/allow --tool "Write /tmp/file.json"` produces **no sentinel at all** for any
filename with an extension (`.json`, `.py`, `.sh`, etc.).

**CRITICAL-1 (prerequisite)**: When the user runs the natural form `/allow Write
/tmp/file.json` (no `--tool` flag), the argument parser treats `Write` as the
first bare token (the pattern) and `/tmp/file.json` as the comment (bare tokens
after the first). `rest` is empty in the sentinel writer. Sentinel emitted is
`{"op":"Write"}` — a bare wildcard, not a scoped grant. Runtime verified by QA:
stdout shows `pattern='Write' is_regex=false comment='/tmp/specific-file.json'`.

**ORIGINAL bug (consequence)**: Even when `IS_REGEX=false` and `rest` is populated
(e.g., for a dotless path like `/tmp/allowed`), the sentinel writer at lines
199-204 emits `{"op":"Write","args_contain":[rest]}` — no `"target"` field.
`match_sentinel_grant_for_write` at `hooks/lib/allowlist.py:521-523` checks
`entry.get("target")` → returns `None` → wildcard fires. Result: even the one
remaining reachable path produces a wildcard grant, not a scoped one.

All three bugs must be fixed atomically; fixing only one or two leaves the
privilege escalation in place.

## Edge Cases & Risks

1. **`entry.get("target") is None` conflates absent key with `"target": null`.**
   Fix requires branching on `"target" in entry` (key presence), not `is None`
   (value presence). If `"target"` is present but null or empty string, deny
   (fail-closed).

2. **`args_contain` substring semantics would admit prefix/suffix paths.**
   Legacy sentinel fallback must use exact string equality: `args_contain == [target_path]`
   (one-item list, string equals). `/specific/path` must not match `/specific/path-suffix`.

3. **Backward compatibility for bare `/allow Write` (no path argument).**
   When the pattern is just `Write` with no second token, the writer emits
   `{"op": "Write"}` — no `target`, no `args_contain`. This is the intentional
   wildcard shape. Both CRITICAL-1 fix and matcher must continue to allow it.

4. **M2 scope: bare `Write <path>` special-casing must be narrowly defined.**
   Only the specific two-token bare form `Write <path>` (where `bare[0] == "Write"`
   and `len(bare) >= 2`) is special-cased. Any additional bare tokens beyond the
   path become the comment. The existing comment-parsing for non-Write bare patterns
   is unchanged.

5. **Unquoted `--tool Write /path`.**
   `/allow --tool Write /tmp/file.json` (unquoted) causes `--tool` to consume only
   `Write` as the pattern; `/tmp/file.json` lands in `bare`. After M1 fix,
   `is_regex=False` and `pattern="Write"` → sentinel emits `{"op":"Write"}` — still
   a wildcard. This is out of scope but must be documented: the correct syntax is
   `/allow --tool "Write /path"` (quoted) or `/allow Write /path` (bare two-token form).

6. **CF-1 interaction in `pretool-write-guard.sh`.**
   The guard uses `valid_no_match` output to suppress the legacy `/allow` grant
   fallback for overwrite attempts. A matcher fix that returns `None` instead of
   an entry for the exploit shape will still emit `valid_no_match` when a
   same-session sentinel exists — which correctly blocks the overwrite. No change
   to the guard shell logic is needed.

7. **Path equality policy.**
   `target` matching is exact string equality. Writer emits the path verbatim.
   Users are responsible for providing absolute paths — no `~` expansion, no symlink
   resolution, no normalization at writer or matcher side.

## Acceptance Criteria

### AC1: Scoped grant allows exactly the named file

- GIVEN a sentinel with `allowed_operations: [{"op":"Write","target":"/tmp/allowed.json"}]`
- WHEN `match_sentinel_grant_for_write(task_id, session_id, "/tmp/allowed.json")` is called
- THEN the function returns a non-None dict with `op=="Write"` and `target=="/tmp/allowed.json"`

### AC2: Scoped grant blocks any other file

- GIVEN a sentinel with `allowed_operations: [{"op":"Write","target":"/tmp/allowed.json"}]`
- WHEN `match_sentinel_grant_for_write(task_id, session_id, "/tmp/other.json")` is called
- THEN the function returns None

### AC3: Posttool consume fires and unlinks sentinel after Write to the named file

- GIVEN a valid sentinel with `{"op":"Write","target":"/tmp/allowed.json"}`
  and `posttool-allowlist-consume.py` receives a Write PostToolUse event with
  `tool_input.file_path == "/tmp/allowed.json"`
- WHEN posttool runs
- THEN `consume_sentinel_grant_on_terminal_result` is called and the sentinel file
  at `/tmp/claude-grants/<task_id>.json` is unlinked

### AC4: Legacy broken sentinel exact-equality: args_contain path matches only exact path

- GIVEN an old-schema sentinel with `{"op":"Write","args_contain":["/specific/path"]}`
  (no `target` field — as produced by the buggy writer before this fix)
- WHEN `match_sentinel_grant_for_write(task_id, session_id, "/other/path")` is called
- THEN the function returns None (exact equality required, not substring)

### AC5: Exploit-shape regression — no wildcard leakage from args_contain

- GIVEN an old-schema sentinel `{"op":"Write","args_contain":["/specific/path"]}`
- WHEN called with `/specific/path`: returns non-None
- WHEN called with `/other/path`: returns None
- WHEN called with `/specific/path-suffix`: returns None
  (path-prefix substring must not match)

### AC6 (non-regression): Bare `/allow Write` sentinel preserves wildcard behavior

- GIVEN a sentinel with `{"op":"Write"}` (no `target`, no `args_contain` — the
  intentional wildcard shape produced by `/allow Write` with no path argument)
- WHEN `match_sentinel_grant_for_write(task_id, session_id, "/any/file.py")` is called
- THEN the function returns non-None (wildcard intentionally preserved)

### AC7: End-to-end — `/allow --tool "Write /tmp/file.json"` produces scoped sentinel

- GIVEN `userprompt-consent-allowlist.sh` is invoked as a subprocess with a hook JSON
  payload `{"prompt":"/allow --tool \"Write /tmp/file.json\"","session_id":"<sid>"}` piped
  to stdin, with env vars `CLAUDE_TASK_ID=<task_id>` set
- WHEN the hook exits
- THEN the sentinel file at `/tmp/claude-grants/<task_id>.json` EXISTS
- AND `allowed_operations[0]` has `op=="Write"` and `target=="/tmp/file.json"` (NOT `args_contain`, NOT missing)
- AND `match_sentinel_grant_for_write(task_id, session_id, "/tmp/file.json")` returns non-None
- AND `match_sentinel_grant_for_write(task_id, session_id, "/tmp/other.json")` returns None

### AC8: End-to-end — bare `/allow Write /tmp/file.json` produces scoped sentinel

- GIVEN `userprompt-consent-allowlist.sh` is invoked as a subprocess with payload
  `{"prompt":"/allow Write /tmp/file.json","session_id":"<sid>"}`, with `CLAUDE_TASK_ID=<task_id>`
- WHEN the hook exits
- THEN the sentinel file at `/tmp/claude-grants/<task_id>.json` EXISTS
- AND `allowed_operations[0]` has `op=="Write"` and `target=="/tmp/file.json"` (NOT a bare wildcard `{"op":"Write"}`)
- AND `match_sentinel_grant_for_write(task_id, session_id, "/tmp/file.json")` returns non-None
- AND `match_sentinel_grant_for_write(task_id, session_id, "/tmp/other.json")` returns None

  `if flag_pattern is not None:` branch. `--tool` is an explicit literal declaration;

- **CRITICAL-1 fix** (parser lines 90-97):
  In the `elif bare:` branch, add special-casing: if `bare[0] == "Write"` and
  `len(bare) >= 2` and `bare[1].startswith("/")`, treat `bare[1]` as the path and
  `" ".join(bare[2:])` as the comment. The pattern becomes `"Write " + bare[1]`
  so the sentinel writer can split it correctly. All other bare patterns are unchanged.

- **ORIGINAL writer fix** (lines 199-204):
  In the sentinel write Python block, replace:
  ```python
  entry = {'op': op}
  if rest:
      entry['args_contain'] = [rest]
  ```
  with:
  ```python
  entry = {'op': op}
  if op == 'Write' and rest:
      entry['target'] = rest
  elif rest:
      entry['args_contain'] = [rest]
  ```
  For non-Write ops, `args_contain` behavior is unchanged.

- **ORIGINAL matcher fix** (lines 521-523):
  Replace the `entry_target = entry.get("target")` / `if entry_target is None`
  branch with key-presence logic:
  - `if "target" in entry`: require `isinstance(entry["target"], str)` and
    `entry["target"] == target_path`; all other target values deny (fail-closed).
  - `elif "args_contain" in entry`: allow only if `args_contain` is a one-item
    list whose sole element is a string equal to `target_path` exactly (legacy compat).
  - `else` (bare `{"op":"Write"}`): allow (intentional wildcard).

- **Docstring fix** (lines 488-492):
  Update to: "target present → exact equality; args_contain present → exact equality
  (legacy fallback); neither → intentional wildcard."

- **Posttool**: no logic changes needed; `posttool-allowlist-consume.py` calls
  `match_sentinel_grant_for_write` — it is implicitly fixed.
