## Goal

Establish a git-diff-derived provenance contract for `dev.files_modified` and `dev.files_created` so that QA, changelog-analyst, and the commit chain can trust the dev report reflects actual changes from this cycle — not pre-existing files found in an already-correct state.

---

## Context

In cycle dev-20260519-211515, the dev agent reported 17 modified + 8 created files. All 25 were already present in git HEAD from prior-session commits (6d28883, 28a1e85, 4d9f9f5, 6537d7d). Dev performed work-tree inspection ("files in expected state = files I changed") rather than diffing against a pre-dispatch baseline. QA and changelog-analyst consumed this list as ground truth, producing inflated commit attribution and misleading provenance throughout the quality chain.

The fix is a four-part markdown-only contract change (no hooks, no scripts, no settings.json):

1. `commands/dev.md` Step 8 captures baseline before dispatching dev.
2. `agents/dev.md` mandates git-diff derivation for `files_modified`/`files_created` and introduces `observed_preexisting[]`.
3. `agents/qa.md` adds a provenance check that fails when reported files are absent from the diff.
4. `agents/changelog-analyst.md` marks such paths as anomaly and excludes them from enrichment scope.

## Requirements (MoSCoW)

### Must Have

- M1: `commands/dev.md` Step 8 captures `baseline_head_sha` (from `git rev-parse HEAD`) and `baseline_dirty_snapshot` (from `git status --porcelain`) immediately before dispatching the dev agent, and passes both into the dispatch payload.
- M2: `agents/dev.md` Output Format specifies that `dev.files_modified` and `dev.files_created` MUST be derived from `git diff --name-only <baseline_head_sha>` (comparing working tree to baseline), NOT from work-tree inspection of expected state. The dev-report must also include `baseline_head_sha` as a top-level field so downstream consumers (QA, changelog-analyst) can independently re-derive the diff.
- M3: `agents/dev.md` Output Format adds `observed_preexisting[]` — a list of file paths that dev expected to be in a certain state and confirmed correct but that do NOT appear in `git diff --name-only <baseline_head_sha>`. These are explicitly excluded from `files_modified` and `files_created`. Files that were in `baseline_dirty_snapshot` at dispatch time and match expected state without appearing in the diff belong in `observed_preexisting[]`.
- M4: `agents/qa.md` adds a provenance check: compute `git diff --name-only <baseline_head_sha>` (using the `baseline_head_sha` from the context JSON or dev-report); for every path in `dev.files_modified` that is neither in this diff output NOR in `baseline_dirty_snapshot`, raise a critical FAIL finding with `primary_cause: "dev_implementation"` and label `dev_provenance_violation`.
- M5: `agents/changelog-analyst.md` Phase 2 dev-report enrichment adds a note: paths in `dev.files_modified` / `dev.files_created` that are absent from `git diff --name-only <baseline_head_sha>..HEAD` (at commit time, HEAD is updated) AND were not in `baseline_dirty_snapshot` are classified as `provenance_anomaly` and MUST NOT be used to derive commit type, scope, or summary enrichment. They are staged if present in git status but their dev-report attribution is disregarded.

### Should Have

- S1: When `baseline_head_sha` is absent or null in the dev-report (unborn repo, pre-first-commit cycle), QA skips the provenance check and logs a warning rather than failing. Changelog-analyst also skips the anomaly classification in this edge case.

### Could Have

- C1: A future cycle could add a hook that enforces the baseline capture at dispatch time rather than relying on the markdown spec alone.

- No hooks, no settings.json changes, no new scripts.
- No changes to commit.sh or /commit command.
- No changes to spec files or orchestrator spec integration.

## Edge Cases & Risks

- **Unborn/shallow repo**: If no commits exist at dispatch time, `baseline_head_sha` will be null. Both dev and QA must handle this gracefully (skip diff derivation, log warning).
- **Dirty baseline at dispatch**: Files already dirty before dev runs appear in `baseline_dirty_snapshot`. If dev confirms they are in expected state, they go into `observed_preexisting[]`, not into `files_modified`. The provenance check in QA must exclude `baseline_dirty_snapshot` entries from the FAIL set.
- **Dev commit timing**: Dev agent is forbidden from running `git commit`. At QA time, changes are uncommitted. The correct diff command is `git diff --name-only <baseline_head_sha>` (working-tree vs baseline SHA), not `git diff <baseline_head_sha>..HEAD` (which compares committed HEAD to baseline — will be empty while changes are staged but not committed). At changelog-analyst time (post-commit), `git diff --name-only <baseline_head_sha>..HEAD` is correct.
- **Two repos (nested repo)**: The nested repo at `/dev/shm/dev-workspace/dot-claude` has its own HEAD. If dev modifies files in both repos, both `baseline_head_sha` values must be captured and passed. Dev's dispatch payload should distinguish `baseline_head_sha_root` and `baseline_head_sha_nested` if applicable, or use the repo-relative path to determine which SHA applies.
- **Test-writer + parallel dev**: In parallel-dev cycles, each worker has its own baseline. The aggregate dev-report must union `files_modified` sets, but the provenance check must be per-worker.

## Technical Hints

- **Affected files**: `commands/dev.md` (Step 8 pre-dispatch block, lines ~647–673), `agents/dev.md` (Output Format, lines ~488–574), `agents/qa.md` (Regression Testing Step 5 or new Step, lines ~396–430), `agents/changelog-analyst.md` (Phase 2 dev-report enrichment, lines ~82–116)
- **Diff command distinction**: At dev-execution time (uncommitted changes), use `git diff --name-only <baseline_head_sha>` (working tree vs SHA). At commit/QA time if changes are committed, use `git diff --name-only <baseline_head_sha>..HEAD`. The spec must be precise about which form applies at which agent stage.
- **Two-repo handling**: If baseline needs to cover both the main repo and nested repo, consider `baseline_head_sha_root` and `baseline_head_sha_nested` as separate fields, or document that the provenance check applies per-repo.
- **observed_preexisting[] semantics**: This field is informational — it explains why dev interacted with a file without modifying it. It does not block QA or changelog-analyst.
