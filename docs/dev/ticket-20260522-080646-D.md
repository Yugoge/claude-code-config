## Goal

Establish a git-diff-derived provenance contract for `dev.files_modified` and `dev.files_created` so that QA, changelog-analyst, and the commit chain can trust the dev report reflects actual changes from this cycle — not pre-existing files found in an already-correct state.

---

## Context

In cycle dev-20260519-211515, the dev agent reported 17 modified + 8 created files. All 25 were already present in git HEAD from prior-session commits (6d28883, 28a1e85, 4d9f9f5, 6537d7d). Dev performed work-tree inspection ("files in expected state = files I changed") rather than diffing against a pre-dispatch baseline. QA and changelog-analyst consumed this list as ground truth, producing inflated commit attribution and misleading provenance throughout the quality chain.

The fix is a four-part markdown-only contract change (no hooks, no scripts, no settings.json):

1. `commands/dev.md` Step 8 captures baseline before dispatching dev.
2. `agents/dev.md` mandates git-diff derivation for `files_modified`/`files_created` and introduces `observed_preexisting[]`.
3. `agents/qa.md` adds a provenance check that fails when reported files are absent from the diff.
4. `agents/changelog-analyst.md` marks such paths as anomaly and excludes them from enrichment scope.

## Requirements (MoSCoW)

### Must Have

- M1: `commands/dev.md` Step 8 captures `baseline_head_sha` (from `git rev-parse HEAD`) and `baseline_dirty_snapshot` (from `git status --porcelain`) immediately before dispatching the dev agent, and passes both into the dispatch payload.
- M2: `agents/dev.md` Output Format specifies a split derivation rule — NOT from work-tree inspection of expected state:
  - `dev.files_modified`: MUST be derived from `git diff --name-only <baseline_head_sha>` (working-tree diff against the baseline SHA received in the dispatch payload — lists modified tracked files).
  - `dev.files_created`: MUST be derived from `git ls-files --others --exclude-standard` (untracked new files at end of dev execution), EXCLUDING any paths that appeared as untracked entries (`??`-prefixed lines) in the parsed `baseline_dirty_snapshot` captured before dispatch (those were already untracked before this task started and were not created by it).
  The dev-report must also include `baseline_head_sha` and `baseline_dirty_snapshot` as top-level fields so downstream consumers (QA, changelog-analyst) can independently re-derive both sets.
- M3: `agents/dev.md` Output Format adds `observed_preexisting[]` — a list of file paths that dev expected to be in a certain state and confirmed correct but that do NOT appear in `git diff --name-only <baseline_head_sha>`. These are explicitly excluded from `files_modified` and `files_created`. Files that were in `baseline_dirty_snapshot` at dispatch time and match expected state without appearing in the diff belong in `observed_preexisting[]`.
- M4: `agents/qa.md` adds a provenance check with two sub-checks: (a) for `dev.files_modified` — compute `git diff --name-only <baseline_head_sha>` (working-tree vs baseline, using the `baseline_head_sha` from the context JSON or dev-report); for every path in `dev.files_modified` that is neither in this diff output NOR in `baseline_dirty_snapshot`, raise a critical FAIL finding with `primary_cause: "dev_implementation"` and label `dev_provenance_violation`; (b) for `dev.files_created` — compute `git ls-files --others --exclude-standard` (untracked files at QA time); for every path in `dev.files_created` that is absent from this untracked list AND absent from `baseline_dirty_snapshot`, raise a critical FAIL finding with `primary_cause: "dev_implementation"` and label `files_created_provenance_violation`.
- M5: `agents/changelog-analyst.md` Phase 2 dev-report enrichment adds a split provenance filter (Phase 2 runs before staging and commit — `..HEAD` form is WRONG here and returns empty set):
  - For paths in `dev.files_modified`: check absence from `git diff --name-only <baseline_head_sha>` (working-tree diff). Paths absent from this diff AND not in `baseline_dirty_snapshot` are classified as `provenance_anomaly`.
  - For paths in `dev.files_created`: check absence from `git ls-files --others --exclude-standard`. Paths absent from this untracked list AND not present as `??`-prefixed entries in the parsed `baseline_dirty_snapshot` are classified as `provenance_anomaly`. (New untracked files do not appear in `git diff --name-only` output; applying git diff to this set would falsely flag all legitimate new files as anomalies.)
  `provenance_anomaly` paths MUST NOT be used to derive commit type, scope, or summary enrichment. They are staged if present in git status but their dev-report attribution is disregarded.

### Should Have

- S1: When `baseline_head_sha` is absent or null in the dev-report (unborn repo, pre-first-commit cycle), QA skips the provenance check and logs a warning rather than failing. Changelog-analyst also skips the anomaly classification in this edge case.

## Edge Cases & Risks

- **Unborn/shallow repo**: If no commits exist at dispatch time, `baseline_head_sha` will be null. Both dev and QA must handle this gracefully (skip diff derivation, log warning).
- **Dirty baseline at dispatch**: Files already dirty before dev runs appear in `baseline_dirty_snapshot`. If dev confirms they are in expected state, they go into `observed_preexisting[]`, not into `files_modified`. The provenance check in QA must exclude `baseline_dirty_snapshot` entries from the FAIL set.
- **Dev commit timing / changelog-analyst Phase 2 runs pre-commit**: Dev agent is forbidden from running `git commit`. At QA time and at changelog-analyst Phase 2 time, changes are uncommitted. The correct diff command is `git diff --name-only <baseline_head_sha>` (working-tree vs baseline SHA) for ALL agent stages — dev, QA, and changelog-analyst Phase 2 alike. The `..HEAD` form (`git diff <baseline_head_sha>..HEAD`) compares committed HEAD to baseline; it returns an empty set while changes are staged but not yet committed, and would falsely classify all legitimate working-tree changes as `provenance_anomaly`. The erroneous notion that "changelog-analyst runs post-commit at Phase 2" is incorrect: Phase 2 is file classification, which runs before Phase 3 (lock acquisition), Phase 5 (staging), and Phase 6+ (commit).
- **Two repos (nested repo)**: The nested repo at `/dev/shm/dev-workspace/dot-claude` has its own HEAD. If dev modifies files in both repos, both `baseline_head_sha` values must be captured and passed. Dev's dispatch payload should distinguish `baseline_head_sha_root` and `baseline_head_sha_nested` if applicable, or use the repo-relative path to determine which SHA applies.
- **Test-writer + parallel dev**: In parallel-dev cycles, each worker has its own baseline. The aggregate dev-report must union `files_modified` sets, but the provenance check must be per-worker.

## Technical Hints

- **Affected files**: `commands/dev.md` (Step 8 pre-dispatch block, lines ~647–673), `agents/dev.md` (Output Format, lines ~488–574), `agents/qa.md` (Regression Testing Step 5 or new Step, lines ~396–430), `agents/changelog-analyst.md` (Phase 2 dev-report enrichment, lines ~82–116)
- **Split derivation rule**: `files_modified` uses `git diff --name-only <baseline_head_sha>` (working-tree form, no `..HEAD`) at ALL agent stages. `files_created` uses `git ls-files --others --exclude-standard` — untracked files never appear in git diff output, so applying diff to files_created would falsely classify ALL legitimate new files as `provenance_anomaly`. The `..HEAD` form must NEVER be used at Phase 2 time (changes are uncommitted; `..HEAD` returns empty set). Changelog-analyst Phase 2 provenance filter applies the same split: diff for files_modified, ls-files for files_created.
- **baseline_dirty_snapshot precision**: when captured at dispatch time, use `git status --porcelain` output. When excluding pre-existing untracked paths from `files_created`, parse only the `??`-prefixed lines from baseline_dirty_snapshot (not modified-tracked lines). This prevents directory-level `?? dir/` entries from incorrectly matching file-level `dir/file` entries.
- **Two-repo handling**: If baseline needs to cover both the main repo and nested repo, consider `baseline_head_sha_root` and `baseline_head_sha_nested` as separate fields, or document that the provenance check applies per-repo.
- **observed_preexisting[] semantics**: This field is informational — it explains why dev interacted with a file without modifying it. It does not block QA or changelog-analyst.
- **Verification harness cleanup contract (R10 cross-reference)**: any harness cleanup with `trap.*EXIT INT TERM` MUST use the canonical pattern in `agents/qa.md`; no `pkill -f`/`killall`; never touch the frozen spec `docs/dev/specs/spec-20260520-044700.md`.
