# BA Specification: Fix /do and /allow Bypass Coverage in PreToolUse Hooks

**Request ID**: dev-20260519-151734
**Created**: 2026-05-19T15:17:34Z
**Tier**: STANDARD

## Goal

Make `/do` and `/allow` actually unlock the subagent-class operations they document. Specifically: (a) the main-agent `Read` on files larger than the user-facing line limit must succeed when `/do` or `/allow Read` is in effect, instead of forcing the user to delegate to a subagent; (b) the documented `/do` carve-out for `EnterWorktree` (everything except `EnterPlanMode`/`ExitPlanMode` is unlockable) must hold.

## Context

The user invoked `/dev 全修` after a prior Codex audit confirmed that `/do` "还是出问题" (still has problems): with both a `/do` consent flag and `/allow Read` granted, `pretool-read-size-guard.py` still exit-2 blocked a 1001-line `Read` from the main agent. The user had to delegate the work to a subagent to make progress. The audit pin-pointed `pretool-read-size-guard.py` as the single high-confidence bypass-coverage gap, with `pretool-block-enterworktree.sh` flagged as a related docs/code mismatch (CLAUDE.md says `/do` unlocks everything except `EnterPlanMode`/`ExitPlanMode`, but the EnterWorktree hook blocks unconditionally). The audit also surfaced `LINE_LIMIT` documentation drift introduced by commit `9204c25` (May 7), which raised the limit from 600→1000 in an unrelated daily-trade defects batch without updating the docstring or CLAUDE.md (which both still say 600).

## Evidence (Contract A)

### Finding-1 (PATCH-1): pretool-read-size-guard.py missing /do and /allow bypass

- **Observed**: User verbatim — `/do` "还是出问题" (still has problems); had to use a subagent to read large files even with `/do` active.
- **Measured**:
  - `/root/.claude/hooks/pretool-read-size-guard.py:21-25` — exempts subagents via `data.get("agent_id")` check, returns True.
  - `/root/.claude/hooks/pretool-read-size-guard.py:63-70` — emits stderr message and `sys.exit(2)` for any non-exempt main-agent `Read` exceeding `LINE_LIMIT`.
  - The hook imports `json`, `os`, `sys` only. It does NOT import `lib.allowlist`. It does NOT read `/tmp/claude-orchestrator-consent-<sid>.flag`. It does NOT read `/tmp/claude-bash-allowlist-<sid>.json`. There is no code path in the hook that honors `/do` or `/allow`.
  - Codex independently verified by direct hook invocation: with both `/do` flag set AND `/allow Read` granted for the session, the hook still returned exit 2 on a 1001-line file. With `agent_id` set, it returned exit 0.
- **Expected**: Main-agent `Read` should be allowed when `/do` consent flag content is exactly `"true"` for the session, OR when `lib.allowlist.read_grant("Read", sid)` returns True for the session. Source: `pretool-orchestrator-gate.py:66-71` (canonical `has_consent` helper) + `pretool-orchestrator-gate.py:177-185` (canonical `/allow` short-circuit using `read_grant`). - **Gap**: Two missing checks (`/do` and `/allow`) in a hook registered globally in `settings.json:352-358`. `Read` is a subagent-class operation in this codebase (it appears in `ALWAYS_ALLOWED` in `pretool-orchestrator-gate.py:33-47`), so the gap is exactly the "subagent gets through, main agent with consent does not" parity failure that `/do` is designed to repair.

### Finding-4 (PATCH-2): pretool-block-enterworktree.sh missing /do and /allow carve-out

- **Observed**: Codex flagged docs/code mismatch.
- **Measured**:

- **Gap**: The hook embodies a safety preference (origin/main base, see hook lines 11-19) but presents it as a permanent block. This contradicts CLAUDE.md's stated permanent-block list. Reconciliation: keep the safety message; add `/do` and `/allow` bypass paths.

## Edge Cases & Risks

- **Risk 1**: `read_grant("Read", sid)` may match an over-broad `/allow` pattern (e.g., `/allow Re` would substring-match `"Read"`). Codex observation #9 notes this is broader than posttool consumption. Mitigation: this risk is pre-existing and identical across all hooks that currently call `read_grant`. We do not change `read_grant` semantics. Test ACs use exact tool-name grants only. Out-of-scope adjacent issue.
- **Risk 2**: A stale `/tmp/claude-orchestrator-consent-<sid>.flag` from a prior session with the same sid could accidentally bypass. Mitigation: the canonical `has_consent` from `pretool-orchestrator-gate.py:66-71` already addresses this by requiring file content exactly `"true"`. PATCH-1 must use the same check (not bare `flag.exists()`). Stop-cleanup hook already removes the flag at session end (`stop-cleanup-allowlist.sh:28-35`).
- **Risk 3**: `pretool-block-enterworktree.sh` switching from `CLAUDE_TOOL_NAME` env to stdin JSON could regress if any caller depends on the env-only path. Mitigation: All Claude Code PreToolUse hooks receive JSON on stdin per the canonical protocol; the existing env var is a Claude-Code-supplied convenience, not the primary contract. `pretool-write-guard.sh`, `pretool-bash-safety.sh`, and every other PreToolUse hook in the directory read stdin JSON. We additionally fall back fail-open on parse failure (same pattern as the other bash hooks).
- **Risk 4** (codex feedback #5): Some readers may expect a subagent-exemption in PATCH-2. Stated explicitly: this fix does NOT add a subagent exemption; it adds the documented `/do`/`/allow` carve-out only. Subagents remain unconditionally blocked by this hook (PATCH-2 is main-agent-only by design); subagents needing EnterWorktree must use the existing safety path `~/.claude/scripts/create-worktree.sh`. No `/do` or `/allow` path is provided for subagents in this cycle.
