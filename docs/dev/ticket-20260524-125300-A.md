# BA Specification: Three Bypass Paths in bash_context_strip.py

**Request ID**: 20260524-125300-A
**Created**: 2026-05-24T12:53:00Z
**Tier**: STANDARD

## Goal

Patch three specific code paths in `hooks/lib/bash_context_strip.py` where the
context-stripping logic fails to recognise a shell-executable context and
therefore passes a command containing dangerous tokens through to the safety
hook without flagging it. Each bypass is exploitable in real shell invocations
that are routinely generated by subagents.

## Context

`bash_context_strip.py` implements `strip_non_executable_contexts()`, which
is called by `pretool-bash-safety.sh` (line 730) to produce
`COMMAND_CONTEXT_STRIPPED`. Four grep rules in the hook run against the
stripped output rather than the raw command, on the theory that dangerous
tokens inside quoted strings / comments / heredoc bodies should not trigger.
If the stripping function fails to recognise a shell context, it leaves a
quoted payload unexpanded, and the grep patterns never see the dangerous token.

The Codex review of cycle 20260522-080646 (item A, confirmed "high recurrence")
identified three such failures. Codex exec verification at baseline
`f79fba1c1684d246928a3cfb39bf1ed7a9a5c8a5` produced the following observed
output (reproduced in the Evidence section for each bug).

Scope of fix: `hooks/lib/bash_context_strip.py` (production logic) and
`hooks/tests/test_bash_safety_context.py` (regression tests). No changes to
`pretool-bash-safety.sh` or any other file.

## Evidence (Contract A)

### BUG 1 — Combined flags in _process_shell_interp

- **Observed**: Codex Codex review finding: `bash -lc 'rm /tmp/x'` is not
  blocked; the rm token is not exposed as executable.
- **Measured**: `_process_shell_interp` at `hooks/lib/bash_context_strip.py:298`:
  ```python
  if i > cmd_idx and kind == "word" and text.strip() == "-c":
  ```
  Exact equality `== "-c"` never matches the tokenised word `-lc`, `-ec`, `-ce`,
  `-xc`, `-cx`, or `+c`.
  Codex exec verified (baseline f79fba1c): `strip_non_executable_contexts("bash -lc 'rm /tmp/x'")` returns `bash -lc 'rm /tmp/x'` — payload remains quoted, `rm` not visible.
- **Expected**: The function should enter the shell-code expansion branch
  whenever the option cluster is a short-option word (starting with `-` or `+`,
  not `--`) that contains the character `c` anywhere in the cluster.
  POSIX specifies `-c` as the command-string option; shells accept it combined
  with any other short options in any order (`-lc`, `-ec`, `-ce`, `-xc`, `+c`,
  `+ec`, etc.). Verified at runtime: `bash -lc 'printf yes'` exits 0; `bash -ce 'printf yes'` exits 0; `dash -cl 'printf yes'` exits 0.
- **Gap**: Option clusters containing `c` are never detected; the `-c` branch
  is unreachable for any combined flag form.

### BUG 2 — Wrapper command options in _find_command_word

- **Observed**: Codex review finding: `sudo -u root bash -c 'rm /tmp/x'` is
  not blocked; the rm token is not exposed.
- **Measured**: `_find_command_word` at `hooks/lib/bash_context_strip.py:219-222`:
  ```python
  if word in _WRAPPER_PREFIXES:
      idx += 1
      continue
  ```
  When `word == "sudo"`, idx advances to the next token (`-u`). On the next
  iteration, `-u` is a `"word"` kind, is not a wrapper prefix, does not match
  the assignment regex, and is not empty — so `return idx` fires immediately,
  returning `-u` as the "command word". `_basename("-u") == "-u"` which is not
  in `_SHELL_INTERPS`, so the segment is treated as a generic command and all
  quoted arguments are stripped rather than expanded as shell code.
  Codex exec verified: output is `sudo -u root bash -c ` (payload stripped).
- **Expected**: After consuming a wrapper word, `_find_command_word` must also
  consume any option-argument pairs that belong to the wrapper before resuming
  the command-word search. For `sudo`, the following options take one argument:
  `-u`/`--user`, `-g`/`--group`, `-C`/`--close-from`, `-D`/`--chdir`,
  `-H` (no argument), `--` (terminates options; next token is command).
  The fix must not over-skip: `sudo -H bash` must resolve to `bash` (not the
  token after `bash`), and `sudo -- bash` must also resolve to `bash`.
- **Gap**: Any wrapper followed by a flag-value pair shifts the command word
  one position beyond where `_find_command_word` looks.

### BUG 3 — Absolute path shell in pipe (heredoc detection)

- **Observed**: Codex review finding: `cat <<'EOF' | /bin/bash\nrm /tmp/x\nEOF`
  does not preserve the heredoc body as executable.
- **Measured**: `_process_heredocs` at `hooks/lib/bash_context_strip.py:353`:
  ```python
  shell_ctx = consumer in _SHELL_INTERPS or bool(re.search(r"\|\s*(bash|sh|zsh|dash)\b", after))
  ```
  The regex matches `| bash` but not `| /bin/bash` or `| /usr/bin/bash`.
  Codex exec verified: `strip_non_executable_contexts("cat <<'EOF' | /bin/bash\nrm /tmp/x\nEOF")`
  returns a structure where the heredoc body line is replaced with an empty
  string — rm is invisible.
- **Expected**: The pipe-to-shell detection must resolve the basename of an
  absolute-path shell and compare it to `_SHELL_INTERPS`, the same membership
  set used everywhere else in the module. The implementation should not use a
  broader regex (risk of matching `/tmp/bash.old` or similar).
  Preferred approach: parse `after` with the existing `_first_word()` helper,
  which already calls `_basename()`, then test `_first_word(after_pipe_segment)
  in _SHELL_INTERPS`. This reuses proven code rather than adding a new regex.
- **Gap**: Heredoc bodies fed to any absolute-path shell interpreter are silently
  classified as non-executable and zeroed out, bypassing all four danger-token
  rules.

## Requirements (MoSCoW)

### Must Have

- M1: Fix `_process_shell_interp` to enter the shell-code expansion branch
  when the option word is any short-option cluster (starting with `-` or `+`,
  not `--`) that contains the character `c`. Do not require `c` to be the last
  or first character.
- M2: Fix `_find_command_word` to skip wrapper-option arguments for `sudo`
  specifically. The minimal required cases are: `-u`/`--user <arg>`,
  `-g`/`--group <arg>`, `--user=<arg>` (inline), `-H` (no argument), `--`
  (terminates option parsing). The fix must not over-skip: `sudo -H bash -c ...`
  must resolve to `bash`, and `sudo -- bash -c ...` must resolve to `bash`.
- M3: Fix `_process_heredocs` pipe-to-shell detection to recognise
  absolute-path shell interpreters by calling `_first_word()` on the segment
  after `|` and comparing its result (which calls `_basename()`) to
  `_SHELL_INTERPS`. The existing regex `r"\|\s*(bash|sh|zsh|dash)\b"` may be
  kept as a fast-path; the basename-check is added as a fallback or replacement.
- M4: Add regression tests in `hooks/tests/test_bash_safety_context.py`
  covering all cases listed in the Acceptance Criteria.

### Should Have

- S1: The wrapper-option arity table is defined as a module-level constant or
  mapping so future wrappers can be added without touching `_find_command_word`
  logic.

### Could Have

  `ionice -c <n>` (all take numeric or string arguments). This is an

  observation.

- Rewriting `strip_non_executable_contexts()` or any function not directly

## Edge Cases & Risks

- **BUG 1 over-broad detection**: a flag like `-rc` (rsync flags) could in
  theory be parsed as "contains c → treat as shell invocation". Mitigation:
  BUG 1 only fires inside `_process_shell_interp`, which is already guarded by
  `cmd_word in _SHELL_INTERPS` at the call site. The shell interpreter itself
  must already be identified before any flag is tested.
- **BUG 2 over-skip**: consuming the argument of a flag that takes no argument
  would skip the actual shell command word, causing the segment to be treated
  as generic (payload stripped, no detection). AC4 (sudo -H case) and AC5
  (sudo -- case) directly test this regression.
- **BUG 3 false negative**: `| /tmp/bash.old` must NOT be classified as a
  shell context. The `_first_word()` + `_SHELL_INTERPS` membership approach
  handles this correctly because `_basename("/tmp/bash.old") == "bash.old"`
  which is not in the set. AC3 tests the positive; the non-shell-name case
  is implied by membership semantics.
- **Fail-closed invariant**: `strip_non_executable_contexts()` wraps the entire
  call in `try/except Exception: return cmd`. The fix must not introduce any
  new exception path that would silently fall back to returning the raw command.
  Dev must verify no new exception paths are introduced.

## Acceptance Criteria

### AC1: Combined flag -lc exposes rm

- GIVEN a Bash command `bash -lc 'rm /tmp/x'`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&(])rm\s'`
  (rm is visible as executable text, not hidden inside a quoted token)

### AC2: Combined flag -ce exposes rm

- GIVEN a Bash command `bash -ce 'rm /tmp/x'`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&(])rm\s'`

### AC3: sudo -u root shifts to bash, rm exposed

- GIVEN a Bash command `sudo -u root bash -c 'rm /tmp/x'`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&(])rm\s'`

### AC4: sudo -H (no-argument flag) does not over-skip — rm exposed

- GIVEN a Bash command `sudo -H bash -c 'rm /tmp/x'`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&(])rm\s'`
  (verifies the fix does not over-skip by treating -H's "next token" as its
  argument)

### AC5: sudo -- terminates options — rm exposed

- GIVEN a Bash command `sudo -- bash -c 'rm /tmp/x'`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&(])rm\s'`

### AC6: Heredoc piped to /bin/bash preserves body

- GIVEN a multi-line command `cat <<'EOF' | /bin/bash\nrm /tmp/x\nEOF`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result contains `rm /tmp/x` (heredoc body preserved as executable)

### AC7: Heredoc piped to /usr/bin/bash preserves body

- GIVEN a multi-line command `cat <<'EOF' | /usr/bin/bash\nkill -9 1234\nEOF`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&])(kill)[ \t]+-'`

### AC8: Non-shell absolute path in pipe does NOT preserve body

- GIVEN a multi-line command `cat <<'EOF' | /tmp/bash.old\nrm /tmp/x\nEOF`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result does NOT match `grep -qE '(^|[ \t;|&(])rm\s'`
  (basename "bash.old" is not in _SHELL_INTERPS; body is stripped)

### AC9: Existing tests still pass

- GIVEN the full test suite `hooks/tests/test_bash_safety_context.py`
- WHEN run with `python3 -m pytest hooks/tests/test_bash_safety_context.py -v`
- THEN all existing tests pass with exit code 0

### AC10: +c form exposes rm (POSIX-compliant variant)

- GIVEN a Bash command `bash +c 'rm /tmp/x'`
- WHEN `strip_non_executable_contexts()` processes it
- THEN the result matches `grep -qE '(^|[ \t;|&(])rm\s'`

## Technical Hints

- **Affected files**: `hooks/lib/bash_context_strip.py`, `hooks/tests/test_bash_safety_context.py`
- **BUG 1 change location**: `_process_shell_interp` line 298 — replace
  `text.strip() == "-c"` with a predicate that checks: the token is a
  short-option word (starts with `-` or `+`, not `--`), and `"c"` is in
  the option characters (i.e., `"c" in text.strip()[1:]`).
- **BUG 2 change location**: `_find_command_word` lines 219-222 — after
  detecting `word in _WRAPPER_PREFIXES`, add a loop that consumes option
  tokens belonging to the wrapper. For `sudo`, options taking one argument
  are: `-u`, `--user`, `-g`, `--group`, `-C`, `--close-from`, `-D`,
  `--chdir`. `--user=<arg>` (inline) and `-H` (no arg) and `--` (terminator)
  must also be handled. The simplest safe approach: define a module-level
  `_SUDO_OPTS_WITH_ARG = frozenset({"-u","--user","-g","--group","-C","--close-from","-D","--chdir"})` and after the wrapper-skip, consume tokens while the next non-space
  token starts with `-` or `+` (but is not `--`): if it is in
  `_SUDO_OPTS_WITH_ARG`, skip it and skip one more token; if it is `--`,
  skip it and stop; otherwise skip the flag only.
- **BUG 3 change location**: `_process_heredocs` line 353 — replace or
  augment the pipe-detection. Preferred pattern: after the existing regex
  fast-path, also check whether `_first_word(after)` is in `_SHELL_INTERPS`
  (which calls `_basename()` internally). Note that `after` is the text
  after the heredoc `<<EOF` marker on the same line, which may contain
  `| /bin/bash`. `_first_word()` tokenises and returns the first command
  word with its basename resolved — exactly what is needed.
- **Verification pattern**: each AC is testable in isolation via pytest in
  `hooks/tests/test_bash_safety_context.py` using the existing `grep_match`
  helper already present in that file.
- **Verification harness cleanup contract (R10 cross-reference)**: no
  background processes are spawned by the test harness; cleanup contract
  is N/A for pytest-based tests.
