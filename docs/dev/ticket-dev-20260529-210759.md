# BA Specification: Fix Layer 1.F False Positive — Protected Name in Quoted String Arg

**Request ID**: dev-20260529-210759
**Created**: 2026-05-29T21:07:59Z
**Tier**: SMALL

---

## Goal

Fix `hooks/pretool-bash-safety.sh` Layer 1.F so that a command whose quoted string argument text happens to contain the substring `write-bulk-commit-sentinel.py` or `/tmp/claude-bulk-commit-sentinel-` is not blocked when that string is only argument data to an unrelated command (e.g., `codex exec --prompt "..."` or `graphify-query.py --requirement "..."`).

---

## Context

Layer 1.F (`bulk-commit-sentinel-write` rule) was added in task 20260524-205206 to prevent any model tool call that writes or interacts dangerously with the bulk-commit sentinel. Its entry gate uses raw `$COMMAND` substring matching:

```bash
if echo "$COMMAND" | grep -qF '/tmp/claude-bulk-commit-sentinel-' \
   || echo "$COMMAND" | grep -qF 'write-bulk-commit-sentinel.py'; then
```

`$COMMAND` is the verbatim command string — it includes quoted argument values. The gate fires on any command that mentions either pattern anywhere, including inside `--prompt "..."` or `--requirement "..."` string arguments.

A context-stripping facility (`hooks/lib/bash_context_strip.py`) already exists and is invoked later in the hook (around line 1091) to produce `COMMAND_CONTEXT_STRIPPED`. The stripper removes quoted string argument content for non-shell, non-danger commands, leaving only the executable structure. Layer 1.F runs before the strip block is reached, so it cannot currently use the stripped view.

During self-review the bug was reproduced at BA level: attempting to invoke `codex exec` with the script name in a shell variable assignment was itself blocked — demonstrating the false positive extends to `VAR="...write-bulk-commit-sentinel.py..."` patterns too.

## Root Cause Deep-Dive

The Layer 1.F entry gate at lines 669-670 uses `grep -qF` (fixed-string substring match) against `$COMMAND` — the unprocessed raw command string from the hook JSON payload. This string contains the full bash command including argument quoting syntax and argument values verbatim.

The existing `bash_context_strip.py` library, when applied to a command like `codex exec --prompt "...write-bulk-commit-sentinel.py..."`, processes it as follows:
1. `_process_segment` is called.
2. `_find_command_word` identifies `codex` as the command word.
3. `codex` is not in `_SHELL_INTERPS`, `_SCRIPT_INTERPS`, or `DANGER_COMMANDS`.
4. Generic path (lines 316-330): for each token after `cmd_idx`, single-quoted tokens → `""`, double-quoted tokens → `_strip_double_content(...)` (only preserving `$(...)` and backtick subshell contents inside double quotes).
5. The `--prompt "...write-bulk-commit-sentinel.py..."` double-quoted token is collapsed to `""`.
6. Result: `codex exec --prompt ` — the protected name is absent.

Similarly for `graphify-query.py --requirement "..."`: `graphify-query` (the basename after stripping `.py`) is not in `_SCRIPT_INTERPS`, so generic path applies.

The fix is to move the context-stripping block to before Layer 1.F, and change the Layer 1.F entry gate to use `COMMAND_CONTEXT_STRIPPED` instead of `$COMMAND`.

**Security invariant preserved**: Commands that actually execute the sentinel writer through indirection (e.g., `python3 -c "...write-bulk-commit-sentinel.py..."`) are handled by `_process_shell_interp`, which detects the `-c` flag and exposes the quoted payload as executable text — the protected name remains visible in the stripped output and the entry gate still fires. Likewise `bash -c "python3 scripts/write-bulk-commit-sentinel.py"` exposes the payload.

**Self-review adversarial questions answered**:
1. **`VAR='...write-bulk-commit-sentinel.py...'` as leading variable assignment**: The stripper (`_find_command_word`) skips leading `VAR=...` assignments to find the real command word. For a command that starts with only an assignment and no subsequent command (pure `PROMPT='...'`), the result is an empty command — no protected name in the executable part. The entry gate would not fire on the stripped form. GOOD.
2. **`bash -c "codex exec --prompt '...write-bulk-commit-sentinel.py...'"` nesting**: `bash -c` causes `_process_shell_interp` to expose the double-quoted payload `codex exec --prompt '...write-bulk-commit-sentinel.py...'` as executable text. The single-quoted `'...write-bulk-commit-sentinel.py...'` content is thus in the stripped output. The entry gate fires on the stripped form. GOOD — real nested execution stays blocked.
3. **Unterminated-quote commands**: The `_process_compound`/`_consume_quoted` scanners advance defensively; on exception, `strip_non_executable_contexts` returns the raw command (fail-closed).

## Edge Cases & Risks

- **Risk**: The context-strip block has a 2-second `timeout` and `ulimit -v` wrapper. Moving it earlier means it runs before the daemon-restart layers (1.A–1.E). Those layers all use raw `$COMMAND`, so they are unaffected. The only effect of moving the strip block earlier is that `COMMAND_CONTEXT_STRIPPED` becomes available for Layer 1.F. The stripping is bounded and fail-closed (on failure, raw command is used).
- **Risk**: The M5 canonical venv-activate form (`source venv/bin/activate && python3 /root/.claude/scripts/write-bulk-commit-sentinel.py`) is checked via bash `[[ =~ ]]` against raw `$COMMAND` BEFORE the Python compound-detection helper. This check runs inside the Layer 1.F block after the entry gate. If the entry gate now uses the stripped view, the M5 check still sees raw `$COMMAND` (unchanged). No regression.
- **Risk**: `COMMAND_CONTEXT_STRIPPED` is initialised to `"$COMMAND"` (line 1091: `COMMAND_CONTEXT_STRIPPED="$COMMAND"`). After the strip block runs, if stripping fails, `COMMAND_CONTEXT_STRIPPED` equals raw `$COMMAND`. This is the desired fail-closed behaviour.
