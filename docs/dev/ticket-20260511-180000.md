## Requirements (MoSCoW)

### Must Have

- **M1 — Hash chain trailer**:
  - At commit-tree time and BEFORE `git commit-tree` is invoked at `hooks/commit.sh:1978`, compute SHA-256 (lowercase hex, 64 chars) of each of these 6 artifacts that EXISTS at run time:
    1. `${DOCS_DIR}/ticket-${TASK_ID}.md` (legacy `ba-spec-${TASK_ID}.md` accepted via the same lookup chain `commit.sh:554-555` already implements)
    2. `${DOCS_DIR}/context-${TASK_ID}.json`
    3. `${DOCS_DIR}/dev-report-${TASK_ID}.json`
    4. `${DOCS_DIR}/qa-report-${TASK_ID}.json`
    5. `${DOCS_DIR}/close-report-${TASK_ID}.md`
    6. `${DOCS_DIR}/completion-${TASK_ID}.md`
  - Hash source MUST be the file contents on disk at commit-tree time (the artifact-as-archived). Do NOT hash a private-index snapshot or a tree blob; hash the raw bytes of the file as the operator/agents authored them so post-commit tampering is detectable by reading the file and rehashing.
  - Append the hashes as a single trailer paragraph at the END of the existing commit message body. Trailer format MUST be git-interpret-trailers compatible:
    - One blank line separates the last body line from the first trailer line.
    - NO blank lines between trailer entries (codex-required correction; the earlier draft said "blank-line-separated block" which was ambiguous).
    - Trailer token names use Title-Case-Hyphenated form: `Ticket-SHA256`, `Context-SHA256`, `Dev-Report-SHA256`, `QA-Report-SHA256`, `Close-Report-SHA256`, `Completion-SHA256`.
    - Value is the 64-hex SHA-256 digest (user's prompt said "40-hex" — that is the SHA-1 length; SHA-256 is 64 hex chars; this BA is using 64-hex per cryptographic correctness, surfaced as a clarification gap in the user prompt).
  - Ordering: **task-flow** order (Ticket → Context → Dev-Report → QA-Report → Close-Report → Completion). NOT lexicographic. Ordering is deterministic regardless of which artifacts are present (omitted artifacts are skipped in place; present artifacts keep their slot in the sequence).
  - Coexistence with existing trailers (e.g., operator-added `Signed-off-by:` if it ever appears, future-proofing): the 6 audit trailers MUST be APPENDED into the same final trailer paragraph as any pre-existing trailer block, NOT placed in a separate paragraph below it. Implementation MUST use `git interpret-trailers --in-place --if-exists replace --if-missing add` (NOT `addIfDifferent` — codex / QA OBJ-1 correction; see Iter-1 note at file foot). Rationale: `addIfDifferent` STACKS old + new values when an artifact's content changes between runs, defeating M1's tamper-evidence property (the trailer block would show both the stale and the fresh hash). `replace` keeps exactly one occurrence per token.
  - Dedup hardening (M1.regex_dedup, QA minor + iter-1 codex round 2 F1): `git interpret-trailers --in-place --if-exists replace` replaces only ONE occurrence per key. If a prior buggy run or operator-pasted message contains ≥2 trailer lines with the same audit-token name (e.g., two `Ticket-SHA256:` lines), `replace` leaves 1 stale + 1 fresh. Implementation MUST therefore pre-strip ALL pre-existing trailer lines whose token name is in the 6-audit-token set BEFORE invoking `interpret-trailers`. The pre-strip step is a single awk/sed pass over the temp message file that drops lines matching the SINGLE CANONICAL POSIX regex `^(Ticket|Context|Dev-Report|QA-Report|Close-Report|Completion)-SHA256:[[:space:]]*` from the message body (case-sensitive on the token; the `[[:space:]]*` tail accommodates both `Token: <value>` (with whitespace) and `Token:<value>` (no whitespace — codex F1 noted this case was missed by `\s` and `[[:space:]]` tail-anchors that required ≥1 whitespace; the canonical form uses `[[:space:]]*` so zero-or-more whitespace matches; trailer-paragraph location is irrelevant — strip globally). Post-strip, `interpret-trailers` is invoked to append the fresh trailers exactly once per token.
  - Idempotency: if `CLAUDE_COMMIT_SKIP_HASH_TRAILERS` is unset and the wrapper is invoked twice in succession on an unchanged tree (re-run scenario), the resulting trailer block MUST be identical — no duplicate `Ticket-SHA256:` lines, no stacked stale+new values. The pre-strip + `--if-exists replace` combination is what guarantees this.
  - The same SHA-256 values MUST be written to the audit JSON's `artifact_sha256` field as an object mapping `{artifact_basename: hex_digest}` so cross-verification (trailer ↔ audit JSON) is possible.
  - If an artifact does NOT exist at run time, the corresponding trailer is OMITTED (NOT emitted with a zero/empty value); the `artifact_sha256` map likewise omits the key.
  - **Bridge mode (`--auto-bulk-bridge`)**: artifact set is empty by construction → emit NO hash trailers (and emit NO blank-line trailer separator either — bridge messages stay byte-identical to the existing BLESSED_BRIDGE_RE format; this is a true no-op).
  - Operator override env `CLAUDE_COMMIT_SKIP_HASH_TRAILERS=1` disables trailer emission entirely (the audit JSON still records `artifact_sha256` for forensic value); this exists for the rare case the message body must stay precisely as `-m` supplied (e.g., a downstream test fixture that pins exact bytes).

## Edge Cases & Risks

- **M1 idempotency** (iter-1 hardening): on retry after a CAS failure, the wrapper must NOT keep stacking trailers. The implementation MUST (a) pre-strip ALL pre-existing trailer lines whose token name is in the 6-audit-token set (case-sensitive on the token), THEN (b) invoke `git interpret-trailers --in-place --if-exists replace --if-missing add`. The combination guarantees exactly-one-occurrence-per-token regardless of the pre-strip input state. Pre-strip is necessary because `--if-exists replace` only replaces ONE occurrence; a message with two stale `Ticket-SHA256:` lines would emerge from `replace` with 1 stale + 1 fresh.
- **M1 trailer parse compatibility**: downstream tools that use `git log --format=%(trailers:key=Ticket-SHA256)` rely on the trailer paragraph being the FINAL paragraph of the message body. If a future operator pastes prose AFTER the trailer block, git's trailer parser will not see the audit trailers. This is a known git-trailer constraint, not a defect of M1; document in DOC-14.
- **M1 hash source choice**: hashing file-on-disk (this BA's choice) vs hashing the tree-blob means a developer who stages an artifact, then mutates the working-tree file before `git commit-tree`, produces a trailer that matches the worktree, NOT the committed tree. This is the desired property for the audit use case (the worktree file is what the agents read and reason over), but DOC-14 must call this out so a forensic reader knows what to rehash.
- **M2 false-positive risk** (mitigated by codex-revised regex): the original draft's bare `\bPASS\b` / `\bverified\b` triggers on "I PASS this baton" / "verified the schema in code review". The codex-tightened set requires QA/AC/tests context proximity, eliminating these. Residual risk: "QA: PASS" appearing in a quoted historical context (e.g., quoting a prior commit message in the new -m) still triggers — operator MUST use `CLAUDE_COMMIT_SKIP_MESSAGE_GUARD=1` for that rare case.
- **M2 status sentinel handling**: codex flagged that `.qa.status` missing/invalid/unknown all need to refuse when a verification regex matches. Without this, a malformed qa-report silently lets the message through. The four sentinels (`__absent__`, `__invalid__`, `__unknown__`, plus literal `"fail"`) all refuse uniformly.
- **M3 revert subject limitation**: git's default `Revert "..."` won't match. Documented as known constraint in DOC-16. Operators wanting clean audit trail use `revert: ...` form.
- **S1 tmpfs-symlink trap** (iter-1: hard-default chosen; warn-and-keep removed): codex's catch. `$HOME/.claude/logs` is itself tmpfs in this env. The iter-1 spec hard-picks `/var/lib/claude/commit-audit/` as the default — filesystem-agnostic (every modern Linux has `/var/lib/`), persistent (FHS variable state convention), and ext4-backed in this environment (QA confirmed `/` is ext4 on /dev/sda1 with 460GB free). There is NO warn-and-keep code path: a tmpfs-resolving `CLAUDE_LOG_DIR` triggers exit-2 refusal unless the operator explicitly sets `CLAUDE_AUDIT_PERSIST_DISABLED=1` to acknowledge loss-on-reboot.
- **AC9 dogfood**: this cycle's own commit (the dot-claude commit landing M1+M2+M3+S1 changes) will exercise the new code paths. If M1/M2/M3 reject the commit message dev writes, the cycle reveals the bug at close time — desirable dogfood property.
