## Goal

Integrate Graphify (a code-to-knowledge-graph tool) into the existing Claude Code multi-agent orchestration system as a dual-touchpoint architecture. The integration adds two new phases to the /dev workflow: Step 1.5 (deterministic pre-BA graph pre-query via Bash, injecting structural context before BA analysis) and Step 7.5 (graphify subagent enrichment after BA-QA validation, before DEV). This solves two problems: DEV lacks structural codebase context during implementation, and BA suffers from confirmation bias when interpreting ambiguous user requirements that reference existing code structures.

---

## Context

The existing /dev workflow runs BA → QA validates BA → DEV → QA. No structural codebase context (import chains, module topology, function call graphs) is available during BA analysis or DEV implementation. This creates two failure modes documented in the 5-round Claude+Codex consensus that produced this spec. The integration is additive: all new components are advisory/fail-open; no existing code path fails if Graphify is absent. The implementation spans 3 PRs and 32 files. Sections 1-4 of the spec are not yet populated (first cycle).

## OBJ-4 Investigation: commands/dev-overnight.md

**QA objection**: commands/dev-overnight.md is absent from PR2 affected-files; spec says "/dev-overnight: enabled, shares global update".

**Investigation performed**: Read commands/dev-overnight.md (1807 lines). Key findings:

1. dev-overnight.md does NOT have a workflow structure analogous to /dev's sequential Steps 1–15. It uses a BA-dispatch-driven pipeline model (Steps 8-12) where BA subagents run per-discovered-pipeline, not a single user-requirement flow.

2. The BA dispatch in overnight Step 8 (lines 889-944) constructs subagent prompts but does NOT include a pre-BA Bash call equivalent to /dev's Step 1.5 graphify-query.py call. The overnight BA receives the user requirement via the pipeline description, not via a Bash-hydrated pre_query.json.

3. "enabled, shares global update" means: overnight sessions reuse the global graphify cache maintained by graphify-maintain.py; they do NOT need a separate Step 1.5 Bash call per pipeline. The graphify subagent enrichment (Step 7.5 equivalent) can run per pipeline in the Step 12 (Dev dispatch) phase.

4. The critical gap that DOES require commands/dev-overnight.md modification: Step 1's sentinel fanout loop (lines 225-267) creates per-agent-type JSON files in dev-registry. If 'graphify' is absent from this list, graphify subagent dispatch in overnight mode will fail the same way as in /dev. This is partially addressed by prompt-workflow.py agent_types fix (PR1), but the overnight Step 1 explicit Bash form (lines 248-264) also enumerates agent types literally and needs 'graphify' added.

**Conclusion**: commands/dev-overnight.md MUST be added to PR2 scope for two modifications:
- Step 1 sentinel fanout: add `graphify` to the agent-types list in the literal Write/printf forms (lines 248-263)
- No Step 1.5 insertion needed: overnight BA discovers pipelines independently; graphify enrichment integrates via the Step 7.5-equivalent graphify subagent dispatch added alongside Dev dispatch

This is a PR2 modification (behavior change — enabling graphify in overnight) not PR3 (maintenance).
