## Context

The `.hook-refactor-allow` sentinel is a per-project empty-file mechanism signaling user authorization for refactoring `.claude/hooks/` and `.claude/commands/`. It was designed to be created by hand before a refactor session and deleted immediately after. On 2026-04-27, four user-authorized push.sh editing sessions used this mechanism; the consumed sentinels are preserved in `/tmp/.hook-refactor-allow-consumed-*`. The primary sentinel at project root was added to `.gitignore` (commit `76f9124`, 2026-04-27) to suppress `git status` noise, which has the side-effect of making stale residue invisible.

## Assessment

**The April-27 root-level file is STALE and HIGH-RISK.**

1. **Stale**: 25 days past last legitimate use; all 4 authorized sessions explicitly intended to delete it.
2. **High-risk via pretool-block-production-files.sh**: The `$HOME/.claude/.hook-refactor-allow` fallback in that hook resolves — via symlink — to the stale file. This silently enables any subagent on this system to edit `pretool-bash-safety.sh`, `pretool-block-production-files.sh`, and `pretool-orchestrator-prompt-purity.py` — the three hook files that form the security enforcement floor.
3. **Partially mitigated**: `pretool-claude-config-guard.py` is NOT bypassed when `CLAUDE_PROJECT_DIR=/dev/shm/dev-workspace/dot-claude` (the normal working session), because it checks the `.claude/` subdirectory path which does not contain the file. But the bypass IS active when `CLAUDE_PROJECT_DIR=/root`.
4. **Gitignore masking**: `.gitignore:73` suppresses the file from `git status`, making this invisible during normal repo inspection.

**Codex adversarial review** confirmed this assessment and added: deletion alone is insufficient; the `$HOME/.claude/.hook-refactor-allow` fallback in `pretool-block-production-files.sh` is a structural weakness that makes any persistent empty file at `$HOME/.claude/` path (or anything symlinking there) a permanent bypass. This is a legitimate concern logged in out-of-scope observations for a follow-up cycle.

**Immediate action (R6 — user decision required)**:

1. **Delete** `/dev/shm/dev-workspace/dot-claude/.hook-refactor-allow` — the stale root-level April-27 file. This re-engages `pretool-block-production-files.sh`'s HOME check for subagent hook-self-modification.
2. **Review and delete or refresh** per-project sentinels:
   - `/dev/shm/dev-workspace/happy-dev/.claude/.hook-refactor-allow` (15 days old — confirm if active session)
   - `/root/multi-asset-portfolio/.claude/.hook-refactor-allow` (21 days old — confirm if active session)
   - `/root/happy-dev.bak/.claude/.hook-refactor-allow` (bak dir — delete)
3. **Delete anomalous file**: `/dev/shm/dev-workspace/dot-claude/.claude/.hook-refactor-allow。` (Unicode suffix, inert but confusing artifact).

**Follow-up hardening (separate cycle)**:
- Narrow or remove the `$HOME/.claude/.hook-refactor-allow` fallback from `pretool-block-production-files.sh:133` — it creates a permanent global bypass surface for any file placed at that location. Replace with per-project check only, consistent with `pretool-claude-config-guard.py`.

**No destructive git operations are required. File deletion only.**

## Edge Cases & Risks

- **Catch-22**: Deleting the file via shell `rm` from within a subagent session may itself be blocked by `pretool-bash-safety.sh` if `rm` on sentinel files is restricted. Recommend using Python `os.remove()` (as documented in worktrees spec `spec-20260424-090315.md:122` which notes this pattern).
- **Concurrent sessions**: If an active overnight or dev session is running with the sentinel intentionally present for a current hook refactor, deleting it mid-session would block pending edits. Confirm no active session before deleting.
- **Happy-dev sentinel**: mtime 2026-05-07 is 15 days ago. If an active happy-dev cycle is still open (unlikely), deleting prematurely would block hook edits in that project.
