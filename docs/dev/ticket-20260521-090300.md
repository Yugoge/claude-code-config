The primary sentinel at project root was added to `.gitignore` (commit `76f9124`, 2026-04-27) to suppress `git status` noise, which has the side-effect of making stale residue invisible.

## Assessment

**The April-27 root-level file is STALE and CRITICAL-RISK. All four live per-project sentinels are HIGH-RISK.**

1. **Primary stale file (CRITICAL)**: 25 days past last legitimate use; all 4 authorized sessions explicitly intended to delete it. The `$HOME/.claude/.hook-refactor-allow` fallback in `pretool-block-production-files.sh` means this file enables any subagent on this system — regardless of project — to edit the three hook files forming the security enforcement floor.
2. **Per-project sentinels (HIGH)**: Happy-dev (15d), travel-planner (16d), multi-asset-portfolio (21d), and orchestra (12d) sentinels activate `pretool-block-production-files.sh` bypass for any subagent running in those project contexts. These are NOT lower-risk than the primary file — they differ only in scope (per-project-session vs. system-global).
3. **Config-guard partially mitigated**: `pretool-claude-config-guard.py` enforcement gate at lines 176-177 is NOT triggered in normal dot-claude sessions (the checked `.claude/` subdirectory path does not contain the file). But is triggered when `CLAUDE_PROJECT_DIR=/root` or when a per-project sentinel exists for sessions in those projects.
4. **Gitignore masking**: `.gitignore:73` suppresses the primary file from `git status`, making stale residue invisible during normal repo inspection.
5. **Removal attempts failed**: 9 removal-record files in `/tmp/` show automated cleanup attempts between Apr 29 and May 4 — none succeeded in removing the primary sentinel.

**Codex adversarial review** confirmed this assessment and added: deletion alone is insufficient; the `$HOME/.claude/.hook-refactor-allow` fallback in `pretool-block-production-files.sh` is a structural weakness. This is logged in out-of-scope observations for a follow-up cycle.
