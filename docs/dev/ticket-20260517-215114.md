## Goal

Add an agentic intelligent-judgment layer to `/push`, `/merge`, and `/pull` commands, mirroring the architecture `/commit` uses via `changelog-analyst`. Each command gets a dedicated analyst subagent that inspects git state before (or after, for pull) execution and provides a structured verdict that the orchestrator body acts on.

---

## Context

`/commit` has four load-bearing components: (1) close-gate pre-authorization, (2) commit grant written by `/commit`, (3) `changelog-analyst` subagent that does the real git work, (4) push-gate token written post-commit. `/push`, `/merge`, and `/pull` today run wrapper scripts directly with no agentic analysis step. The user wants the same intelligent-judgment pattern applied to all three.

The architect specialist has already confirmed the four-component model and key constraints: gates must live in command body `.md` files (not in wrapper scripts), each command needs a dedicated analyst (not a shared fat agent), and the two authorization chains for `/push` must stay independent.

## Architecture Design

### /push: push-analyst (pre-execution gate)

**New orchestrator flow in commands/push.md:**

```
Step 0: Parse arguments — resolve REMOTE (using same fork-prefer-origin logic as push.sh)
Step 1: Resolve push-gate token (Chain A — existing, unchanged)
Step 2: Compute pre-push state snapshot:
        pre_head = git rev-parse HEAD
        branch = git rev-parse --abbrev-ref HEAD
        resolved_remote = <resolved remote name>
        remote_url = git remote get-url <resolved_remote>
        Generate request_id (nonce via python secrets.token_hex(16))
Step 3: Dispatch push-analyst subagent with:
        branch, pre_head, remote_name, remote_url, request_id
Step 4: Read push-analyst grant at:
        /tmp/agentic-commit/push-analyst/<repo-hash>/<session-id>/<nonce>.json
        Validate: exists, not expired, nonce matches request_id, branch matches,
                  head_sha matches pre_head, remote matches
        Consume grant (unlink)
        If verdict=blocked: abort with risks[] displayed
        If verdict=warn: display risks[], proceed
        If verdict=approved: proceed
Step 5: Call push.sh (unchanged) — Chain A push-gate token is consumed here
```

**push-analyst analysis scope:**
- Read commits-to-push range: `refs/remotes/<resolved_remote>/<branch>..HEAD` (where `<resolved_remote>` is the remote resolved by fork-prefer-origin logic — either "fork" or "origin" or an explicit arg, matching push.sh:38-42 exactly). For first push where no remote tracking ref exists, use all commits in `git log HEAD`.
- Check file names in committed blobs for sensitive patterns (.env, *.key, *.pem, *password*, *secret*, *credential*)
- Check ahead/behind count vs resolved remote (NOT vs @{u} which may track a different remote)
- Check for divergence (would push fail with non-fast-forward?)
- Check branch name vs protection rules (main/master direct push)
- Write grant: `{verdict, risks[], analysis_summary, branch, head_sha, remote_name, remote_url, session_id, nonce, expires_at (120s)}`

**Grant path:** `/tmp/agentic-commit/push-analyst/<repo-hash>/<session-id>/<nonce>.json`

**Two independent chains remain separate:**
- Chain A: push-gate token (SHA identity — did changelog-analyst produce this commit?) — consumed by push.sh
- Chain B: push-analyst grant (content review — did analyst inspect what is being pushed?) — consumed by commands/push.md before calling push.sh

### /merge: merge-analyst (pre-execution gate, 60s expiry)

**New orchestrator flow in commands/merge.md:**

```
Step 0: Resolve branch name (existing logic, unchanged — orchestrator resolves from
        overnight-state or explicit arg; NO filesystem guessing)
Step 1: Compute pre-merge state snapshot:
        resolved_branch = <the branch to merge>
        source_tip = git rev-parse refs/heads/<resolved_branch>
        default_branch = <resolved via derive-default-branch.sh>
        default_tip = git rev-parse refs/heads/<default_branch>
        Generate request_id (nonce)
Step 2: Dispatch merge-analyst subagent with:
        resolved_branch, source_tip, default_branch, default_tip, request_id
Step 3: Read merge-analyst grant at:
        /tmp/agentic-commit/merge-analyst/<repo-hash>/<session-id>/<nonce>.json
        Validate: exists, not expired (60s), nonce matches, branch names match,
                  source_tip matches, default_tip matches
        Revalidate tips: re-read current source_tip and default_tip; abort if changed
        Consume grant (unlink)
        If verdict=blocked: abort with risks[] displayed
        Else: proceed
Step 4: Call merge.sh "<resolved_branch>" (unchanged)
```

**merge-analyst analysis scope:**
- Read commit divergence: `git log --oneline <default_branch>..<resolved_branch>` (incoming commits)
- Read diff stat: `git diff --stat <default_branch>...<resolved_branch>`
- Check for conflict markers in diff (via `git diff -U0 <default_branch>...<resolved_branch>`)
- Verify overnight-state consistency: read overnight-state-*.json to confirm branch matches (validation only, NOT branch selection)
- Check unresolved merge state (`.git/MERGE_HEAD` presence)
- Write grant: `{verdict, risks[], analysis_summary, branch, source_tip, default_branch, default_tip, session_id, nonce, expires_at (60s)}`

**Grant path:** `/tmp/agentic-commit/merge-analyst/<repo-hash>/<session-id>/<nonce>.json`

**Critical constraint:** merge-analyst does NOT select or infer the branch — it receives the already-resolved branch from commands/merge.md.

### /pull: pull-analyst (post-execution, advisory only)

**New orchestrator flow in commands/pull.md:**

```
Step 0: Capture pre_pull_head = git rev-parse HEAD
Step 1: Run pull.sh (unchanged); capture exit code
Step 2: Capture post_pull_head = git rev-parse HEAD
        (ALWAYS capture, regardless of pull.sh exit code — pull.sh prints
        "Pull completed successfully" at line 154 and changes HEAD BEFORE
        running stash pop at lines 157-174; stash pop failure causes exit 1
        but HEAD has already changed)
Step 3: If pre_pull_head == post_pull_head:
           If pull.sh exit non-0: display pull.sh error, stop.
           Else: Print "No HEAD change (pull was no-op or cancelled). Skipping pull-analyst."
           Stop
Step 4: Determine pull_exit_phase:
           If pull.sh exit 0: pull_exit_phase = "success"
           If pull.sh exit non-0:
             Check rebase state directory presence:
               If .git/rebase-merge OR .git/rebase-apply exists:
                 pull_exit_phase = "rebase_conflict"
                 (git rebase partially applied commits before stopping;
                  HEAD may or may not have changed — rebase state dir is the
                  primary discriminant, not HEAD equality)
               Else if .git/rebase-merge and .git/rebase-apply do NOT exist
                    AND pre_pull_head != post_pull_head:
                 pull_exit_phase = "stash_restoration_failed"
                 (rebase succeeded and HEAD advanced; stash pop subsequently
                  failed at pull.sh line 174; no rebase state remains)
               Else (no rebase state dir AND pre_pull_head == post_pull_head):
                 pull_exit_phase = "rebase_conflict"
                 (unknown non-zero failure before rebase committed any commits;
                  classify as rebase_conflict for safety)
Step 5: Dispatch pull-analyst subagent with:
           pre_pull_head, post_pull_head, branch, pull_exit_phase
Step 6: Display pull-analyst report to user
        (pull-analyst report includes a prominent stash warning when pull_exit_phase="stash_restoration_failed":
        "NOTE: git pull --rebase succeeded but stash restoration failed — local changes remain in stash@{0}; run `git stash pop` after resolving any conflicts")
```

**pull-analyst analysis scope:**
- Read new commits: `git log --oneline <pre_pull_head>..<post_pull_head>`
- Read diff stat: `git diff --stat <pre_pull_head>..<post_pull_head>`
- Flag semantic risks:
  - Merge commits in range (non-linear history)
  - Changes to dependency manifests (package.json, requirements.txt, go.mod, Cargo.toml, pyproject.toml)
  - Changes to schema files (*.sql, *migration*, *schema*)
  - Changes to API surface files (openapi*, swagger*, *.proto)
  - Changes to CI/CD files (.github/workflows/, .gitlab-ci.yml, Dockerfile, docker-compose*)
  - Changes to hook or agent files (.claude/hooks/, .claude/agents/)
- Output: structured report to stdout (no grant, no gate — advisory only)

**No grant written.** pull-analyst is pure advisory: it informs the user but does not block any future operation.

## Edge Cases & Risks

- **push-analyst dispatch adds latency** to /push — subagent dispatch takes seconds. Acceptable: /push is human-initiated and occasional.
- **merge-analyst 60s expiry** can expire if the user is slow to confirm. Resolution: orchestrator detects expired grant and re-dispatches analyst automatically.
- **push.sh remote resolution (fork vs origin)** must be replicated in commands/push.md Step 0 to ensure analyst analyzes the same remote. The logic: if `git remote get-url fork` succeeds, use "fork"; else "origin"; explicit arg overrides.
- **Nested repo** (/dev/shm/dev-workspace/dot-claude): push-analyst and merge-analyst must be repo-specific. The repo-hash in the grant path disambiguates. pull-analyst operates on the git root of the current working directory at pull time.
- **First push (no upstream)**: push-analyst must handle the case where no remote tracking ref exists for `refs/remotes/<resolved_remote>/<branch>` — use all commits in `git log HEAD` as the range being pushed. The resolved remote is determined by fork-prefer-origin logic (matching push.sh:38-42), NOT by @{u}.
- **@{u} vs resolved remote divergence**: In fork workflows, `@{u}` tracks `origin` (upstream, read-only) while push target is `fork` (writable). Using `@{u}..HEAD` would analyze zero commits (or the wrong set) for push-analyst. Must use `refs/remotes/<resolved_remote>/<branch>..HEAD`.
- **Pull with stash conflicts (verified from pull.sh source)**: pull.sh line 154 prints "Pull completed successfully" and HEAD has changed. Then lines 157-174 run `git stash pop`; if pop fails, line 174 exits 1. **Crucially**, when a rebase conflict occurs at line 120 (`git pull --rebase`), git can partially apply commits before stopping, leaving `.git/rebase-merge` or `.git/rebase-apply` directories and potentially also changing HEAD. Both paths (rebase-conflict and stash-pop-failure) can produce `exit non-0 + HEAD changed`. The orchestrator MUST use the rebase-state-directory as the primary discriminant: if `.git/rebase-merge` or `.git/rebase-apply` exists after non-zero exit, `pull_exit_phase="rebase_conflict"` regardless of HEAD equality. Only if no rebase state directory exists AND HEAD changed, set `pull_exit_phase="stash_restoration_failed"`.
- **Stale/concurrent grants**: nonce + session_id scoping makes cross-session grant consumption impossible. But if the analyst fails to write its grant, commands/push.md or commands/merge.md must abort with a clear error (do not silently proceed).

## Acceptance Criteria

### AC1: push-analyst is dispatched before push.sh with correct range

GIVEN a valid push-gate token exists AND the user invokes /push
WHEN commands/push.md executes
THEN push-analyst subagent is dispatched BEFORE push.sh is called
AND push-analyst reads the commits-to-push range as `refs/remotes/<resolved_remote>/<branch>..HEAD` (where `<resolved_remote>` is the fork-prefer-origin resolved remote: "fork" if `git remote get-url fork` succeeds, else "origin", or the explicit arg)
AND push-analyst does NOT use `@{u}..HEAD` (which may track a different remote than the push target in fork workflows)
AND for a first push where no remote tracking ref exists, push-analyst uses all commits in `git log HEAD`
AND push-analyst writes a grant at `/tmp/agentic-commit/push-analyst/<repo-hash>/<session-id>/<nonce>.json`

### AC2: push verdict=blocked aborts push

GIVEN push-analyst returns verdict=blocked
WHEN commands/push.md reads the grant
THEN push.sh is NOT called
AND the risks[] are displayed to the user
AND the push-analyst grant is consumed (unlinked)

### AC3: push grant is state-bound and single-use

GIVEN a push-analyst grant at the nonce path
WHEN commands/push.md validates it
THEN the grant must contain matching: nonce, branch, head_sha (== current HEAD), remote_name
AND if any field mismatches, commands/push.md aborts
AND the grant is unlinked on successful validation (single-use)

### AC4: merge-analyst is dispatched before merge.sh

GIVEN the user invokes /merge with a resolved branch name
WHEN commands/merge.md executes
THEN merge-analyst subagent is dispatched BEFORE merge.sh is called
AND the branch name is passed to merge-analyst (analyst does NOT infer branch from filesystem)
AND merge-analyst writes a grant with source_tip + default_tip

### AC5: merge branch tip revalidation before execution

GIVEN a merge-analyst grant with source_tip and default_tip
WHEN commands/merge.md is about to call merge.sh
THEN commands/merge.md re-reads current `git rev-parse refs/heads/<branch>` and current default branch tip
AND if either has changed since the grant was written, commands/merge.md aborts and requires re-running /merge
AND the grant is consumed (unlinked) on successful validation

### AC6: merge grant expires in 60s

GIVEN a merge-analyst grant written at time T
WHEN commands/merge.md attempts to read it at time T+61s or later
THEN the grant is rejected as expired
AND commands/merge.md reports the expiry and re-dispatches merge-analyst

### AC7: pull-analyst dispatch and pull_exit_phase use rebase-state-directory as primary discriminant

GIVEN the user invokes /pull
WHEN commands/pull.md captures post_pull_head (always, regardless of pull.sh exit code)
THEN if pull.sh exits 0: pull_exit_phase = "success"; dispatch pull-analyst if pre_pull_head != post_pull_head
AND if pull.sh exits non-0 AND (.git/rebase-merge OR .git/rebase-apply directory exists):
    pull_exit_phase = "rebase_conflict"; display error; do NOT dispatch pull-analyst
    (rebase state dir is the primary discriminant: git rebase partially applied commits
    before conflict; HEAD may have changed but stash pop was never reached)
AND if pull.sh exits non-0 AND no rebase state directory exists AND pre_pull_head != post_pull_head:
    pull_exit_phase = "stash_restoration_failed"; dispatch pull-analyst with this phase
    (rebase fully succeeded — HEAD changed — then stash pop failed; no rebase state remains)
AND if pull.sh exits non-0 AND no rebase state directory exists AND pre_pull_head == post_pull_head:
    pull_exit_phase = "rebase_conflict"; display error; do NOT dispatch pull-analyst
    (unknown pre-rebase failure; classify conservatively)
AND if pull.sh exits 0 AND pre_pull_head == post_pull_head: print "no HEAD change (no-op pull)"; skip pull-analyst

RATIONALE: pull.sh line 120 runs `git pull --rebase`; if rebase hits a conflict, line 151 exits 1 BEFORE stash pop (lines 158-174). Git rebase may partially apply commits and change HEAD before stopping, leaving .git/rebase-merge or .git/rebase-apply. The stash-pop failure path (exit at line 174) only runs AFTER successful rebase (line 154 success message), so no rebase state directory exists in that case. HEAD-equality alone is an unreliable discriminant because rebase conflicts can also change HEAD. The rebase state directory presence resolves the ambiguity.

### AC8: pull-analyst is advisory only; pull_exit_phase surfaced in report

GIVEN pull-analyst produces a semantic risk report
THEN no grant token is written
AND no gate blocks any subsequent command based on the report
AND the report is displayed to the user
AND when dispatched with `pull_exit_phase="stash_restoration_failed"`, the report includes a prominent warning: "NOTE: git pull --rebase succeeded but stash restoration failed — local changes remain in stash@{0}; run `git stash pop` after resolving conflicts"

### AC9: analyst agents have explicit DO NOT sections

GIVEN agents/push-analyst.md, agents/merge-analyst.md, agents/pull-analyst.md
WHEN their DO NOT sections are read
THEN each explicitly forbids: git commit, git push, git merge, git pull, git rebase, git reset --hard, git branch -D, any force operation, writing grants for other agents

### AC10: wrapper scripts are unchanged

GIVEN commands/push.md, commands/merge.md, commands/pull.md are updated
THEN hooks/push.sh, hooks/merge.sh, hooks/pull.sh are byte-for-byte identical to their pre-change versions

### AC11: policy registration enables analyst subagent dispatch with correct per-role tool restrictions

GIVEN policies/tool-policy.v1.json is updated
WHEN the roles section is read
THEN `push-analyst`, `merge-analyst`, and `pull-analyst` are each present as named role keys (following the `changelog-analyst` entry format at lines 792-818)
