# BA Specification: Fix Push-Gate Token `session_id` Mismatch

**Request ID**: dev-20260527-063758-T1
**Task ID**: dev-20260527-063758-T1

## Goal

`agents/changelog-analyst.md` Phase 10 does not specify which env var to use for the `session_id` field when writing the push-gate token. Subagents naturally reach for their own `CLAUDE_SESSION_ID` (ephemeral), while the DO NOT rule #7 overwrite guard compares against "current session" — which can then block a subsequent commit run from overwriting a token written with a different session ID. The fix pins `session_id` to `CLAUDE_CODE_SESSION_ID` (the stable orchestrator/user-level session) with a defined fallback chain, and ensures the overwrite guard uses the same resolved value.

---

## Context

When `changelog-analyst` runs as a subagent dispatched by `/commit`, its `CLAUDE_SESSION_ID` env var holds an ephemeral subagent session ID (e.g., `ed539f16-...`). The orchestrator's `CLAUDE_CODE_SESSION_ID` (e.g., `96c8ad3a-...`) is the user-level session ID set by Claude Code at startup and inherited by all child processes, including subagents.

Phase 10 of `changelog-analyst.md` (line 394) instructs the agent to write a push-gate token containing `{commit_sha, branch, repo_root, session_id}` but does not specify which env var provides `session_id`. The agent's proximity to `CLAUDE_SESSION_ID` (which is used at line 75 for the commit-manifest lookup) makes the subagent's own session ID the natural fallback.

DO NOT rule #7 (line 36-37) protects tokens from being overwritten by different sessions, comparing the existing token's `session_id` against "the current session". If a prior token was written with a different subagent session ID, the guard may fire and skip the write, causing the push gate to fail because the old token has a stale `commit_sha`.

`hooks/push.sh` (lines 197-232) validates the push-gate token by comparing `commit_sha` only — it does NOT compare `session_id`. The failure mode is: the token does not exist (was never written due to the overwrite guard skip), not a session_id comparison.

`scripts/write-push-gate-token.py` does not exist; changelog-analyst uses an inline Python script.

`CLAUDE_CODE_SESSION_ID` is confirmed present in the environment (observed: `env | grep CLAUDE_CODE_SESSION_ID` in this session returned `96c8ad3a-d367-49a2-87e5-c29ec88e2947`). Subagents inherit parent env vars, so this value is available to changelog-analyst at runtime.

### Technical Rationale: Three-Part Chain vs. Two-Part Default Form

The user's verbatim requirement included the form `os.environ.get("CLAUDE_SESSION_ID", "unknown")` as the second element of the fallback. This spec intentionally substitutes the three-part `or` chain for the following reasons:

1. **Coverage of CLAUDE_SESSION_ID when CLAUDE_CODE_SESSION_ID is absent**: `os.environ.get("CLAUDE_CODE_SESSION_ID", "unknown")` (two-part, single-var form) would fall back to `"unknown"` if `CLAUDE_CODE_SESSION_ID` is unset, silently discarding the subagent's own `CLAUDE_SESSION_ID`. The three-part chain — try `CLAUDE_CODE_SESSION_ID` (orchestrator's stable ID) → try `CLAUDE_SESSION_ID` (agent's own ID) → `"unknown"` — ensures that inner subagents or older Claude Code versions that only have `CLAUDE_SESSION_ID` still produce a non-`"unknown"` session identifier.

2. **Empty-string safety (Codex finding #2)**: `os.environ.get(VAR, "unknown")` returns `"unknown"` only when `VAR` is absent from the environment; it returns `""` (empty string) if `VAR` is set to the empty string. Python `or` short-circuits on falsy values, so `os.environ.get(VAR) or "unknown"` correctly collapses both unset and empty-string cases to `"unknown"`. The `default=` form does NOT provide this protection.

**Required form** (mandatory — deviates from user's literal text by design):
```
PUSH_GATE_SID = os.environ.get("CLAUDE_CODE_SESSION_ID") or os.environ.get("CLAUDE_SESSION_ID") or "unknown"
```

**Forbidden forms** — any of these FAIL the empty-string-safety requirement or silently discard the CLAUDE_SESSION_ID fallback:
```
# Forbidden: user-literal form (positional default on CLAUDE_SESSION_ID lookup)
PUSH_GATE_SID = os.environ.get("CLAUDE_CODE_SESSION_ID") or os.environ.get("CLAUDE_SESSION_ID", "unknown")

# Forbidden: nested default (positional default on CLAUDE_CODE_SESSION_ID lookup)
PUSH_GATE_SID = os.environ.get("CLAUDE_CODE_SESSION_ID", os.environ.get("CLAUDE_SESSION_ID", "unknown"))

# Forbidden: keyword default on either lookup
PUSH_GATE_SID = os.environ.get("CLAUDE_CODE_SESSION_ID", default="unknown")
```
Any form that passes a positional or keyword default argument to either `os.environ.get()` call is forbidden. Only the bare `get(VAR) or ...` chain is acceptable.
