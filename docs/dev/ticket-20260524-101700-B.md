## Goal

Fix the two critical bugs in the write-guard sentinel integration that caused QA rejection of task 20260522-080646-B: (1) the task_id key mismatch that prevented sentinels from being found in realistic usage, and (2) the CF-1 regression that broke targetless `/allow Write` grants. Both fixes exist in the working tree but are uncommitted; this cycle's job is to validate, complete, and commit them with proper test coverage for both bug classes.

## Context

Task 20260522-080646-B implemented `match_sentinel_grant_for_write()`, a sentinel check in `pretool-write-guard.sh`, and Write sentinel consumption in `posttool-allowlist-consume.py`. QA rejected the implementation with two critical bugs:

**Bug 1 â€” task_id key mismatch** (confirmed via live subprocess):
The sentinel writer (`userprompt-consent-allowlist.sh:183`) keys the sentinel file as `${CLAUDE_TASK_ID:-${SID}}`. When `CLAUDE_TASK_ID` is unset (which is common â€” it is a user-managed env var, not automatically set by Claude Code), the key is `SID` (session_id). The write-guard in the committed code resolved `_WG_TASK_ID` as `data.task_id` only (always populated by Claude Code with the invocation task_id, e.g. `"20260522-080646-B"`). `load_sentinel_grant_for_task("20260522-080646-B")` finds no file when the sentinel was keyed by `session_id`. The sentinel bypass is silently skipped.

**Bug 2 â€” targetless grant CF-1 regression** (confirmed via code trace):
When the user issues `/allow Write` (no target file), the sentinel writer produces `{op: "Write"}` with no `target` field. In the committed `match_sentinel_grant_for_write()`, the only match condition was `entry.get("target") == target_path`, which evaluates to `None == "/path/to/file"` â†’ False â†’ returns None â†’ `valid_no_match`. CF-1 fires on existing-file writes, suppressing the legacy `/allow` fallback. Every user who issued a targetless write permission had their legitimate overwrites blocked.

The working tree already contains uncommitted fixes for both bugs: the write-guard uses a candidate-list `[writer_key, data_task_id, session_id]` where `writer_key = env_task_id if env_task_id else session_id` (mirroring the writer's priority), and `match_sentinel_grant_for_write()` has a wildcard case (`entry_target is None or entry_target == target_path`). However, test coverage is incomplete: there is no test for the targetless wildcard case.
