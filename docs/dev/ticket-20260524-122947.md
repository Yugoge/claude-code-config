## Goal

Fix the structural incompatibility between the orchestrator's 3-consecutive-Bash limit and the
`/push` protocol's requirement that Steps 4+5 (grant validation + Chain-B sentinel + push.sh
execution) run in a single process. Deliver two artifacts: (1) a Python wrapper script at
`scripts/execute-push.py` that encapsulates the entire validate+consume+sentinel+exec sequence
as a single Bash invocation, and (2) an explicit, hard-to-miss warning in `commands/push.md`
Step 4 and Step 5 that these steps MUST NOT be delegated to a subagent.

## Context

The `/push` protocol defines a 6-step orchestrator sequence (Steps 0-5). Steps 0-2 consume the
orchestrator's 3-Bash budget, leaving Steps 4+5 un-executable by the orchestrator directly.
When the orchestrator delegates Steps 4+5 to a subagent, three compounding failures occur:
(a) HOW-prescriptive prompts cause the subagent to delete the wrong file (Chain-A token instead
of the Chain-B grant); (b) the subagent legitimately rejects writing the Chain-B sentinel as
a privilege escalation; (c) the Chain-B sentinel's 60-second mtime gate in push.sh cannot
survive two sequential agent dispatch delays (30-90s each). The successful pattern is a single
Python process that validates, consumes the grant, writes the sentinel, and exec's push.sh,
staying under the 60s window by construction.

This was identified via a completed root-cause analysis by the user (user requirement document
`docs/dev/user-requirement-dev-20260524-122947.md`). There is no prior failed attempt at this
specific fix — this is new implementation work.

## Edge Cases & Risks

- **Risk 1 (tilde expansion)**: `os.execv` does NOT expand `~`. Script MUST use `os.path.expanduser("~/.claude/hooks/push.sh")` and verify the expanded path exists and is executable before exec'ing. Failure to expand causes `FileNotFoundError` at exec time rather than a clear error message.
- **Risk 2 (HEAD drift)**: Between push-analyst grant write and execute-push.py invocation, HEAD could advance (e.g., auto-commit hook). Script re-reads HEAD at sentinel-write time and must abort if `git rev-parse HEAD` no longer matches `head_sha` from the grant.
- **Risk 3 (grant unlink before exec)**: Grant must be unlinked BEFORE sentinel is written and BEFORE exec. If exec fails after grant unlink but before sentinel write, the push cycle must be restarted from `/commit` first. This is the same failure mode as the existing inline bash approach.
- **Risk 4 (--auto forwarding)**: The existing Step 5 note says "if `--auto` is required, append it to the exec line." The script must expose `--auto` as a flag and forward it to push.sh. Forgetting this regresses non-interactive push sessions.
- **Risk 5 (sentinel JSON via printf vs json.dumps)**: The existing `commands/push.md` Step 5 bash block uses `printf` to write the sentinel. Python's `json.dumps` is safer for special characters in branch/remote names. Dev MUST use `json.dumps`, not string interpolation, to avoid JSON injection via branch names with quotes or backslashes.
- **Risk 6 (blocked verdict): Script MUST NOT unlink the grant or write the sentinel when `verdict == "blocked"`. The grant unlink must happen only after the verdict check passes.
- **Risk 7 (Step 4 doc update scope)**: Step 4 in `commands/push.md` currently instructs the orchestrator to perform manual grant validation. Updating Step 4 to redirect to `execute-push.py` without removing the old instructions would create conflicting guidance. Dev must replace the orchestrator-imperative bash in Step 4 with a reference paragraph.
