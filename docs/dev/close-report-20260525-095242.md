### Finding #1: Entry-gate regex misses literal glob `*` character

- **Location**: hooks/pretool-bash-safety.sh:662
- **Description**: The entry-gate regex `[A-Za-z0-9_\-]*` matches concrete session IDs but not literal `*` (shell glob character). Commands using `/tmp/claude-bulk-allowed-*.flag` (literal glob form) do not enter Layer 1.F at all.
- **Codex reproduced**: `ls /tmp/claude-bulk-allowed-*.flag ; true` -> rc=0 (entry gate not triggered)

**Classification**: `out_of_scope` per codex calling protocol Rule 3/4.

**Rationale**:
1. The user's verbatim complaint used a concrete session-ID (`<sid>`), not a literal glob.
2. All 33 AC-01 matrix rows use `testsid` (a concrete materialized string), not `*`.
3. The entry-gate regex character class `[A-Za-z0-9_\-]*` was UNCHANGED from the prior cycle -- this is not a regression introduced by Cycle 6.
4. BA spec Dev constraint says "Do NOT widen the protected-path glob" -- widening the regex character class to include `*` is a scope expansion, not a fix of the named bypass.
5. BA spec Risk 1 section discusses glob expansion: "the model emits either a literal path or a literal glob string; both are fine" -- but the entry-gate regex must support both if this is to be fixed, which is entry-gate widening, not compound-bypass closing.
6. This is the SAME class of finding as (a) codex dev-phase finding #1 (env-var `$CLAUDE_SESSION_ID` bypass) and (b) QA observation #1 (ANSI-C quoting bypass) -- all three are entry-gate/encoding gaps that bypass the Layer 1.F predicate stack entirely. All are real but all are outside the 33-row matrix scope.

**Recommendation**: Surface to user as "codex close-debate also found that literal glob forms bypass the entry gate -- same class as the env-var and ANSI-C quoting gaps already noted. Recommend a dedicated follow-up cycle for entry-gate hardening."
