## Goal

Fix a one-line Bash redirect-ordering bug in `hooks/userprompt-tmpfs-pressure.sh` at line 129. When `${COUNTER_FILE}.lock` is pre-created as a directory, Bash evaluates `9> "$LOCK_FILE"` before `2>/dev/null` takes effect, leaking one stderr line to the terminal. Reorder the redirects so stderr is silenced first.

## Context

Cycle 3 (task 20260524-171714) implemented arch-3 (COUNTER_FILE EISDIR guards, AC7 and AC7b) and arch-6. The LOCK_FILE EISDIR case was identified as out-of-scope for Cycle 3 (no AC defined; outside M2 scope). The spec (Section 8b) authorises this as a standalone micro-cycle. AC7 and AC7b already pass and must not be regressed.

The bug: at line 129, the brace-group that wraps the flock critical section terminates with `} 9> "$LOCK_FILE" 2>/dev/null`. Bash processes redirections left-to-right. `9> "$LOCK_FILE"` is evaluated first; if `$LOCK_FILE` is a directory, `open(2)` fails immediately and Bash emits an error on file-descriptor 2 (still the real stderr at that point). `2>/dev/null` takes effect afterwards, too late to suppress the message.

The fix reverses the order: `} 2>/dev/null 9> "$LOCK_FILE"`. Stderr is redirected to /dev/null first; the subsequent `9>` failure is then silenced.
