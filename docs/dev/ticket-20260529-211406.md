# BA Specification: Recovery commit when task files pre-empted by foreign bulk session

**Task-ID**: 20260529-211406

## Goal

When another session's `--bulk` run accidentally picks up this session's changed files before this session runs `/commit <task-id>`, the commit workflow finds nothing to commit (working tree clean) and is blocked — no push-gate token written, task-id attribution lost. Add a recovery path inside `agents/changelog-analyst.md` so that `nothing_to_commit_precommitted` detection automatically produces a `git commit --allow-empty` with proper attribution and writes the push-gate token, unblocking `/push` without touching the prior bulk commit.

---

## Context

`/commit <task-id>` dispatches `changelog-analyst`, which (in Phase 8's structured status output section) detects `nothing_to_commit_precommitted` via the three-step SHA-stable check: candidate set empty + HEAD matches `^auto-bulk:` + HEAD commit touches at least one file from the dev-report. When all three conditions hold, changelog-analyst currently returns `commit_status: nothing_to_commit_precommitted` with `auto_bulk_commits[]` — and stops. The push-gate token is never written. `/push` is blocked.

The single-use commit grant written by `/commit` Step 5 is NOT consumed when working-tree was clean (the guard only consumes on a live `git commit` call; nothing fires against an empty working tree). The grant is therefore still valid and available for use by the recovery commit.

## Requirements (MoSCoW)

### Must Have

- R1: When `nothing_to_commit_precommitted` is detected in changelog-analyst (all three conditions hold), instead of returning that status, execute a `git commit --allow-empty -F <tmpfile>` with an attribution commit body (see Technical Hints for format), write the push-gate token (Phase 10 logic), and return `commit_status: committed`.
- R2: The recovery commit must use the existing single-use commit grant (already written by `/commit` Step 5, not consumed because no `git commit` fired against a clean working tree). No new grant write by changelog-analyst itself.
- R3: For the multi-repo case (both root repo and nested repo have precommitted content detected in their respective `nothing_to_commit_precommitted` checks): two recovery commits are needed. The single-use grant is consumed after the first recovery commit. `commands/commit.md` Step 5 (BULK=false) must be extended to write TWO commit grants when dispatching changelog-analyst (one for root, one for nested), so the second recovery commit has a valid grant. This is the minimal, safest multi-repo solution — no changelog-analyst self-grant-write, no script changes required.
- R4: Push-gate token collision: if the push-gate token path already exists and its `session_id` differs from `PUSH_GATE_SID`, the existing DO NOT rule 7 skip-with-warning applies (unchanged). If the existing token's `session_id` matches `PUSH_GATE_SID` OR the token path does not exist, write (overwrite) the token normally. No CAS ancestor check needed: recovery commits always land on top of the bulk commit, so the existing token (if any, from a different session) represents a stale attribution from that session — overwriting with this session's recovery commit SHA is correct behavior.
- R5: The `nothing_to_commit_precommitted` structured status remains valid in the output schema as a possible value — BUT the spec's detection section must be updated to say: "when detected, execute recovery commit rather than returning this status; the status is only returned if recovery commit fails."

### Should Have

- R6: If the recovery `git commit --allow-empty` itself fails (e.g., hook blocked, git error), return `commit_status: failed` with an appropriate `failure_code` (`git_error` or `hook_blocked`) and `failure_reason` describing that the failure occurred during precommitted recovery.

### Could Have

- R7: The `auto_bulk_commits[]` metadata from the detection step can be included in the structured output even when `commit_status: committed` (as an informational field), so `/commit` Step 6 can log it. Not strictly needed for unblocking `/push`.

### Won't Have (Non-Goals)

- Modifying or reverting the prior bulk commit (user requirement explicitly says "不动之前bulk的" — do not touch prior bulk commits).
- Any change to the `nothing_to_commit` (no precommit detected) path — purely additive.

## Edge Cases & Risks

- **Grant availability**: The grant is assumed unconsumed at recovery time because no `git commit` fired against a clean tree. This assumption holds because `pretool-git-privilege-guard.py` only consumes the grant via `_lock_grant_for_posttool` on a live Bash tool call matching `git commit`. If somehow the grant was consumed (e.g., a concurrent commit in same session), the recovery commit fails with `failure_code: grant_missing` — same retry protocol as normal grant_missing already handles this.
- **Multi-repo: second grant must be pre-written**: changelog-analyst MUST NOT invoke `write-commit-grant.py` itself (DO NOT rule applies: all git grant mechanics are orchestrated by `/commit`, not the subagent). The solution is for `commands/commit.md` Step 5 (BULK=false) to call `write-commit-grant.py` TWICE — one grant for root repo recovery and one for nested repo recovery — when dispatching changelog-analyst. The second grant is consumed by the nested repo's recovery commit. If only one repo has precommitted content, the second grant expires unused (30-min TTL, harmless).
- **Detection scan range**: The THREE-STEP check currently inspects only HEAD (`git rev-parse HEAD`). Architect finding notes that `baseline_head_sha..HEAD` range should be scanned to handle multi-commit bulk sweeps. Dev MUST implement the range scan: iterate commits from `baseline_head_sha` (exclusive) to HEAD (inclusive) and collect any that match `^auto-bulk:` AND contain at least one file from `task_cycle_files`. All such SHAs populate the `Precommitted-by` lines in the recovery commit body.
- **Commit message NOT matching forbidden patterns**: recovery commit subject must not match `\bsync\b.*\buncommitted\b` or `chore\(claude\)\s*:\s*sync` (DO NOT rule 10). Proposed subject: `chore(<scope>): recovery commit — task <TASK_ID> pre-empted by bulk session` — does not match either pattern.
- **BULK_THRESHOLD**: recovery commit stages zero files (empty commit), so it touches 0 subsystem prefixes. No BULK_THRESHOLD risk.
- **Tmpfile + trap**: recovery commit must use `mktemp` + `trap "rm -f ${TMPFILE}" EXIT` before `git commit --allow-empty -F ${TMPFILE}` (same as Phase 8 contract, DO NOT rule 12).

## Acceptance Criteria

### AC1: Single-repo recovery commit created
- GIVEN: `/commit <task-id>` is run, changelog-analyst detects `nothing_to_commit_precommitted` (candidate set empty, HEAD matches `^auto-bulk:`, HEAD commit intersects `task_cycle_files`)
- WHEN: recovery path executes
- THEN: `git log HEAD --format="%s" -1` shows a subject containing `task` and `TASK_ID` (not `auto-bulk:` prefix); `git show HEAD --stat` shows `0 insertions, 0 deletions` (empty commit); the push-gate token file exists at the expected path; changelog-analyst emits `commit_status: committed` in the structured output block.

### AC2: Prior bulk commit is untouched
- GIVEN: recovery commit is created (AC1)
- WHEN: `git log HEAD~1 --format="%s" -1`
- THEN: the output starts with `auto-bulk:` (the prior bulk commit SHA is unchanged, it is still HEAD~1).

### AC3: Recovery commit body contains required attribution
- GIVEN: recovery commit is created
- WHEN: `git show HEAD --format="%B" -s`
- THEN: output contains lines matching `Task-id: <TASK_ID>`, `Precommitted-by: <sha>`, and `Attributed-files:` followed by at least one file path from the dev-report's `files_modified` or `files_created` lists.

### AC4: Push-gate token written and parseable
- GIVEN: recovery commit is created
- WHEN: the push-gate token file at `/tmp/agentic-commit/push/<repo_hash>/<branch>.json` is read
- THEN: `commit_sha` field equals the SHA of the recovery commit (HEAD after recovery); token is valid JSON.

### AC5: Multi-repo — both repos get recovery commits with no grant error
- GIVEN: both root and nested repo have `nothing_to_commit_precommitted` detected; two grants are pre-written by `commands/commit.md` Step 5
- WHEN: changelog-analyst processes both repos
- THEN: two recovery commits are created (one per repo); both push-gate tokens written; `commit_status: committed` returned; no `grant_missing` or `grant_expired` failure.

### AC6: Single-repo recovery on nested repo only (no root-repo precommit)
- GIVEN: only nested repo has precommitted content; root repo has normal staged changes
- WHEN: changelog-analyst runs
- THEN: root repo gets a normal real commit (Phase 8), nested repo gets a recovery commit; both push-gate tokens written; `commit_status: committed` overall.

### AC7: Recovery commit fail path returns correct failure_code
- GIVEN: recovery path detects precommitted state but the `git commit --allow-empty` is blocked by a hook
- WHEN: the git commit fails
- THEN: changelog-analyst returns `commit_status: failed`, `failure_code: hook_blocked` (or `git_error`), `failure_reason` contains text indicating failure during precommitted recovery.

### AC8: Non-precommitted path is unaffected (regression guard)
- GIVEN: normal `/commit <task-id>` run where working tree has changes (no precommit)
- WHEN: changelog-analyst runs
- THEN: behavior is identical to pre-change — real diff-based commit created, no recovery path entered.

## Technical Hints

- **Primary file**: `agents/changelog-analyst.md` — add a new subsection after the THREE-STEP SHA-STABLE CHECK, titled "Recovery path when `nothing_to_commit_precommitted` is detected (BULK=false only)". This subsection replaces the current "return `nothing_to_commit_precommitted`" behavior.
- **Secondary file**: `commands/commit.md` Step 5 — when `BULK=false`, write TWO commit grants instead of one (call `write-commit-grant.py --task-id "$TASK_ID"` twice; each call produces a distinct nonce). Add a comment explaining this covers multi-repo recovery. Total change: one extra `write-commit-grant.py` invocation and a comment.
- **Scan range for detection**: Replace the current single-HEAD check with a range scan: `git log --format="%H %s" baseline_head_sha..HEAD` then filter commits matching `^auto-bulk:`. For each matching commit, run `git show --name-only --format= <sha>` and check intersection with `task_cycle_files`. Collect all matching SHAs into `precommitted_shas[]`.
- **Recovery commit message format**:
  ```
  chore(<scope>): recovery commit — task <TASK_ID> pre-empted by bulk session

  Task-id: <TASK_ID>
  Precommitted-by: <sha1>
  Precommitted-by: <sha2>   [if multi-commit bulk sweep]
  Attributed-files:
    <file1>
    <file2>
    ...
  ```
  `<scope>` is derived from the repo being committed (`agents`, `commands`, etc.) using the same scope-derivation logic as Phase 6.
- **Tmpfile handling**: same pattern as Phase 8 — `TMPFILE=$(umask 077; mktemp ...)` + `trap "rm -f ${TMPFILE}" EXIT` + `git commit --allow-empty -F "${TMPFILE}"`.
- **Push-gate write**: after recovery commit, run Phase 10 push-gate write logic unchanged (same `sha256(realpath(GIT_ROOT))[:16]` hash, same JSON schema). DO NOT rule 7 applies: if existing token's `session_id` differs from `PUSH_GATE_SID`, print WARNING and skip.
- **Structured output**: return `commit_status: committed`; optionally include `"recovery": true` and `"precommitted_shas": [...]` as informational fields in the JSON (not part of the schema's defined fields, so `/commit` ignores them harmlessly).
- **Verification harness cleanup contract (R10 cross-reference)**: any AC verification harness that spawns background processes must use `trap '<cleanup>' EXIT INT TERM`; see `agents/qa.md` for canonical pattern.
