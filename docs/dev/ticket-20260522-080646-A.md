## Goal

Rewrite the matching core of `hooks/pretool-bash-safety.sh` for the four generic danger-token rules (`kill\s+-`, `killall|pkill`, workflow `rm|mv`, filesystem `rm`) to distinguish "command-form context" (token is the actual shell-level executable being invoked) from "string-content context" (token appears inside a quoted argument, heredoc body, Python -c payload, or comment). This eliminates the documented false-positive class that blocked 5+ legitimate meta-audit commands during cycle 20260519-211515.

Scope is **only** `hooks/pretool-bash-safety.sh` plus optional helper `hooks/lib/bash_context_strip.py` and regression tests in `hooks/tests/`. Settings.json and all other hooks are NOT in scope.

## Context

The hook is invoked on every Bash tool call. It reads the raw command string and applies a set of `grep -qE` pattern matches. For most rules, the raw-string match is intentional and correct: Layer 1.A–E daemon-restart blocks and the absolute-ban rules (`session_dirs.txt`, `happy-restart`, `happy-session-recovery`) are designed to trigger on any appearance of their tokens regardless of context — these are NEVER candidates for context-stripping.

The false-positive class affects only four rules in the "general safety" section:

| Line | Rule | Documented FP example |
|------|------|-----------------------|
| 715 | `(killall\|pkill)\s+.*(happy\|claude\|docker)` | codex prompt body explaining that killall is forbidden — contains "killall" and "claude" |
| 722 | `kill\s+-` | echo argument containing "kill -TERM" as documentation |
| 743 | `(rm\|mv)\s` + workflow path check | codex prompt body containing "rm" |
| 751 | `(^|[;|\&]\s*)rm\s` | codex prompt body containing "rm" |

The fix: compute a `COMMAND_CONTEXT_STRIPPED` view that elides non-executable text spans from the raw command. Apply ONLY those four rules against `COMMAND_CONTEXT_STRIPPED`; all other rules continue to operate on raw `$COMMAND`.

## Requirements (MoSCoW)

### Must Have

- M1: Compute a `COMMAND_CONTEXT_STRIPPED` variable that strips non-executable text spans (quoted literals, non-shell heredoc bodies, shell comment text) from the raw `$COMMAND` string.
- M2: Apply the four affected rules (lines 715, 722, 743, 751) against `COMMAND_CONTEXT_STRIPPED` instead of raw `$COMMAND`. For line 743's compound condition specifically: ONLY the dangerous-token grep `(rm|mv)\s` is applied against `COMMAND_CONTEXT_STRIPPED`; the workflow-path grep `(workflow-[^/]*\.json|\.claude/todos/)` MUST remain on raw `$COMMAND`. This preserves the security property that workflow-enforcement-file deletion is blocked regardless of context stripping — a command like `rm ".claude/todos/x"` would strip the quoted path from the executable-context view, so the workflow-path grep must operate on raw `$COMMAND` to still fire.
- M3: Leave ALL other rules (Layer 1.A–E, absolute bans, sensitive-file rules, docker rules, git rules, etc.) operating on raw `$COMMAND` — the stripped view must NOT be used outside the four false-positive-prone rules.
- M4: The stripper must classify heredoc bodies by the consuming command/pipeline (e.g. `bash|sh|zsh <<EOF` = shell context, preserve; `python3 <<'PY'` = non-shell, strip body) — not by the heredoc delimiter label (finding 2).
- M5: The stripper must split on top-level shell control operators (`;`, `&&`, `||`, `|`, newlines) before stripping within simple commands — it must never strip across command boundaries (finding 3). A command like `echo "rm text"; rm /tmp/foo` must still block.
- M6: Inside double-quoted strings, `$()` and backtick command substitutions must be preserved for scanning (finding 4). Only the literal text portions between substitutions may be stripped.
- M7: For `sh -c` / `bash -c` payloads, extract and recursively analyze only the string argument immediately following the `-c` flag. Later positional arguments (`$0` etc.) are non-executable content and must not be scanned (finding 6).
- M8: The stripper must strip shell comment text (`# ...` through end-of-line) but only when the `#` appears in command position (not inside a string token).
- M9: Add a new regression test file `hooks/tests/test_bash_safety_context.py` (or `test_bash_safety_context.sh`) with the concrete MUST BLOCK and MUST NOT BLOCK cases from the ACs below.
- M10: On any parse ambiguity or stripper exception, fall back to returning the raw command text unchanged — never silently drop content that could be an executable context (safe side = block).

### Should Have

- S1: Add `hooks/lib/bash_context_strip.py` as a separate module (rather than an inline heredoc within the shell script) for testability and future reuse.
- S2: The stripping function's performance impact must be acceptable: it runs inline on every Bash tool call. Aim for sub-100ms on a 10KB command string using stdlib-only Python.

### Could Have

- C1: For `python3 -c` / Python `-c` payloads: strip Python comments (lines starting with `#` after tokenization) before running the shell-pattern grep. Full Python AST stripping is not required (finding 5).

### Won't Have

- W1: Full bash AST parsing via `bashlex` or any non-stdlib package. Layered heuristics only.
- W2: Changes to any file other than `hooks/pretool-bash-safety.sh`, `hooks/lib/bash_context_strip.py` (new), and `hooks/tests/test_bash_safety_context.*` (new).
- W3: Context-stripping for Layer 1.A–E daemon-restart blocks. Those rules are intentionally broad and must NOT be weakened.
- W4: Context-stripping for absolute-ban rules (`session_dirs.txt`, `happy-restart`, `happy-session-recovery`).

## Edge Cases & Risks

- **Risk 1 (BLOCKER from codex finding 3)**: A compound command like `echo "rm text"; rm /tmp/foo` must block. The stripper must NOT erase across `;`/`&&`/`||`/`|`/newline boundaries.
- **Risk 2 (BLOCKER from codex finding 4)**: `echo "$(rm /tmp/foo)"` and `echo "\`kill -9 1234\`"` must block. Command substitutions inside double-quoted strings are executable contexts.
- **Risk 3 (MAJOR from codex finding 5)**: `python3 -c 'import os; os.system("rm /tmp/foo")'` must block. The spec does NOT promise that `python3 -c` payloads are fully stripped — only Python comments (`# ...` at Python token level) may be stripped if C1 is implemented. Shell-dangerous strings inside `os.system()` / `subprocess` calls are NOT stripped.
- **Risk 4**: Stripper failure modes must be fail-safe — any exception or ambiguity falls back to raw `$COMMAND` (M10).
- **Risk 5**: Layer 1.A–E operate correctly on raw `$COMMAND` and must not be inadvertently touched (M3).
- **Risk 6 (MAJOR from codex finding 7)**: Even after context-stripping, the four rules should ideally match command tokens, not arbitrary substrings. The spec requires that `kill\s+-` match at a command-word boundary after stripping (so `--no-kill-timeout` does not match even on a stripped command). Dev must tighten the `kill\s+-` pattern to `(^|[\s;|&])(kill)\s+-` or equivalent to ensure word-boundary matching.

### Stripping algorithm (layered heuristic — not full AST)

The function `strip_non_executable_contexts(cmd)` operates in passes:

**Pass 1 — split on top-level control operators** (`;`, `&&`, `||`, `|`, newlines not inside quotes/heredocs). Process each simple-command segment independently. Reassemble with original separators.

**Pass 2 — within each simple-command segment:**
1. Identify the first command word (skip `env VAR=val` prefixes, `time`, `sudo`, etc.).
2. If the command word is a shell alias (`bash`, `sh`, `zsh`, `dash`) and is followed by `-c` flag: extract the next token as the `-c` payload. Strip only the payload from the outer command, then recurse into `strip_non_executable_contexts(payload)` and append the result as the canonical executable text.
3. If the command word is a non-shell command (anything not in `{bash, sh, zsh, dash, exec, eval}`): strip all quoted string arguments past the command word. Preserve `$()` and backtick spans (recursively analyze those).
4. If the command word is `echo`, `printf`, `cat` (data-output commands): strip all arguments past the command word (they are data, not commands). Exception: `cat <<HEREDOC | bash` — see Pass 3.
5. Strip `# comment text` from end of line when `#` appears outside a string token.

**Pass 3 — heredoc classification** (applied after Pass 1 separation, operating on the full compound command to detect piped heredocs):
- For each heredoc marker `<<` or `<<'...'`:
  - Look at the command word LEFT of the `<<`. If it is `bash|sh|zsh|dash|eval` or the pipeline ends with `| bash` / `| sh`: body is shell-executable context — preserve raw or recursively strip.
  - If it is `python3|ruby|node|perl` or `python3 -c`: body is non-shell — strip the heredoc body.
  - If it is `cat` and the pipeline continues `| bash` / `| sh`: shell context — preserve.
  - If it is `cat` and the pipeline does not continue into a shell: data context — strip body.
  - Ambiguous or unrecognized consumer: preserve raw (fail-safe per M10).

**Pass 4 — tighten the `kill` pattern** in the shell script after stripping:
- Change the existing pattern `kill\s+-` to `(^|[\s;|&])(kill)\s+-` to ensure word-boundary matching even on the stripped output. This prevents `--no-kill-timeout` from matching even if stripping failed to remove it.
