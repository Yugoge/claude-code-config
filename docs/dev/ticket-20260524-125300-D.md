## Goal

Close three structural gaps in the dev self-report provenance system that allowed cycle 20260522-080646 item D to pass QA despite the dev-report omitting half of the files it actually changed: (D1) QA checks over-reporting but not under-reporting; (D2) the parallel-dev aggregate drops three provenance fields QA depends on; (D3) `files_created` derivation misses staged-but-not-yet-committed new files.

## Evidence (Contract A)

### Bug D1

- **Observed**: cycle 20260522-080646 item D dev-report listed 2 files in `files_modified`; `git diff` against baseline showed 4. QA still issued `pass`.
- **Measured**: `agents/qa.md` lines 411-422 — QA iterates `dev.files_modified` and checks each against `git diff --name-only`. No inverse iteration exists.
- **Expected**: QA must also iterate `git diff --name-only` output and flag any path absent from `dev.files_modified ∪ dev.files_created` (and absent from `baseline_dirty_paths` parsed from `baseline_dirty_snapshot`) as under-reporting.
- **Gap**: Omission direction is completely unchecked. A dev that forgets to list files it modified will always pass the current provenance gate.

### Bug D2

- **Observed**: `commands/dev.md` parallel aggregate construction rule (lines 720-734) and example JSON (lines 739-756) list fields to union but omit `baseline_head_sha`, `baseline_dirty_snapshot`, and `dev.observed_preexisting`. Summary line at commands/dev.md:699 also omits them.
- **Measured**: lines 731-732 list only `dev.files_modified` and `dev.files_created` as the provenance-relevant fields in the union; no mention of `baseline_head_sha`, `baseline_dirty_snapshot`, or `dev.observed_preexisting` anywhere in the aggregate spec.
- **Expected**: Aggregate must carry `baseline_head_sha` and `baseline_dirty_snapshot` at the top level (same positions as in per-worker reports per `agents/dev.md:501-503`) and `dev.observed_preexisting` nested under `dev` (per `agents/dev.md:521`), so QA can run the provenance check against the aggregate without reading per-worker files.
- **Gap**: When QA reads the aggregate, `baseline_head_sha` is absent → provenance check is skipped with a warning. `baseline_dirty_snapshot` is absent → exclusion set is empty → false positives on files pre-dirty at baseline. `dev.observed_preexisting` is absent → QA cannot populate this informational field.

### Bug D3

- **Observed**: `agents/dev.md:495` specifies `git ls-files --others --exclude-standard` for `files_created`. `agents/qa.md:427` mirrors the same command for the QA-side check.
- **Measured**: `git ls-files --others --exclude-standard` enumerates only UNTRACKED files. A file staged via `git add` is tracked (index entry exists) and does NOT appear in `--others` output.
- **Expected**: `files_created` derivation must also include staged new files: `git diff --cached --name-only --diff-filter=A` (files added to the index but not yet committed).
- **Gap**: Any dev that stages a new file before writing the report will omit it from `files_created`. QA's parallel check will not detect the staged file as new and will miss the over-report direction for staged new files.

## Requirements (MoSCoW)

### Must Have

- M1: QA must add a reverse (under-reporting) provenance check: compute `diff_files` from `git diff --name-only <baseline_head_sha>`, subtract `baseline_dirty_paths` (paths parsed from `baseline_dirty_snapshot` porcelain output), then flag any path in the remainder that is absent from `dev.files_modified ∪ dev.files_created` as `files_modified_underreport_violation` with severity critical.
- M2: `commands/dev.md` aggregate construction rule and example JSON must include: top-level `baseline_head_sha` (equality-verified across all workers; aggregate blocked if any worker differs), top-level `baseline_dirty_snapshot` (taken verbatim from the orchestrator's dispatch value, equality-verified), and nested `dev.observed_preexisting` (UNION of all per-worker `dev.observed_preexisting` lists). Both the summary (line 699) and the detailed rule (lines 720-734) and example JSON (lines 739-756) must be updated.
- M3: `agents/dev.md:495` must update `files_created` derivation to: `git ls-files --others --exclude-standard` UNION `git diff --cached --name-only --diff-filter=A`, minus `baseline_dirty_paths` (paths parsed from `baseline_dirty_snapshot`). QA's check at `agents/qa.md:427` must mirror this combined set.

- C1: Warn (non-blocking) when a path in `dev.observed_preexisting` also appears in `diff_files - baseline_dirty_paths` — this would indicate dev claimed a file was preexisting but it actually changed since baseline.

## Edge Cases & Risks

- **Risk 1 (D1 — overlap between files_modified and files_created for staged new files)**: `git diff --name-only <baseline>` with `--diff-filter` defaults include added files (A), so a staged new file appears in both the diff and in `git diff --cached --diff-filter=A`. The reverse check must compare diff_files against `files_modified ∪ files_created` (not `files_modified` alone) to avoid false positives. The S1 requirement captures this.
- **Risk 2 (D1 — observed_preexisting misuse)**: Codex correctly identified that `dev.observed_preexisting` (dev.md:499) is defined as paths "not in git diff" — it cannot logically overlap with diff_files. Using it as an exclusion in the reverse check would create an escape hatch where dev lists a changed file as "preexisting" to suppress a violation. The exclusion set for D1 must be `baseline_dirty_paths` only, not `observed_preexisting`.
- **Risk 3 (D2 — baseline fields are dispatch-owned, not worker-owned)**: `baseline_head_sha` and `baseline_dirty_snapshot` are captured once by the orchestrator pre-dispatch and passed verbatim to all workers (agents/dev.md:503). Workers copy them — they do not own them. Aggregating them by union would be semantically wrong (these are scalars, not lists) and could whitelist spurious file paths. The correct semantic is: assert equality across workers; take the orchestrator's dispatch value as authoritative; mark the aggregate blocked if any worker disagrees.
- **Risk 4 (D2 — both edit locations in commands/dev.md)**: The summary rule at line 699 and the detailed section at lines 720-756 are separate text regions. Patching only the detailed section leaves the summary inconsistent. Both must be updated.
- **Risk 5 (D3 — baseline_dirty_snapshot is porcelain, not a path list)**: The field is the raw `git status --porcelain` string (agents/dev.md:503). Subtraction of "baseline_dirty entries" requires parsing it to extract file paths. The spec and implementation must be explicit that "baseline_dirty_paths" means paths extracted from the porcelain output, not the raw string itself.

## Acceptance Criteria

### AC1: QA flags under-reporting (D1)

- GIVEN a dev-report with `baseline_head_sha` set, `dev.files_modified = ["agents/qa.md"]`, and `baseline_dirty_snapshot = ""`
- WHEN `git diff --name-only <baseline_head_sha>` returns `agents/qa.md`, `commands/dev.md`, `agents/dev.md`
- THEN `agents/qa.md:Step 5` provenance check raises `files_modified_underreport_violation` for `commands/dev.md` and `agents/dev.md` (both in diff, both absent from `dev.files_modified ∪ dev.files_created`, not in `baseline_dirty_paths`)
- AND the finding has `severity: critical` and `label: "files_modified_underreport_violation"`
- AND `agents/qa.md` does NOT raise `files_modified_underreport_violation` for itself (it is listed in files_modified)

### AC2: Aggregate dev-report carries provenance fields and blocks on baseline mismatch (D2)

- GIVEN two per-worker dev-reports: worker-A has `baseline_head_sha: "abc123"`, worker-B has `baseline_head_sha: "abc123"`, both have matching `baseline_dirty_snapshot`, and each has a non-empty `dev.observed_preexisting`
- WHEN the orchestrator writes the aggregate per `commands/dev.md Step 9`
- THEN the aggregate JSON contains top-level `baseline_head_sha: "abc123"`, top-level `baseline_dirty_snapshot` matching the dispatch value, and `dev.observed_preexisting` as a UNION of both workers' lists
- AND if worker-B instead has `baseline_head_sha: "xyz999"`, the aggregate's `dev.status` is `"blocked"` with a blocking_issue citing the baseline mismatch

### AC3: files_created includes staged new files (D3)

- GIVEN a dev execution where one new file is staged (`git add new-file.py`) and a second new file is untracked (not staged)
- WHEN dev derives `dev.files_created` per `agents/dev.md` derivation rules
- THEN `dev.files_created` includes both `new-file.py` (from `git diff --cached --name-only --diff-filter=A`) and the untracked file (from `git ls-files --others --exclude-standard`)
- AND paths that appear in `baseline_dirty_snapshot` parsed path list are excluded from both sources before union

### AC4: QA files_created check mirrors the combined derivation (D3, QA side)

- GIVEN a dev-report with `dev.files_created = ["staged-new.py", "untracked-new.py"]` where `staged-new.py` was staged (in git index, not in `git ls-files --others`) and `untracked-new.py` is untracked
- WHEN QA runs the files_created provenance check at `agents/qa.md Step 5`
- THEN QA checks against the combined set (`git ls-files --others --exclude-standard` UNION `git diff --cached --name-only --diff-filter=A`) and does NOT raise `files_created_provenance_violation` for `staged-new.py`

### AC5: D1 reverse check excludes baseline_dirty_paths (not observed_preexisting) (D1 boundary)

- GIVEN a dev-report with `baseline_dirty_snapshot` containing `" M pre-dirty.md"` (a file dirty at baseline time), `dev.files_modified = []`, and `dev.observed_preexisting = ["pre-dirty.md"]`
- WHEN `git diff --name-only <baseline_head_sha>` returns `pre-dirty.md`
- THEN QA does NOT raise `files_modified_underreport_violation` for `pre-dirty.md` (it is in `baseline_dirty_paths`)
- AND if a different file `other.md` appears in diff and is absent from `dev.files_modified ∪ dev.files_created` and absent from `baseline_dirty_paths` but present in `dev.observed_preexisting`, QA DOES raise `files_modified_underreport_violation` for `other.md` (observed_preexisting is not an exclusion)

### AC6: Staged new file in files_created passes D1 reverse check (D1 union boundary)

- GIVEN a dev-report with `baseline_head_sha` set, `dev.files_modified = []`, `dev.files_created = ["staged-new.py"]`, and `baseline_dirty_snapshot = ""`
- WHEN `git diff --name-only <baseline_head_sha>` returns `staged-new.py`
- THEN QA does NOT raise `files_modified_underreport_violation` for `staged-new.py` (it is present in `dev.files_created`, which satisfies the `dev.files_modified ∪ dev.files_created` union check)
- AND no violation is fired for a diff path that appears in `dev.files_created` even if absent from `dev.files_modified`
