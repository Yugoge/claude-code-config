## Item A — bash-safety executable-context bypass (bash_context_strip.py)

**Problem statement.** Three precision gaps in `hooks/lib/bash_context_strip.py` allowed executable-context bypass of the bash safety hook:

1. BUG 1: `-c` detection used exact equality `== '-c'`. POSIX allows `-c` anywhere in a combined-flag cluster (e.g., `-lc`, `-ce`, `-xc`). Combined-flag forms were not recognised as shell contexts, letting `bash -lc 'rm -rf /'` slip past the hook.
2. BUG 2: `_find_command_word` skipped only the `sudo` token itself. A sudo invocation with option-argument pairs like `sudo -u root bash -c 'rm ...'` returned `-u` as the command word rather than `bash`.
3. BUG 3: The heredoc pipe-to-shell detector used the regex `r'\|\s*(bash|sh|zsh|dash)\b'`. The `\b` word boundary matches between `h` and `.`, so `| bash.old` was incorrectly treated as a shell context. Additionally, the regex did not recognise absolute paths like `| /bin/bash`.

**Solution.**
- BUG 1: Replaced `text.strip() == '-c'` with a compound predicate: token starts with `-` or `+` (not `--`) AND `'c' in token[1:]`. Handles all POSIX combined-flag forms.
- BUG 2: Added module-level `_SUDO_OPTS_WITH_ARG` frozenset. After detecting `sudo`, an inner loop consumes option-argument pairs, `--` terminator, and no-arg flags. Non-listed flags like `-H` are skipped without consuming their successor.
- BUG 3: Removed the regex entirely. Replaced with: `any(_first_word(seg) in _SHELL_INTERPS for seg in after.split('|')[1:])`. Uses `_basename()` internally, correctly rejecting `bash.old` while accepting `/bin/bash`.

**Test coverage.** 40 unit tests total (30 original + 10 new) in `hooks/tests/test_bash_safety_context.py`. 10 generated stubs in `tests/generated/20260524-125300-A/`. New test classes: `TestBug1CombinedFlagWithC` (3 tests), `TestBug2WrapperCommandOptions` (3 tests), `TestBug3AbsolutePathShellInPipe` (4 tests, including the `bash.old` false-positive regression).

**Key decisions / bugs caught by Codex.**
Codex found 1 in-scope real bug during review: the BUG 3 fix initially only augmented the regex rather than replacing it, leaving the false-positive path (`| bash.old`) intact. Codex fix applied: removed the legacy regex fast-path entirely. Codex also flagged 1 OBSERVATION_ONLY: `_SUDO_OPTS_WITH_ARG` is incomplete for `-p/-A/-h/-T` options (deferred, C1 class).

**QA iterations:** 2 implementation iterations (iter 2 incorporated Codex BUG 3 false-positive fix and added AC10 BA fix). QA PASS on iter 2.

**Files modified:** `hooks/lib/bash_context_strip.py`, `hooks/tests/test_bash_safety_context.py`

## Item B — Write guard sentinel schema mismatch (privilege escalation)

**Problem statement.** Four interacting bugs in the `/allow` sentinel pipeline allowed path-scoped Write grants to silently become unconditional global Write grants — a privilege escalation:

1. `_looks_regex()` classifies any string containing `.` as a regex. Dotted file paths like `/tmp/file.json` triggered `IS_REGEX=true`, causing the sentinel write block to be skipped entirely (sentinel write is skipped for regex grants by design). The sentinel was never created.
2. The bare two-token form `/allow Write /tmp/file.json` placed the path token in `bare[1]`, which was treated as a comment. The resulting sentinel was `{op: "Write"}` with no `target` field — a wildcard grant.
3. The sentinel writer always emitted `args_contain` for all operations. The matcher in `allowlist.py` keyed on `target` for Write operations. Key absence caused `.get('target')` to return `None`, and `None is None` evaluated to `True`, granting unconditional Write regardless of path.
4. The `match_sentinel_grant_for_write` function used `.get()` fallthrough semantics. When `target` key was absent, the condition `entry_target is None or entry_target == target_path` matched every entry, leaking any `args_contain`-schema entry into a wildcard.

**Solution.**
- Bug 1 (is_regex override): Force `is_regex = False` unconditionally in the `--tool` flag branch. `--tool` input is always an explicit literal declaration.
- Bug 2 (bare Write+path detection): Added `elif first == 'Write' and len(bare) >= 2 and bare[1].startswith('/'):` branch. Also removed double-quote characters from Python comments embedded in the bash `python3 -c "..."` block — these caused bash to terminate the double-quoted string early, producing no output and silently falling through to the wildcard default.
- Bug 3 (target vs args_contain writer): Replaced `if rest: entry['args_contain'] = [rest]` with key-presence logic: Write ops emit `target`, non-Write ops emit `args_contain`.
- Bug 4 (key-presence matcher): Replaced `.get()` fallthrough with explicit branching: `if 'target' in entry` -> exact string equality; `elif 'args_contain' in entry` -> legacy exact one-item list equality; `else` -> intentional bare wildcard.

**Test coverage.** 54 tests total: `hooks/tests/test_allowlist_consolidation.py` (49 tests including 5 new AC4-AC8 methods) + 8 generated stubs in `tests/generated/20260524-125300-B/`. Key regression guards: AC5 blocks path-prefix substring matching exploit; AC6 preserves intentional bare-Write wildcard; AC7/AC8 are subprocess e2e tests verifying the full hook pipeline.

**Key decisions / bugs caught by Codex.** Codex found that all 4 code fixes were correct but the generated test stubs still hard-failed (`TEST_INCOMPLETE` marker). Applied: implemented all 8 generated stubs; fixed `HOOKS_DIR` path depth from `parent*3` to `parent*4` in all stubs. Codex also identified the bash double-quote-in-comment as the critical subtlety explaining why AC7/AC8 previously failed silently.

**QA iterations:** 1. All 54 tests passed on the first QA pass. QA PASS.

## Item D — dev provenance gaps (under-reporting check, aggregate fields, staged files_created)

**Problem statement.** Three gaps in the dev/QA provenance pipeline:

- D1 (under-reporting): The QA provenance check only iterated in the over-reporting direction (verifying that `dev.files_modified` entries match actual diff). No inverse check existed: a dev could silently omit files they actually modified.
- D2 (aggregate fields): The aggregate construction rule in `commands/dev.md` omitted `baseline_head_sha`, `baseline_dirty_snapshot`, and `dev.observed_preexisting`. QA skips the provenance check when `baseline_head_sha` is absent; an empty exclusion set causes false positives; `observed_preexisting` could not be populated.
- D3 (files_created derivation): `agents/dev.md` derived `dev.files_created` using only `git ls-files --others --exclude-standard` (untracked files). **Solution.**

- D2: Updated `commands/dev.md` aggregate construction rule to include all three provenance fields with equality-verification semantics and explicit block-on-mismatch. Added to example aggregate JSON.
- D3: Changed `files_created` derivation to UNION of `git ls-files --others --exclude-standard` and `git diff --cached --name-only --diff-filter=A`. Added note that `files_modified` and `files_created` are non-disjoint by design (staged new file appears in both).

## Item push — execute-push.py atomic push script

**Problem statement.** The `&&`-chained orchestration pattern in `commands/push.md` Step 5 had a timing window: `python3 validate_sentinel.py && bash push.sh` runs as two separate processes. An attacker with filesystem access could substitute the sentinel file or grant between validation and consumption. The `scripts/execute-push.py` artifact that was designed in task 20260519-211515 to close this window had never been created as an enforcing artifact — what existed was stub pseudocode missing `--branch`, with wrong step order, and re-reading HEAD twice.

**Solution.** Complete replacement of `scripts/execute-push.py` with a 13-step atomic push protocol:
1. Parse CLI args: `--repo-hash`, `--branch`, `--remote`, `--request-id`, `[--auto]`.
2. Read `SESSION_ID` from `CLAUDE_SESSION_ID` env; exit 2 if unset.
3. Compute grant path from `_SENTINEL_BASE/repo_hash/session_id/request_id.json`.
4. Check `push.sh` existence and executable bit BEFORE any grant access; exit 2 if absent.
5. Read grant file; exit 1 if absent.
6. Parse grant JSON; exit 1 if malformed.
7. Validate all grant fields (`expires_at` with tz-aware normalization, nonce, branch, remote_name, session_id, verdict, risks); exit 1 with grant unconsumed.
8. Blocked verdict: print risks to stderr, exit 1 with grant unconsumed. Warn verdict: print risks to stderr, flush, continue.
9. Single HEAD drift check: `git rev-parse HEAD` vs `grant.head_sha`; exit 1 on mismatch with grant unconsumed.
10. `os.unlink(grant_path)`; exit 1 on failure.
11. Write push sentinel atomically via `mkstemp + os.replace` with exactly 5 keys.
12. Set `CLAUDE_PUSH_REQUEST_ID`, flush stderr with BrokenPipeError guard.
13. `os.execv(push_sh)`. On `OSError`: unlink sentinel, exit 2.

Updated `commands/push.md` Steps 4 and 5 to establish `execute-push.py` as the sole grant validator and consumer, with tilde-form invocation including `--branch`.

**Test coverage.** 10 generated stubs in `tests/generated/20260524-125300-push/`. AC9 and AC10 are grep-verifiable against `commands/push.md`. AC1-AC8 behavioral tests noted for a follow-on cycle (`hooks/tests/test_execute_push.py`).

**Key decisions / bugs caught by Codex.** Codex returned 7 findings: (BLOCKER) executable bit not set — APPLIED: chmod 0755; (MAJOR) naive `expires_at` TypeError risk — APPLIED: added `tzinfo` check with `astimezone(utc)` normalization; (MAJOR) push.md Step 5 prose operation order wrong — APPLIED: reworded; (MINOR) handoff message flush — APPLIED: `print(flush=True)` with BrokenPipeError guard. Rejected 3 findings as out-of-scope: branch-name drift check (spec mandates single SHA check at step 9 only), CLI input format validation (not in requirements), sentinel `mkdir` placement (style preference).

**QA iterations:** 1. QA PASS on first iteration.

**Files modified:** `commands/push.md`, `scripts/execute-push.py`
