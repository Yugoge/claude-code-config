## Round 1b': Close Debate Analysis of Codex Finding C2

### The Core Question
Does Codex finding C2 (writer produces `{"op":"git"}` without `args_contain`, broader grant than intended) constitute a blocking finding that prevents CLOSE: YES?

### Analysis

**Scope boundary**: The BA spec explicitly states:
- Won't Have: "Changes to `userprompt-consent-allowlist.sh` — the writer is correct."
- Won't Have: "Free-text `/allow` input validation (e.g., `/allow 你自己收尾` producing unmappable op) — separate ticket."
- Edge Case #3: "Sentinel-exists-but-no-match should suppress legacy matching — out of scope."
- Context JSON `out_of_scope_observations[1]` (logged by BA): "Free-text /allow inputs produce a sentinel with op equal to the entire free-text token. This never matches any bash subcommand's first token. Silent failure — separate usability ticket."

The writer producing `{"op":"git"}` without `args_contain` is **pre-existing behavior** of `userprompt-consent-allowlist.sh`. This cycle does not touch that file. The BA spec explicitly marked writer changes as Won't Have.

**Pre-existing equivalence**: Before this cycle, the legacy `/allow` grant used `substr_only` matching (allowlist.py line 259). A legacy grant with pattern `"git"` written by the writer would also match `git reset --hard` (since "git" is a substring of "git reset --hard"). The sentinel system's structural `{"op":"git"}` grant is semantically equivalent to what the legacy system would have allowed — both allow any `git` subcommand. This cycle does not introduce a new security hole; it makes an existing grant mechanism available to the privilege guard.

**The requirement IS satisfied**: The user's requirement is "agents must be able to execute that command." When `/allow git branch -D foo` is run, an agent CAN run `git branch -D foo` (exits 0, confirmed by Codex live simulation). The requirement is met. The fact that the grant is broader than the specific command is a pre-existing design choice in the writer, not a failure of this cycle's fix.

**The separate-ticket disposition**: Dev report recommendations list "userprompt-consent-allowlist.sh writer should emit `args_contain` for multi-token /allow inputs (codex finding 1)" as a separate ticket. BA logged the same in out_of_scope_observations. The writer narrowing is a usability/tightening improvement — it does not block the cycle's core correctness.

**Verdict on C2**: Out of scope for this cycle. Matches BA spec Won't Have list. Does not block CLOSE: YES. A separate ticket should track writer narrowing.

**Verdict on C3** (integration test gap): Same reasoning as C2. The tests validate the structural behavior of `_check_git_allowlist()` against the sentinel library — which is what the BA spec prescribed (pytest fixture writes sentinel, calls function directly). An integration test spanning writer→guard is valuable but is a future improvement, not a requirement of this cycle.

**Verdict on C6** (session_id guard ordering): The "if not sid: return False" guard was present in the original `_check_git_allowlist()` before this cycle and is preserved in the "After (proposed)" code in the BA spec. The BA spec's proposed code explicitly includes this guard. Any fix would expand scope beyond the BA spec.
