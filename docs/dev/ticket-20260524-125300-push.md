## Goal

Create `scripts/execute-push.py` — a single Python script that consolidates Steps 4+5 of `/push` into one atomic process: it validates the push-analyst grant, consumes it, writes the Chain-B sentinel, checks HEAD drift, and then replaces itself with `push.sh` via `os.execv`. This eliminates the timing window that exists when `&&` chains a validator exit with a separate `bash push.sh` invocation.

---

## Context

The `/push` command's agentic dispatch protocol (commands/push.md) specifies a single-process exec pattern in its Step 5 bash block: write Chain-B sentinel then `CLAUDE_PUSH_REQUEST_ID="${REQUEST_ID}" exec ~/.claude/hooks/push.sh`. However, the actual push invocation during a live cycle was performed as `python3 validate_sentinel.py && bash push.sh` — a two-process pattern with a measurable window.

Two additional failure modes compound this:

1. When Steps 4+5 are delegated to a subagent, that subagent may correctly refuse to write the Chain-B sentinel (it identifies the write as orchestrator-only authorization). The subagent cannot hold the auth chain.
2. Subagent dispatch latency (observed 60s+ on cold paths) can cause the push-analyst grant's 180s TTL to expire before the orchestrator reads it — or can cause the 60s sentinel mtime window in `push.sh` to expire before `push.sh` reaches the sentinel check.

The fix is a single dedicated script that the orchestrator invokes directly (no subagent delegation), which performs grant validation, grant consumption, sentinel write, HEAD-drift check, and `os.execv` in one uninterrupted Python process. Because `os.execv` replaces the process image, the sentinel is written and `push.sh` reads it within the same kernel process — the 60s mtime window cannot expire between them.

**Terminology distinction**: the push-analyst writes the *grant*; `execute-push.py` converts an approved/warn grant into the *Chain-B success sentinel*. These are different files with different lifetimes.

## Edge Cases & Risks

1. **Sentinel orphan on execv failure**: if `os.execv` raises (push.sh path wrong, permissions issue), the sentinel is already written. The try/except must unlink it before returning exit 2.
2. **Grant TTL race**: the grant is validated at parse time; by execv time it could technically expire — acceptable because the 180s TTL is generous and the sentinel mtime clock in push.sh is independent.
3. **Branch name with `/` in encoded form**: `branch.replace("/", "__")` must be applied consistently; mismatch between the encoded form used for sentinel path and the raw form bound in sentinel JSON's `branch` field would cause push.sh binding check to fail. The sentinel `branch` field must use the RAW branch name (matching `_CHAIN_B_BRANCH_RAW` in push.sh:94), not the encoded form.
4. **`verdict=warn`**: script must display risks[] to stderr as warnings, flush stderr, then proceed (not abort). Only `verdict=blocked` aborts.
5. **`expires_at` with trailing `Z`**: `datetime.fromisoformat` in Python < 3.11 does not parse trailing `Z`; use `replace("Z", "+00:00")` for Python 3.8–3.10 compatibility.
6. **Branch drift between Step 2 and execv**: `--branch` is the Step 2 snapshot. If the operator changes branches between dispatch and execution, grant field `branch` will not match `--branch` and M5 will catch it at step 7. This is intended.
7. **Multiple concurrent /push invocations**: the sentinel path is branch-scoped (not request-id-scoped). A second concurrent invocation would overwrite the first sentinel. This is an existing design limitation in push.sh; execute-push.py does not worsen it.
8. **`os.execv` on Linux replaces env**: `os.environ` mutations before `os.execv` ARE inherited by the child because `os.execv` passes `os.environ` — dev must NOT use `os.execle` with a separate env dict unless it includes all current env vars.
9. **Blocked grant intentionally unconsumed**: a blocked grant left on disk cannot be replayed to authorize a push — it will always abort at verdict=blocked. A new `/push` generates a new request-id and thus a fresh grant path.
10. **Stderr buffering across execv**: Python's stderr may buffer output when not connected to a terminal. Flush stderr before `os.execv` so warn-path risk messages are not lost in the buffer when the process image is replaced.

## Security Window Analysis

The `&&` pattern creates three distinct attack surfaces:
1. **Sentinel staleness window**: between validator process exit and push.sh start, the sentinel's mtime ages. On a loaded system, scheduler delay alone can push this into the seconds range — approaching the 60s limit under adversarial load.
2. **Grant replay window**: if the grant is NOT unlinked before push.sh starts, a concurrent process that holds the grant path could re-trigger validation. The `exec` pattern collapses this window to zero.
3. **State substitution window**: between validator exit and push.sh invocation, the `/tmp/agentic-commit/push-analyst/` directory is writable. An attacker with local write access could replace the sentinel file with one bound to a different HEAD/branch/remote. `exec` eliminates this window entirely.

`os.execv` is a POSIX primitive that replaces the current process image without creating a new PID. The sentinel write and push.sh sentinel-read happen within the same process lifetime. There is no inter-process gap.

## Script Interface Specification

**Invocation**:
```
python3 ~/.claude/scripts/execute-push.py \
  --repo-hash <REPO_HASH> \
  --branch <BRANCH> \
  --remote <RESOLVED_REMOTE> \
  --request-id <REQUEST_ID> \
  [--auto]
```

All four named arguments are required. `CLAUDE_SESSION_ID` must be set in the environment.

**Exit codes**:

| Exit | Meaning |
|------|---------|
| 0    | Never returned — successful path calls `os.execv` (process replaced by push.sh) |
| 1    | Grant validation failed, HEAD drift, blocked verdict, or grant-consume failure |
| 2    | Setup/infrastructure error: CLAUDE_SESSION_ID unset, bad CLI args, push.sh missing/not executable, os.execv OSError after sentinel write |

**Operation order** (13 steps — all abort conditions must pass before sentinel is written):

1. Parse CLI args (`--repo-hash`, `--branch`, `--remote`, `--request-id`, `[--auto]`); abort exit 2 if required args missing
2. Read `CLAUDE_SESSION_ID` from env; abort exit 2 if missing or empty
3. Compute grant path: `/tmp/agentic-commit/push-analyst/<repo-hash>/<session-id>/<request-id>.json`
4. Check `~/.claude/hooks/push.sh` exists and is executable; abort exit 2 if not — NO sentinel written before this step
5. Read grant file from grant path; abort exit 1 if not found
6. Parse grant as JSON; abort exit 1 if invalid JSON or not a dict
7. Validate grant fields: `expires_at` not past, `nonce==request-id`, `branch==--branch arg`, `remote_name==--remote arg`, `session_id==CLAUDE_SESSION_ID`; abort exit 1 if any mismatch (grant NOT consumed)
8. Act on verdict: `blocked` → print `risks[]` to stderr, abort exit 1, grant NOT consumed; `warn` → print `risks[]` to stderr as WARNING, flush stderr, continue
9. Check HEAD drift: `git rev-parse HEAD` must equal `grant.head_sha`; abort exit 1 if mismatch (grant NOT consumed)
10. Consume (unlink) grant file; abort exit 1 on failure
11. Write Chain-B sentinel atomically: `mkstemp` in sentinel dir, write JSON (exactly 5 keys: `result="PASS"`, `request_id`, `head` = current HEAD verified at step 9, `branch` = raw branch name, `remote`), `os.replace` to final path at `/tmp/agentic-commit/push-analyst/<repo-hash>/<branch-encoded>-chainB.validated.sentinel.json`
12. Set `os.environ["CLAUDE_PUSH_REQUEST_ID"] = request_id`; flush `sys.stderr`
13. `os.execv(push_sh_path, [push_sh_path, remote] + (["--auto"] if auto else []))` — if raises `OSError`: unlink sentinel, return exit 2

**Grant path**: `/tmp/agentic-commit/push-analyst/<repo-hash>/<session-id>/<request-id>.json`

**Sentinel path**: `/tmp/agentic-commit/push-analyst/<repo-hash>/<branch-encoded>-chainB.validated.sentinel.json`
where `branch-encoded = branch.replace("/", "__")` and sentinel `branch` field = raw branch name.
