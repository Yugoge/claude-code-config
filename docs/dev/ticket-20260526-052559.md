# BA Specification: Close Layer 1.F bypass paths for write-bulk-commit-sentinel.py

**Request ID**: 20260526-052559
**Created**: 2026-05-26T05:25:59Z
**Tier**: STANDARD

## Goal

Two bypass paths remain in `pretool-bash-safety.sh` Layer 1.F that allow an agent to forge a bulk-commit sentinel without authorization: (1) invoking `write-bulk-commit-sentinel.py` via absolute path, which bypasses the outer guard trigger entirely; (2) dynamically constructing the sentinel JSON path in `python3 -c` to avoid the literal `/tmp/...` trigger. Additionally, `agents/changelog-analyst.md` is missing an explicit DO NOT item prohibiting both sentinel forging and use of the `auto-bulk:` commit prefix in `BULK=false` mode.

## Context

The primary bypass path (changelog-analyst using `auto-bulk:` prefix without a sentinel) was closed by commits 1238f42 + 41063a1, which added `_has_bulk_commit_sentinel()` to `pretool-git-privilege-guard.py`. The sentinel is written by `scripts/write-bulk-commit-sentinel.py`, which is invoked by `/commit --bulk` (Step 5, `commands/commit.md`). Layer 1.F of `pretool-bash-safety.sh` was designed to block unauthorized writes to the sentinel file; however, two bypass paths remain that allow an agent to forge the sentinel, and changelog-analyst.md lacks an explicit prohibition on using the `auto-bulk:` prefix in non-bulk mode.

**Primary fix verified**: Commits 1238f42+41063a1 correctly added `_has_bulk_commit_sentinel()` check inside `_evaluate_commit()`. An `auto-bulk:` commit without a valid sentinel now blocks at `pretool-git-privilege-guard.py:562`. This primary path is CLOSED.

**Residual gaps (this ticket)**: Two sentinel-forging bypass paths in Layer 1.F + one missing changelog-analyst DO NOT item.

## Edge Cases & Risks

- **Regression risk — relative-path bare invocation**: Extending the outer guard trigger to also match `write-bulk-commit-sentinel\.py` string must not block the RELATIVE-PATH bare invocation that is legitimate (`scripts/write-bulk-commit-sentinel.py` from `/commit --bulk` flow). The fix must: (1) extend the OUTER trigger to also fire on script-name presence, AND (2) keep the bare-writer allowlist as relative-path-only (which already denies absolute-path forms). This combination means: relative-path bare invocation → outer trigger fires → bare-writer allowlist allows; absolute-path → outer trigger fires → bare-writer allowlist rejects → DENY branch.

- **Critical regression risk — `/commit --bulk` Step 5 venv-activate compound form**: `commands/commit.md:81` invokes the sentinel writer as `source venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py`. After M2 fix (outer trigger extended to match script name), this command WOULD fire the outer guard. It is compound (`&&`), so `_bulk_is_compound=1` → deny branch fires → `/commit --bulk` BROKEN. The fix must add a third allowlist condition inside Layer 1.F that recognizes this specific canonical venv-activate compound form. The pattern is sufficiently specific: `^source[[:space:]]+venv/bin/activate[[:space:]]*&&[[:space:]]*python3?[[:space:]]+/root/\.claude/scripts/write-bulk-commit-sentinel\.py$` — this matches the canonical `/commit --bulk` Step 5 invocation and nothing else that could be a bypass. Dev MUST verify that this allowlist entry does NOT create a new bypass (i.e., the `&&` after the venv-activate cannot be exploited because the pattern anchors on the trailing `write-bulk-commit-sentinel.py` with no further commands allowed after it).
- **arch-2b: Fully-dynamic construction** (no complete protected pathname, no script name in command text): not reliably blockable by targeted command-text filtering. Technically blockable only via broad, high-false-positive interpreter/write bans (blocking all `python -c`, all heredocs, all Bash writes), which are out of scope for this ticket. The agent-level prohibition (M3/M4 in changelog-analyst.md) is the appropriate and feasible control layer for this vector. Explicitly listed in Won't Have.
- **`python3 -c` with literal `/tmp/` path**: Already blocked by existing Layer 1.F (outer guard fires on `/tmp/` literal, compound check blocks `python3 -c`). No change needed for this case.

## Technical Hints

- Affected files:
  - `hooks/pretool-bash-safety.sh` — Layer 1.F outer guard trigger at line 662 and allowlist section at lines 729–751
  - `agents/changelog-analyst.md` — DO NOT section, append items 13 and 14
- The fix for M1 and M2 in `pretool-bash-safety.sh` is to extend the outer `if echo "$COMMAND" | grep -qE '...'` condition at line 662 with a second OR-condition that matches `write-bulk-commit-sentinel\.py` appearing anywhere in the command. Example: `|| echo "$COMMAND" | grep -qE 'write-bulk-commit-sentinel\.py'`
- The bare-writer allowlist at line 732 must NOT be extended to include absolute paths — absolute-path invocations should fall through to the DENY branch.
- **CRITICAL: Add a third allowlist condition for the canonical `/commit --bulk` venv-activate form.** After M2, the command `source venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py` (from `commands/commit.md:81`) would fire the outer guard (script name match) AND be flagged compound (`&&`), breaking `/commit --bulk`. Dev MUST add a third allowlist check (`_bulk_is_canonical_commit_bulk`) using a FULL-LINE anchored pattern that matches ONLY this specific form:
  ```
  ^source[[:space:]]+venv/bin/activate[[:space:]]*&&[[:space:]]*python3?[[:space:]]+/root/\.claude/scripts/write-bulk-commit-sentinel\.py$
  ```
  This pattern: (1) anchors on `^` so it must start with `source venv`; (2) anchors on `$` with NO optional argument group — no arguments are permitted after the script name; (3) fixes the unescaped dot: `/root/\.claude/` not `/root/.claude/`; (4) only matches the absolute-path canonical form, not relative-path forms; (5) blocks `--sid FORGED` (no arg group → NO MATCH), `$(...)` injection (same reason), and any other argument form. **The canonical `/commit --bulk` Step 5 form at `commands/commit.md:81` passes ZERO arguments to `write-bulk-commit-sentinel.py` — verified by file read.**

  **CRITICAL — implementation must use `[[ $COMMAND =~ $REGEX ]]` not `echo | grep -qE`**: `grep -qE` is line-oriented; `^` matches start-of-any-line, so a command string containing a newline before the canonical form would match the second line even though line 1 is arbitrary. Bash `[[ $COMMAND =~ $REGEX ]]` uses POSIX ERE against the entire string and is NOT line-oriented — newline injection is blocked. The M5 allowlist condition must therefore be implemented as a bash `[[ =~ ]]` test, not `echo "$COMMAND" | grep -qE`.

  Bash test verification (run against the corrected regex):
  - `echo "source venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py" | grep -qP '...'` → MATCH (canonical no-arg form allowed)
  - `echo "source venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py --sid FORGED" | grep -qE '...'` → NO MATCH (arg injection blocked)
  - `echo "source venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py \$(touch /tmp/foo)" | grep -qE '...'` → NO MATCH (\$() injection blocked)
  - `echo "source venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py && touch /tmp/foo" | grep -qE '...'` → NO MATCH (append-command bypass blocked)
  - Newline test with `[[ =~ ]]`: `CMD=$'evil\nsource venv/bin/activate && python /root/.claude/scripts/write-bulk-commit-sentinel.py'; [[ "$CMD" =~ $REGEX ]]` → NO MATCH (newline injection blocked by bash `[[ =~ ]]`)
