## Goal

Modify `/usr/local/bin/playwright-mcp-stealth` (CDP-mode branch only) to (a) hold an exclusive `flock`-style file lock for the lifetime of each `playwright-mcp` child and (b) perform a bounded best-effort cleanup of stale Chrome page targets against the wrapper's resolved `--cdp-endpoint` URL (normally `http://127.0.0.1:8080` in production / in-place deployment, but resolved from argv — NOT hardcoded — so AC3 can exercise the no-Chrome path with a disposable unreachable endpoint such as `http://127.0.0.1:9`) before the child launches. Also ensure the persistent Chrome service's user-data-dir `/tmp/chrome-debug-profile` is no longer eligible for nightly rm by the daily tmp-cleanup script at `/usr/local/sbin/tmp-cleanup.sh`. The change to this script is CONDITIONAL on live-file state at edit-time (see R7 below). This is in-place deployment.

## Context

A single Chrome page target wedges in renderer-side futex-wait. Chrome itself stays healthy, but `connectOverCDP()` auto-attaches to all existing page targets at connect time and hangs ~4s+ per `Runtime.evaluate` / `Page.getFrameTree` against the dead target. The persistent Chrome model means dead pages accumulate; no journald event fires (it's a hang, not a crash). Concurrent MCP sessions cannot blindly close each other's pages.

The locked design is the product of a multi-turn discussion (orchestrator + 3 rounds of OpenAI Codex adversarial review) — see the validated artifacts cited in Technical Hints below. The investigation is closed. This spec formalizes the design for dev to implement.

## Requirements (MoSCoW)

### Must Have

- **R1 (Part A1)** — Wrapper opens `/run/playwright-mcp.lock` via a numeric file descriptor and acquires an exclusive blocking lock on it. The descriptor MUST be inherited across the eventual `exec`/process-launch of the `playwright-mcp` child (no close-on-exec).
- **R2 (Part A2)** — Lock acquisition uses a 30-second timeout (exclusive blocking). On timeout, wrapper exits with status 75 (EX_TEMPFAIL) without launching the child, emitting a stderr diagnostic.
- **R3 (Part A3)** — Failure to open the lock file itself (e.g., `/run` not writable, fd exhaustion) also exits 75. NO fallthrough to unlocked execution.
- **R4 (Part A4)** — After lock acquisition, wrapper performs bounded best-effort cleanup against the CDP endpoint that the WRAPPER WAS INVOKED WITH. The endpoint is resolved from the wrapper's argv (`--cdp-endpoint <URL>`), NOT hardcoded — this is what makes AC3 testable with a disposable unreachable endpoint:
  - Resolve `CDP_URL` from the `--cdp-endpoint` argv value passed to this wrapper invocation. In production / in-place deployment this URL is normally `http://127.0.0.1:8080`. Cleanup probes use this resolved URL (NOT a hardcoded string).
  - Probe `${CDP_URL}/json/version` with 1s connect/read timeout. If unreachable, SKIP cleanup; proceed to child launch.
  - Fetch `${CDP_URL}/json` (1s timeout). Parse JSON array. Filter to `type == "page"` AND url not in {empty, `about:blank`, `chrome://*` prefix, `devtools://*` prefix}.
  - For each surviving target, POST to `${CDP_URL}/json/close/<id>` with 2s per-call timeout. Per-target failures silently ignored.
  - Poll `${CDP_URL}/json` every ~500 ms for up to ~3 s, waiting for closed ids to disappear. Persistent presence logged but non-fatal.
  - All parse / network errors silently tolerated; flow continues.
- **R5 (Part A5)** — Existing `--config` argument handling at wrapper lines 90-94 is preserved for the CDP-mode branch. When `/root/.claude/playwright-config.json` exists, the child receives `--config /root/.claude/playwright-config.json`.
- **R6 (Part A6)** — Child is launched with original args (preserving `--cdp-endpoint` and all passthroughs) AND `--config` when applicable. Lock fd remains open across launch; OS releases lock on child exit (normal / signal / error) via close-of-final-open. Watchdog at `@playwright/mcp` `watchdog.js:26-38` provides the 15s forced-exit upper bound.
- **R7 (Part B) — CONDITIONAL on live-file state at edit-time**: the user-need is "the persistent Chrome service's `/tmp/chrome-debug-profile` is no longer eligible for nightly rm by tmp-cleanup.sh". Dev MUST first inspect the live `/usr/local/sbin/tmp-cleanup.sh` and branch:

No other lines of the script change. AC7/AC8 are evaluated against the post-edit state.
  - **CASE 7b (already-protected state)**: if the Playwright-profiles `run_tmp` find expression OR-chain does NOT contain `chrome-debug-profile` AND the script's EXCLUDE_PRUNE clause (currently at line 69 of the 261-line live file: `-path /tmp/chrome-debug-profile -o`) already protects `/tmp/chrome-debug-profile` from every top-level /tmp sweep, then Part B is a documented NO-OP. ## Edge Cases & Risks

- **E1 — Lock-file pre-existence**: `/run/playwright-mcp.lock` may not exist yet (this is the first cycle that creates it). The open must create-if-missing. If `/run` is tmpfs (it is on this host), the file vanishes on reboot — recreated on next wrapper invocation. No persistence issue.
- **E2 — Concurrent first invocation**: two wrappers racing to first-create the lock file are both safe — `flock` semantics serialize on the underlying inode regardless of which process created it.
- **E3 — Child does not exit when stdin is open**: `@playwright/mcp` `setupExitWatchdog` (`watchdog.js:26-38`) plus 15 s forced-exit fallback at `program.js:41-45` bound the worst-case lock-hold to ~15 s after stdin EOF. Codex Round 3 Q4 validated this explicitly.
- **E4 — Lock held by zombie / orphaned wrapper**: an orphaned wrapper that never exits would hold the lock until killed. Mitigation: 30 s acquisition timeout converts this into a clean exit-75 for the second wrapper, surfacing the issue rather than deadlocking. The leak itself is a Won't-Have (user's "in-place deployment" decision).
- **E5 — `/run/playwright-mcp.lock` permissions**: opened by whoever runs the wrapper (root in current deployment). If a non-root user later runs the wrapper, ensure they have access — outside scope of this cycle (no non-root invocation today).
- **E6 — JSON parse failures**: malformed `/json` output (extremely unlikely from Chrome but defended) → treat as empty list → no targets closed → child launches normally. R4 explicitly tolerates parse errors.
- **E7 — `curl` not present**: `/usr/bin/curl` is verified present on the host. If absent on a future host, cleanup would fail silently per R4's tolerance; child still launches. Acceptable.
- **E8 — `flock` not present**: `/usr/bin/flock` (util-linux 2.39.3) verified present. If absent, the lock-open phase would fail at R1, hitting R3's exit-75 path. Acceptable.
- **E9 — Cron edit syntax slip (applies only in CASE 7a)**: in CASE 7a the closing-paren relocation is the highest-risk part of Part B. A typo (e.g., paren on wrong line, escaped wrong) leaves the script broken. Dev MUST run `bash -n` post-edit to verify (AC7). In CASE 7b (no-op) there is no edit and no syntax risk.
- **E10 — Existing leaked playwright-mcp processes**: the ~23 leaked processes pre-existing this cycle are out-of-scope. Their behavior with the new lock: each is already past wrapper launch, so they don't contend on `/run/playwright-mcp.lock`. The first NEW wrapper invocation will acquire the lock freely. No conflict.

## Acceptance Criteria

### AC1: Wrapper concurrency — second invocation times out cleanly AND does not spawn a child
- GIVEN the wrapper has just been invoked in CDP mode and is holding the lock with a live `playwright-mcp` child
- WHEN a second wrapper invocation is launched in CDP mode within the same time window
- THEN the second invocation waits up to 30 seconds for the lock, then exits with status 75 without launching its own `playwright-mcp` child, and emits a stderr diagnostic indicating lock-acquisition timeout
- AND throughout the timeout window QA captures pgrep/process-tree snapshots (recommended cadence: every 5 s while the second wrapper is waiting; minimum a single snapshot taken in the last 5 s before the second wrapper exits 75) that confirm "no `/usr/bin/playwright-mcp` process and no `node` child with parent-PID equal to the SECOND-WRAPPER-PID exists at any sampled moment before the second wrapper exits 75". Recipe: capture SECOND_WRAPPER_PID by `$!` immediately after launching the second wrapper in background; sample with `pgrep -P "$SECOND_WRAPPER_PID"` and also with `ps --ppid "$SECOND_WRAPPER_PID" -o pid,ppid,comm,cmd` — both MUST be empty of any `playwright-mcp` or `node` entry for every sample taken before exit. Exit-75 alone is insufficient evidence; the parent-PID snapshots are the binding evidence.

### AC2: Wrapper lock release — next invocation acquires cleanly
- GIVEN the first wrapper's `playwright-mcp` child has exited (normal / signal / error)
- WHEN a fresh wrapper invocation is launched in CDP mode
- THEN it acquires the lock without intervention and launches its child normally

### AC3: Wrapper no-Chrome path — cleanup fails silently, child still launches
- GIVEN the wrapper is invoked with a DISPOSABLE unreachable `--cdp-endpoint` (e.g., `http://127.0.0.1:9` or any other unbound TCP port verified empty via `ss -lnt | grep -c ':9 '` returning 0 immediately before the test) — this decouples AC3 from the live `chrome-debug.service` state entirely and satisfies AC10's no-service-restart constraint
- WHEN the wrapper is invoked in CDP mode against that disposable unreachable endpoint
- THEN the cleanup phase's `/json/version` probe must fail within approximately 1 second (per the bounded 1 s connect-timeout in R4); cleanup is skipped without aborting the run; and the `playwright-mcp` child still launches normally (verifiable by `pgrep -af /usr/bin/playwright-mcp` showing a fresh child whose argv contains the disposable endpoint URL). NOTE: the disposable endpoint will subsequently fail `playwright-mcp`'s own CDP connect — that downstream failure is expected and is NOT evidence against AC3; AC3's scope ends at "cleanup probe failed fast AND child process was launched", which is what the AC literal text asserts.

### AC4: Wrapper non-regression — single Claude session works end-to-end
- GIVEN a single Claude Code session using the wrapper in CDP mode with the existing CDP endpoint
- WHEN the session performs end-to-end Playwright work (navigation, evaluation, screenshot)
- THEN the work completes without the wrapper introducing new failures, hangs, or argv corruption
- NOTE: AC4 validates a NEW post-launch session; preservation of pre-existing non-system page targets (e.g., the currently-active localhost:8765 Travel Plan tab) is explicitly NOT required and may be transiently closed by the cleanup phase on first invocation. This is the user-accepted "就地部署" impact.

### AC5: Wrapper --config preservation — verified on CHILD PID, not wrapper PID
- GIVEN `/root/.claude/playwright-config.json` exists
- WHEN the wrapper is invoked in CDP mode and reaches the child-launch line (wrapper line 103 uses direct invocation — NOT exec-replace — so the bash wrapper PID and the playwright-mcp/node child PID are two distinct processes; this is confirmed by codex Round 3 Q1 and by BA scope analysis)
- THEN the `playwright-mcp` CHILD process (NOT the bash wrapper parent) receives `--config /root/.claude/playwright-config.json` in its argv. Verification recipe: (1) identify the wrapper PID as the bash shell whose argv contains `playwright-mcp-stealth`; (2) identify the CHILD PID by listing the wrapper's direct descendants (`pgrep -P "$WRAPPER_PID"`) and filtering to the entry whose `comm` is `node` or `playwright-mcp` (NOT bash); (3) read `/proc/<child-pid>/cmdline` (NUL-separated argv) via `tr '\0' '\n' < /proc/<child-pid>/cmdline` and assert the sequence `--config` followed by `/root/.claude/playwright-config.json` appears as two consecutive tokens. Plain `ps -ef | grep --config` is INSUFFICIENT because it cannot disambiguate the wrapper's passthrough argv from the child's actual argv.
