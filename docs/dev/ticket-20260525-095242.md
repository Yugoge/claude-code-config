## Goal

Close the QA-reported AC-04 bypass from prior cycle 20260524-205206: the Layer 1.F allowlist branch in `hooks/pretool-bash-safety.sh` matches a read-only verb (`ls|stat|cat|...`) as the PREFIX of a compound command but does not assert the REMAINDER is also read-only, so `ls <flag> ; python3 -c <write to flag>` silently returns exit 0 and lets a model create `/tmp/claude-bulk-allowed-<sid>.flag` without user authorization — defeating the M4 user-gated bulk-disable contract.

## Evidence (Contract A)

- **Observed** (verbatim from QA + user): "复合命令绕过" — compound-command bypass: `ls /tmp/claude-bulk-allowed-<sid>.flag ; python3 -c 'open("/tmp/claude-bulk-allowed-<sid>.flag","w").write("y")'` returns hook exit 0.

- **Measured** (BA reproduced this cycle, observed via subprocess against the current hook):
  - Bypass pattern with separator `;`: hook exit `0` (BUG)
  - Bypass pattern with separator `&&`: hook exit `0` (BUG)
  - Bypass pattern with separator `||`: hook exit `0` (BUG)
  - Bypass pattern with separator `|`: hook exit `0` (BUG)
  - Pure `ls /tmp/claude-bulk-allowed-<sid>.flag` (no separator): hook exit `0` (correct — read-only allow is intentional)
  - Pure `python3 -c 'open(...).write("y")'` (no `ls` prefix, path inside python source): hook exit `2` (correct — current write-only path is already caught)
  - Root location: `hooks/pretool-bash-safety.sh:648-671` (Layer 1.F).
  - Root cause in code (verbatim): at lines 651-654 the allowlist branch (b) matches
    ```
    grep -qE '(^|[[:space:];|&])(ls|stat|cat|file|wc|head|tail|grep|jq|find|test|\[)\s+[^>]*/tmp/claude-bulk-'
    ```
    against the whole command and then negates `grep -qE '(>|>>|tee\b)'` against the whole command — never splits on shell separators, never re-evaluates per segment. A compound command with a read-only verb prefix slips through because (a) the prefix-anchor regex `(^|[[:space:];|&])` matches `ls` at the very start AND `[^>]*/tmp/claude-bulk-` matches anywhere through `;` characters until the protected path, and (b) the redirect-veto regex looks for `>`/`>>`/`tee` only, missing `python -c`, `node -e`, `perl -e`, `ruby -e`, `sh -c`, `bash -c`, `eval`, `source`, here-docs, process substitution, etc.

- **Expected** (source: user's verbatim requirement + Codex review of fix proposal):
  - Single pure read-only verb on the protected path (no separators, no substitution, no nested shell, no redirect) → ALLOW (exit 0). Concretely: command matches `^[ \t]*(ls|stat|cat|file|wc|head|tail|grep|jq|find|test|\[)\b` and contains NO shell control characters (`;`, `&&`, `||`, `|`, `&`, `|&`, `$(`, `` ` ``, `<(`, `>(`, `\n`, `<<`, `<<<`) and NO redirect (`>`, `>>`, `tee`) on the protected path.
  - Anything else (compound command of any kind, command substitution, process substitution, here-doc, recursive shell `-c`/`-e`, variable-assignment-then-use, etc.) → DENY (exit 2) with `bulk-commit-auth-flag-write` in stderr.
  - Allowlist (a) `scripts/write-bulk-commit-sentinel.py` invocation: kept, but tightened — only the BARE script invocation form (no separators, no substitution, no nested shell) is allowed. Compound forms involving the script and a protected-glob mention are denied.

- **Gap**: Layer 1.F currently treats prefix-read as command-read; the fix replaces that with whole-command-read, so any compound command involving the protected path is denied unless it is exactly a single pure-read.

## Edge Cases & Risks

- **Risk 1** (over-tight ALLOW): pure `ls -la /tmp/claude-bulk-allowed-*.flag` (wildcard expansion) might fall outside the strict "no shell control structure" predicate if the glob expands to multiple files. Mitigation: wildcards (`*`, `?`, `[...]`) are NOT in the no-control list — they expand BEFORE the hook sees the command from Claude (the hook sees the literal command string the model emitted), and the model emits either a literal path or a literal glob string; both are fine. Dev should verify pytest case with `ls /tmp/claude-bulk-allowed-test-sid.flag` (literal) passes.
- **Risk 2** (over-tight DENY for the official writer): the bare `python3 scripts/write-bulk-commit-sentinel.py` invocation does NOT mention the protected glob in its command text — the script writes the sentinel internally from Python — so Layer 1.F is not triggered at all for this invocation. The M2 tightening only affects the case where the command text BOTH invokes the script AND mentions the protected glob in the same line, which is the compound-bypass shape we want to deny.
- **Risk 3** (regex false-positive on innocuous text): the no-control-structure check must distinguish `&&` operator from `\\&\\&` literal, etc. Mitigation: dev implements the predicate with care; pytest negative controls (g_hook_6_bash_unrelated_path retained, plus a new control case for compound on unrelated path) cover the false-positive surface.
- **Risk 4** (implementation choice — bash regex vs Python helper): codex finding #1 PROPOSED_FIX recommends an anchored single-command regex in bash (no shell parsing). This is the recommended approach because (a) it keeps the hook self-contained, (b) avoids spawning a Python subprocess per Bash tool call (latency), (c) matches the existing hook style. Dev may use a small Python helper if preferred for predicate readability, but bash regex is the leaner default. The AC validates BEHAVIOR (exit code + stderr label), not implementation form, so either is acceptable.
- **Risk 5** (regression on legitimate read flows): if any existing tests use a compound `ls <flag>; <something>` pattern for diagnostic purposes, they will break. Mitigation: review and update any such test (none currently exist per grep; if dev finds one, treat as the test's bug, not the hook's).
