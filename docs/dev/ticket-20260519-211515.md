## Context

The triage that produced this cycle's scope is saved at
`/var/tmp/codex-outputs/codex-triage-546511-1779225178.txt`. It enumerated
items A–H over the just-closed cycle 20260519-151734 and classified D + H
as "do-next, code-fix"; B/E/F as "do-eventually"; C/G as DROPPED (would
either loosen permissions or chase pre-existing repo dirt outside this
diff). Item A is live-reproduction work that requires main-agent context
and orchestrator authority — outside BA/dev scope.

D's root cause is two functions in the same file applying different
literal-match semantics to the same grant file. PreTool `read_grant`
(hooks/lib/allowlist.py:139) passes `literal_policy="exact_or_substr"`
to the shared `_match_loaded_grant`; PostTool `consume_grant_for_posttool`
branch 3 (hooks/lib/allowlist.py:286-292) hard-codes exact equality.
A user typing `/allow Re` writes grant `{pattern: "Re", is_regex: false}`;
PreTool matches `"Re" in "Read"` → True, hook exits 0 (request allowed);
PostTool tests `"Re" == "Read"` → False, grant is NOT unlinked, grant
persists past intended single-use. This is a single-step permission-grant
leakage — narrow blast radius (one grant file, one session) but breaches
the "single-use" contract documented in `userprompt-consent-allowlist.sh`
line 12 ("Stop hook (stop-cleanup-allowlist.sh) wipes any unconsumed flag")
and the file-header docstring of `consume_grant_for_posttool` which
explicitly names the case (`Branch 3 (non-Bash, literal): exact-only ...
— prevents /allow Write consuming TodoWrite`).

H's root cause is that `posttool-subagent-track.py` infers "which canonical
step is the dispatch for" from the workflow bookmark's `in_progress`
status. When the orchestrator emits TodoWrite (Step N pending → in_progress)
and Agent (matching Step N) in a single response, the PostToolUse:Agent
tracker reads the bookmark before posttool-todo-sequence.py has written
the new `last_todos`. The tracker sees the old in_progress step (N-1),
checks canonical step N-1's `subagent_call` (often absent), and exits
early via `_resolve_context` (line 357-358) without writing
`subagent_calls[N-1]=True`. Subsequently, the next TodoWrite (Step N →
completed) is blocked at Gate 4 (`pretool-todo-validate.py:251-270`)
because `state.subagent_calls.get(str(N))` is False. Mid-cycle
20260519-151734 the orchestrator had to manually patch the bookmark to
unstick the workflow.

Therefore `read_grant("Read", sid)` with grant `{pattern: "Re"}` returns True.
- **Measured (PostTool)**: `hooks/lib/allowlist.py:291-292` (consume_grant_for_posttool branch 3, non-Bash literal): `matched = (pattern == tool_name)`. Therefore consumption with grant `{pattern: "Re"}` and `tool_name = "Read"` returns False; grant file is NOT unlinked.
- **Expected**: PreTool and PostTool MUST use identical match semantics for non-Bash literal grants so that any tool call that PASSES PreTool also CONSUMES the grant at PostTool. Source: design contract documented in `consume_grant_for_posttool` docstring at `hooks/lib/allowlist.py:240-241` (`Branch 3 ... exact-only — prevents /allow Write consuming TodoWrite`). Tightening PreTool to exact-only aligns the two paths and tightens the permission boundary (does not loosen any current grant).
- **Gap**: One policy mismatch; tightening PreTool removes one branch of the OR (`pattern in cand`), preserving the equality branch.

**H — measurement and gap**:

- **Observed**: During cycle 20260519-151734, orchestrator dispatched TodoWrite (Step 2 → in_progress) and Agent (dev subagent) in the same response; on the next TodoWrite (Step 2 → completed) the pretool-todo-validate.py Gate 4 blocked with `Cannot complete Step 2 ... requires a subagent call (dev) that has not been made`. Manual bookmark patch was required to proceed.
- **Measured (race)**: `hooks/posttool-subagent-track.py:340-358` (`_resolve_context`): reads `workflow-<sid>.json`, locates in_progress canonical step via `_current_in_progress_index(state)` (lines 114-125), bails if `_step_has_subagent_call(canonical, ip_index)` is False (line 357-358). When TodoWrite + Agent dispatch in a single orchestrator response, `posttool-todo-sequence.py:170-173` (`save_state`) write of `last_todos` and `posttool-subagent-track.py:384` (`_record_subagent_call_legacy`) write of `subagent_calls` are ordered by tool execution order, not by hook priority — the Agent posthook fires against a bookmark whose `last_todos` has NOT yet been updated.
- **Expected**: When PostToolUse:Agent fires, the tracker should identify the target canonical step by the **dispatched subagent_type** (deterministic, available in `stdin.tool_input.subagent_type` — confirmed by `pretool-subagent-enforce.py:128`), not by inference from the bookmark's in_progress flag (race-prone). Source: design contract — Gate 4 strictness must be preserved, which means the tracker must record subagent_calls deterministically with respect to *what was dispatched*, not *what the bookmark currently says is in progress*.
- **Gap**: Single point of inference (`_current_in_progress_index`) is replaced by a deterministic canonical-step lookup keyed on the dispatched subagent_type with **bounded-window first-unflipped-match semantics (option (b), K=3)**: when `subagent_type` is non-empty, the tracker scans `canonical[max(0, ip_index):ip_index+3]` for the earliest unflipped step whose `subagent_call.subagent_type` matches; if found, flip; if NOT found, exit 0 with NO write (no fall-through to legacy in_progress inference). When `subagent_type` is ABSENT, the legacy chain runs as a separate Case B (also exits 0 if it resolves to None). No None-key writes anywhere.

## Edge Cases & Risks

- **D risk — call-site regression**: tightening PreTool match could break a grant pattern that intentionally relied on substring. Audit (above) confirms all 6 production call sites pass EXACT canonical tool names (or, for orchestrator-gate.py:184, a canonical tool_name read from stdin); no caller relies on substring. Risk: LOW.
- **D risk — settings.json `allow` confusion**: settings.json uses Claude Code's `Bash(...)` permission grammar, NOT `/allow <pattern>` runtime grants. These are different systems; the fix does not touch settings.json. Risk: N/A.
- **H risk — option (c) over-flip**: option (c) in the dispatch prompt would flip ALL canonical steps with matching subagent_type on a single dispatch. If a `/command` has TWO `dev` dispatch steps (e.g., parallel dev followed by serial dev), option (c) would let one Agent call satisfy both and the validator would never detect the missing second dispatch. **The spec rejects option (c) and adopts option (b)**: first-unflipped-match semantics — the tracker flips the *earliest* canonical step with matching subagent_type that has not yet been flipped. This preserves call-count semantics.
- **H risk — out-of-order / stale-bookmark / session-restart (OBJ-3/OBJ-5 resolution path (a))**: a global first-unflipped-by-type search could mark a wrong step under stale state. Example: canonical `[qa-1, dev, qa-2]`; on session restart with `qa-1` already-completed-but-not-flipped (stale state), a new `Agent(subagent_type="qa")` dispatch intended for `qa-2` would otherwise be applied to `qa-1`. **Mitigation (adopted)**: Case A typed lookup is gated on a single valid in_progress anchor; if there is no in_progress step or more than one in_progress step, the helper returns None and the hook exits 0 with no write. With a single valid anchor, the helper scans only `canonical[max(0, ip_index):ip_index+3]` (K=3 — covers the current in_progress step plus the next 2 candidates, sized to accommodate the common parallel-dispatch race AND a one-step-ahead Agent emission, but small enough to prevent reaching back to steps that should already be considered complete). Rationale for K=3: typical /dev pipeline emits TodoWrite (Step N → in_progress) and Agent (matching Step N) in the same response — that requires the window to include `ip_index` itself (offset 0) plus a defense margin of 2 for cases where the Agent emission slightly precedes the TodoWrite update (orchestrators that batch Agent before TodoWrite). K>3 starts admitting stale-state misflips; K<3 risks missing legitimate parallel dispatches. **No-anchor, multi-anchor, or outside-window cases**: the tracker exits 0 with no write (per Case A miss handling in Technical Hints H step 1).
- **H risk — empty subagent_type**: if `stdin.tool_input.subagent_type` is empty or missing (defensive parse), the tracker falls back to the existing legacy logic via Case B only (`_current_in_progress_index` + `_step_has_subagent_call` gate). Case B is unreachable from Case A — there is NO fall-through when `subagent_type` is non-empty. Existing legacy behavior is preserved ONLY for the absent-subagent_type case. Risk: LOW.
- **H risk — cross-type Agent dispatch (OBJ-2 resolution path)**: when `subagent_type` is non-empty (e.g., `"ba"`) but the bookmark's `in_progress` step expects a different type (e.g., `"qa"`), the typed lookup at Case A finds no match within the bounded window. The hook exits 0 with no write rather than flipping the unrelated `in_progress` step. This preserves Gate 4 strictness for cross-type dispatches and prevents the silent-weakening scenario that motivated OBJ-2 sub-defect (ii).
- **H risk — None-key write hazard (OBJ-2 resolution path)**: an unchecked None resolution at any branch of the chain could otherwise write `subagent_calls["None"]=True`, corrupting bookmark shape. **Mitigation (adopted)**: Technical Hints H step 1 mandates an explicit `target is not None and isinstance(target, int)` guard before `_record_subagent_call_legacy` is called. No None-stringification at any layer.
- **H risk — contract path (Path A) interaction**: Path A (overnight-cycle contract runtime) uses its own atomic reconciliation (`contract_runtime.reconcile_accepted_artifact`) and does NOT rely on bookmark `in_progress` inference for the index. The fix only changes Path B (legacy `/dev` / `/redev`). Risk: LOW.
- **H risk — orphan Agent**: when no canonical step matches the dispatched subagent_type within the bounded window, no single in_progress anchor exists for non-empty `subagent_type`, multiple in_progress anchors exist for non-empty `subagent_type`, or `subagent_type` is absent and the legacy chain returns None, the tracker performs no write. AC-H3b covers all no-write sub-cases; AC-H3a separately preserves the valid legacy-write path for absent `subagent_type`. Risk: LOW.
- **H risk — multiple Agent dispatches in one response**: each PostToolUse:Agent fires independently with its own subagent_type. Each finds the next unflipped matching canonical step within its bounded window. Correct multi-dispatch semantics preserved. Risk: LOW.
- **H risk — Case A single-anchor gating (OBJ-5 BLOCKER resolution)**: Case A is gated on a single valid in_progress anchor. No-anchor (`ip_index is None`) or multi-anchor bookmark states (multi-in_progress, returned as None by the multi-anchor guard in `_current_in_progress_index`) MUST NOT open the typed-lookup window to `canonical[0:3]`. Both states fall through to AC-H3b no-write. This closes the OBJ-5 stale-state misflip class (session-restart with stale subagent_calls, or inconsistent bookmark shape) that the prior iter1 `else: start = 0` fallback would have admitted. Risk: LOW (after guard).

### AC-H3b: Orphan Agent does not corrupt bookmark (all no-write cases)

- WHEN PostToolUse:Agent fires
- THEN the hook completes with exit 0 (no exception escapes)
- AND `subagent_calls` is left UNCHANGED in shape (no keys added, no keys removed, no values flipped)
