## Goal

Identify every hook in the PreToolUse chain that blocks `/tmp/xlsx-venv/bin/python /root/orchestra/docs/dev/find_red_cells.py > /root/orchestra/docs/research/case-3` despite a valid `/allow` sentinel grant, and fix the gaps so that a user-issued `/allow` for this command type is correctly honored.

## Context

The `/allow` command writes two grant artifacts: (1) a legacy pattern-string grant at `/tmp/claude-bash-allowlist-<sid>.json`, and (2) a structured sentinel grant at `/tmp/claude-grants/<task_id>.json`. Despite a valid sentinel grant existing, the command was blocked.

**OBJ-1 resolution — blocking hook identified**: The primary blocker is `pretool-tool-policy.py`, which fires for ALL tools (matcher `*` in settings.json lines 449-454). For a Bash command with `> /root/orchestra/docs/research/case-3` as redirect target, it calls `extract_bash_write_paths()` which extracts that path as a Write target, then checks `is_allowed(role, "Write", "/root/orchestra/docs/research/case-3")` against `policies/tool-policy.v1.json`. For non-dev subagent roles (`ba`, `general-purpose`, etc.), this check fails because `/root/orchestra/` is not in their `allowed_write_path_prefixes`. Codex runtime execution confirmed: exit=2 (BLOCKED) for `ba` role subagent, exit=0 for main agent (no agent_id → `resolve_agent_type` returns None → `sys.exit(0)` at lines 95-101), exit=0 for `dev` role.

**OBJ-2 resolution — pretool-tool-policy.py fully investigated**: Confirmed at `hooks/pretool-tool-policy.py:47-92`: `extract_bash_write_paths()` extracts redirect targets from Bash commands. `is_allowed(role, "Write", path)` checks `policies/tool-policy.v1.json` per-role allowed_write_path_prefixes. `/root/orchestra/` is absent from all non-dev roles.

**OBJ-3 resolution — IS_SUBAGENT promoted to Must Have**: No evidence exists that the command was issued in main-agent context. The command `find_red_cells.py > docs/research/case-3` is typical subagent data-analysis work. Absent session logs proving main-agent context, IS_SUBAGENT handling is Must Have.

**OBJ-4 resolution — Gap C (args_contain) diagnosis corrected**: For bare-token `/allow /tmp/xlsx-venv/bin/python /root/orchestra/docs/dev/find_red_cells.py`, `userprompt-consent-allowlist.sh:86-97` parser takes ONLY the first bare token (`/tmp/xlsx-venv/bin/python`) as PATTERN; subsequent tokens (`/root/orchestra/docs/dev/find_red_cells.py`) become COMMENT via `comment = " ".join(bare[1:])` at line 97. `_looks_regex('/tmp/xlsx-venv/bin/python')` = False (no dots, no metacharacters). Sentinel written as `{'op': '/tmp/xlsx-venv/bin/python'}` with NO `args_contain`. Since `args_contain=[]`, `all()` over empty = True, so the match succeeds for any command starting with the op token. The redirect token contamination diagnosis in the prior spec was WRONG for bare-token /allow syntax.

**OBJ-5 resolution — R4 removed**: `hooks/lib/allowlist.py:470-472` uses pure string equality `want_op != head_op`. `/tmp/xlsx-venv/bin/python == /tmp/xlsx-venv/bin/python` → True. allowlist.py works correctly for /tmp paths; it is not a bug location.

**OBJ-7 resolution — task ID derivation mismatch identified**: Writer (`userprompt-consent-allowlist.sh:176`) uses `TASK_ID="${CLAUDE_TASK_ID:-${SID}}"` where SID = UserPromptSubmit session_id. Reader (`pretool-bash-safety.sh:484-488`) uses `CLAUDE_TASK_ID`, then falls back to `d.get('task_id','') or d.get('session_id','') or os.environ.get('CLAUDE_SESSION_ID','default')`. If `CLAUDE_TASK_ID` is unset, writer uses UserPromptSubmit `session_id`; reader falls back to PreToolUse JSON `task_id` field if present — which may differ from session_id. Sentinel written under one key, read under another → `SENTINEL_NONE`.
