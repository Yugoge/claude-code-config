# BA Specification: /allow Grant Not Honored for /tmp Venv Interpreter + Redirect Commands

**Request ID**: 20260521-090200
**Created**: 2026-05-21T09:02:00Z
**Tier**: STANDARD

## Goal

Fix the hook system so that a user-issued `/allow` grant is correctly honored for commands whose executable path is under `/tmp` (e.g., a venv Python interpreter), running a script, with stdout redirected to a file. Three distinct gaps cause the grant to be silently discarded or its legacy fallback suppressed.

## Context

The `/allow` command writes two grant artifacts: (1) a legacy pattern-string grant at `/tmp/claude-bash-allowlist-<sid>.json`, and (2) a structured sentinel grant at `/tmp/claude-grants/<task_id>.json`. The `pretool-bash-safety.sh` hook checks both at PreToolUse time. The user reported that despite a valid sentinel grant existing, the command `/tmp/xlsx-venv/bin/python /root/orchestra/docs/dev/find_red_cells.py > /root/orchestra/docs/research/case-3` was still blocked.

Investigation identifies three independent gaps. The Codex adversarial review confirmed all three.

## Evidence (Contract A)

- **Observed**: "the /allow sentinel grant existed but execution was still blocked" for `/tmp/xlsx-venv/bin/python /root/orchestra/docs/dev/find_red_cells.py > /root/orchestra/docs/research/case-3`
- **Measured**: `IS_SUBAGENT = "1"` check at `hooks/pretool-bash-safety.sh:145` causes `check_and_consume_allowlist()` to unconditionally `return 1`, bypassing both grant paths for subagent contexts. Sentinel block is additionally wrapped at `hooks/pretool-bash-safety.sh:483`. CF-1 at `hooks/pretool-bash-safety.sh:525` sets `SENTINEL_EXISTS_FOR_TASK=1` on `SENTINEL_EXISTS_NO_MATCH`, which suppresses legacy path at line 551. Sentinel construction in `hooks/userprompt-consent-allowlist.sh:197-205` puts the full remainder of the pattern (including redirect `> output`) into `args_contain[0]` as one string, but the hook's token-based `rest = " ".join(tokens[1:])` may not match if the redirect is shell-consumed before the hook sees the command.
- **Expected**: A user-granted `/allow` for a specific command covering a `/tmp` venv interpreter should be honored — the command should reach the PreToolUse allow decision and exit 0.
- **Gap**: Three gaps combine to prevent this: (A) IS_SUBAGENT firewall blocks all grant paths in subagent context; (B) CF-1 suppresses legacy fallback when a regex/wildcard sentinel exists but does not match (specifically: `op: "*"` sentinels written for regex grants never structurally match, so CF-1 always fires for regex grants, permanently suppressing legacy); (C) sentinel `args_contain` construction is redirect-unaware — the full `> /output-path` string is stored as a single `args_contain` fragment, which breaks if the hook payload normalizes or omits the redirect.

## Requirements (MoSCoW)

### Must Have
- M1: When a user grants `/allow` for a command whose first token is a `/tmp/` path (e.g., `/tmp/xlsx-venv/bin/python`), the sentinel grant match must succeed and the command must be allowed (exit 0 from `pretool-bash-safety.sh`).
- M2: When a `/allow` pattern contains a shell redirect (`> /output`), the sentinel `args_contain` must NOT include the redirect token in its fragment check — only pre-redirect arguments should be matched.
- M3: When a regex or bare wildcard `/allow` is issued (pattern `.*` or pattern starting with `re:`), the corresponding sentinel must NOT suppress the legacy `/allow` path via CF-1. Either: (a) do not write a sentinel for such grants, or (b) mark the sentinel with `legacy_fallback_allowed: true` so CF-1 can exempt it.

### Should Have
- S1: Add `Bash(/tmp/*-venv/bin/python*:*)` to `settings.json` allow list as a structural secondary fix for the common venv case, so these commands require neither `/allow` nor hook bypass for typical data-analysis workflows.

### Could Have
- C1: Add structured sentinel grant support for subagent contexts (when `IS_SUBAGENT=1`) — allow user-created sentinel grants whose `session_id` matches the active session to be honored even in subagent Bash calls. This preserves the subagent firewall against grant creation while permitting user-authorized pass-through.

### Won't Have (Non-Goals)
- W1: No changes to daemon-restart prohibition layers (1.A–1.E).
- W2: No weakening of the IS_SUBAGENT firewall for legacy pattern-string grants.
- W3: No changes to PostToolUse sentinel consumption logic.
- W4: No changes to the `userprompt-consent-allowlist.sh` subagent write firewall.

## Edge Cases & Risks

- **CF-1 re-opened bypass risk**: relaxing CF-1 for `legacy_fallback_allowed` sentinels must be scoped narrowly — only sentinels that explicitly declare the flag, not all sentinels. Without this scoping, the comment-only attack (AC2 invariant) is reopened.
- **Redirect token normalization**: Claude Code may or may not include the `> file` redirect in `tool_input.command`. The fix must be forward-compatible with both behaviors (redirect present vs. absent in hook payload).
- **op: "*" wildcard sentinel**: Codex confirmed this never structurally matches (matcher requires `op == head_op` which can never equal `"*"` for a real command). So CF-1 fires for all regex grants even when the legacy pattern would have matched. The fix must prevent sentinel write for regex/wildcard grants OR add `legacy_fallback_allowed: true`.
- **settings.json deny precedence**: Per D3b, a project-local deny rule in settings.json can still block even after `/allow` emits an "allow" permissionDecision. Adding the `/tmp/*-venv/bin/python*` allow entry in the global settings.json does not help if a project-level settings.json has a conflicting deny.

## Acceptance Criteria

### AC1: Sentinel grant match for /tmp interpreter
- GIVEN a user issues `/allow /tmp/xlsx-venv/bin/python /root/orchestra/docs/dev/find_red_cells.py` (no redirect in pattern)
- WHEN the sentinel is written and the command `/tmp/xlsx-venv/bin/python /root/orchestra/docs/dev/find_red_cells.py > /root/orchestra/docs/research/case-3` is executed
- THEN `match_sentinel_grant_for_bash_command` returns a non-None entry (op matches, args_contain check passes or is absent)

### AC2: Redirect tokens excluded from args_contain
- GIVEN a user issues `/allow /tmp/xlsx-venv/bin/python /root/script.py > /root/output`
- WHEN the sentinel is written via `userprompt-consent-allowlist.sh`
- THEN the `allowed_operations[0]` entry does NOT contain `> /root/output` in `args_contain`; the redirect is stripped before constructing `args_contain`

### AC3: CF-1 does not suppress legacy for regex/wildcard grants
- GIVEN a user issues bare `/allow` (wildcard) or `/allow re:.*python.*`
- WHEN the sentinel is written (op: "*" or a regex entry)
- THEN EITHER: (a) no sentinel file is created for the regex/wildcard grant, so CF-1 never fires and legacy path proceeds; OR (b) sentinel is created with `legacy_fallback_allowed: true` and `pretool-bash-safety.sh` exempts it from SENTINEL_EXISTS_FOR_TASK=1 assignment
- AND the legacy `check_and_consume_allowlist` path IS consulted for the intended Bash command

### AC4: IS_SUBAGENT firewall preserved for legacy grants
- GIVEN a subagent (agent_id present) attempts to execute a Bash command
- WHEN `check_and_consume_allowlist` is called
- THEN the legacy path still returns 1 immediately due to IS_SUBAGENT=1 (no change to this path)

### AC5 (Should Have): settings.json entry for common venv pattern
- GIVEN `settings.json` is updated with `"Bash(/tmp/*-venv/bin/python*:*)"` in the allow list
- WHEN a command starting with any `/tmp/*-venv/bin/python` path is attempted
- THEN Claude Code's built-in permission check allows it without requiring `/allow`
