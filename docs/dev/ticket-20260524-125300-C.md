# BA Specification: Spec-Continuation Marker Missing from Producer

**Request ID**: 20260524-125300-C
**Created**: 2026-05-24T12:53:00Z
**Tier**: MICRO

## Goal

`commands/spec-update.md` continuation-spec mode never writes the machine-readable
`<!-- spec-continuation-of: <task-id> -->` marker that `commands/commit.md` Step 7
Stage 3 requires for candidate matching. Until the producer is fixed, Stage 3 always
finds zero candidates and silently exits 0, making spec-continuation inert for any
cycle that relies on Stage 3 discovery.

## Context

Cycle 20260522-080646 item C tightened the Step 7 Stage 3 grep predicate in
`commands/commit.md` (line 164) to require the exact HTML comment marker
`<!-- spec-continuation-of: ${TASK_ID} -->` inside each spec file, replacing the
broader literal task-id grep. The consumer side (commit.md) is now correct.
However, no corresponding change was made to the producer.

`commands/spec-update.md` continuation-spec mode (lines 50-81) instructs the agent
to determine the next Cycle N number and append a cycle block — but contains no
instruction to write the marker. Result: every spec produced by spec-update lacks
the marker; Stage 3 returns an empty candidate set; the fail-closed outcome prints
`No spec associated with task-id ${TASK_ID}` and exits 0 silently.

This was documented as close-report W3 of cycle 20260522-080646 ("Graceful
degradation by design") but is now the blocker for all future cycles.

## Requirements (MoSCoW)

### Must Have

- `commands/spec-update.md` continuation-spec mode instructs the agent to write `<!-- spec-continuation-of: <task-id> -->` into the target spec file, with the actual task-id value substituted
- The marker is written exactly once per (task-id, spec file) pair — idempotent: if the marker for that task-id is already present, do not write a second copy

- The marker is written only in continuation-spec mode, not in `--temp` mode

- The instruction is explicit that the agent must substitute the resolved task-id value and must NOT write a literal placeholder (`${TASK_ID}`, `<task-id>`, etc.)

## Edge Cases & Risks

- **Multi-task specs**: a long-lived spec updated across multiple task-ids could accumulate multiple markers (one per task-id) — this is correct behaviour; Stage 3 greps for the current task-id literally, so only the matching marker passes
- **Idempotent re-runs**: if spec-update is invoked twice for the same task-id on the same spec, the second run must not add a duplicate marker — the instruction must check for presence before writing
- **Literal placeholder risk**: if the agent writes `<!-- spec-continuation-of: ${TASK_ID} -->` as a literal string, the grep for the resolved task-id will never match — the instruction must be explicit about substitution
- **Temp mode isolation**: `--temp` mode does not produce a file in `docs/dev/specs/`; no marker should appear in temp notes

## Acceptance Criteria

### AC1: Continuation run writes marker with resolved task-id

- GIVEN `commands/spec-update.md` continuation-spec mode is invoked for an active task-id `T`
- WHEN the agent appends a new cycle block to a spec file at `docs/dev/specs/spec-*.md`
- THEN the spec file contains the literal string `<!-- spec-continuation-of: T -->`  where `T` is the resolved task-id value (not a placeholder)

**Verification**: `grep -F "<!-- spec-continuation-of: T -->" <spec_path>` exits 0 and prints the matching line.

### AC2: Stage 3 content predicate passes for marker-bearing spec

- GIVEN a spec file at `docs/dev/specs/spec-YYYYMMDD-HHMMSS.md` produced by a spec-update continuation run for task-id `T`
- WHEN Stage 3 runs `grep -lF "<!-- spec-continuation-of: T -->" <spec_path>`
- THEN grep exits 0 and returns the spec path as a candidate

**Verification**: `grep -lF "<!-- spec-continuation-of: T -->" <spec_path>` exits 0. (Basename pattern and mtime window are enforced by commit.md separately; this AC tests only the content predicate.)

### AC3: No literal placeholder in written marker

- GIVEN `commands/spec-update.md` instructs the agent to write the marker
- WHEN the agent produces the marker in the spec file
- THEN the spec file does NOT contain `<!-- spec-continuation-of: ${TASK_ID} -->` (literal) or `<!-- spec-continuation-of: <task-id> -->` (literal angle-bracket form)

**Verification**: `grep -F "spec-continuation-of: \${TASK_ID}" <spec_path>` exits 1 (no match). `grep -F "spec-continuation-of: <task-id>" <spec_path>` exits 1.

### AC4: Marker instruction lives inside continuation-spec mode only

- GIVEN `commands/spec-update.md` after the fix
- WHEN a static grep checks mode placement
- THEN the file contains `spec-continuation-of` text inside the `## Continuation-spec mode` section AND the `## Temp-note mode` section contains no `spec-continuation-of` text

**Verification**: `grep -n "spec-continuation-of" commands/spec-update.md` returns lines that fall between the `## Continuation-spec mode` heading and the `## Temp-note mode` heading only.

### AC5: Marker is idempotent (no duplicate for same task-id)

- GIVEN a spec file already containing `<!-- spec-continuation-of: T -->` for task-id `T`
- WHEN spec-update continuation mode is invoked again for the same task-id `T` on the same spec
- THEN the spec file contains exactly one occurrence of `<!-- spec-continuation-of: T -->`

**Verification**: `grep -c "spec-continuation-of: T" <spec_path>` returns `1`.

## Technical Hints

- Affected file: `commands/spec-update.md` only — lines 50-81 (continuation-spec mode cycle-block appending instructions)
- The marker must be added to the instructions in `## Continuation-spec mode` before the prose that lists the 8 sections to populate
- Placement guidance for the agent executing spec-update: check for existing `<!-- spec-continuation-of: <active-task-id> -->` first; if absent, write it immediately before the first new `### Cycle N` subsection being appended
- The marker is an HTML comment; it renders as invisible in Markdown preview and is harmless to spec readability
- `grep -lF` treats `<`, `>`, and `--` as literals; no shell escaping is needed beyond double-quoting the argument
