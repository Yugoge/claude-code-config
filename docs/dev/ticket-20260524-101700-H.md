# BA Specification: Three Structural Gaps in happy-session-recovery.sh

**Request ID**: 20260524-101700-H
**Created**: 2026-05-24T10:17:00Z
**Tier**: STANDARD

## Goal

Fix three verified structural gaps in `/root/bin/happy-session-recovery.sh` that leave the session recovery system vulnerable to silent data loss in specific timing windows:

1. Manually killed/archived sessions can be resurrected after a daemon crash within the 5-minute tombstone observation window (HIGH).
2. Daemon-tracked recovered sessions are not immediately persisted to `session_dirs.txt`, leaving a 60-second window where a daemon crash loses them (MEDIUM).
3. Per-session memory limits default to `infinity`, providing no OOM protection for sessions that run amok (MEDIUM).

## Context

The script is the single source of truth for session lifecycle. It runs in three modes: `watch` (continuous, run by `happy-session-watcher.service`), `restore` (run by `happy-daemon-post-start.sh` after each daemon restart), and `save` (run by daemon `ExecStartPre`). The tombstone mechanism prevents resurrected sessions from re-appearing after user-intentional kills, but currently has a timing gap: the watcher must observe the loss as stable for 5 full minutes before writing a tombstone. If the daemon crashes within those 5 minutes, the next restore sees the UUID in `session_dirs.txt` and resurrects it.

## Evidence (Contract A)

### GAP 1 — Tombstone timing window

- **Observed**: "If the daemon crashes within those 5 minutes, no tombstone exists and the manually-killed session gets resurrected on next boot."
- **Measured**: `record_stable_loss_tombstones()` is called only at line 792 inside `watch_sessions()`, guarded by `loss_stable_count >= loss_required` where `loss_required=30` (line 749) and each tick is 10s. The stable-loss window is 300 seconds. The `record_stable_loss_tombstones()` function at lines 359–386 compares against `session_dirs.txt` and appends to `session_tombstones.jsonl`. There is no code path that writes a tombstone at kill-time (signal-time, archive-time) — it is purely observer-driven.
- **Expected**: A tombstone should be written immediately at the moment the watcher first detects a loss (transition from "session present" to "session absent"), rather than only after the 5-minute stability window. The stability window would remain for accepting the count drop into `session_dirs.txt`, but tombstone protection should apply from first detection.
- **Gap**: Tombstone lags by up to 300 seconds from actual kill event. Any daemon crash within those 300 seconds causes the killed session to resurrect.

### GAP 2 — Webhook-to-session_dirs.txt delay

- **Observed**: "After a recovered session passes wait_for_daemon_tracking() the watcher has not yet written it to session_dirs.txt."
- **Measured**: In `daemon_spawn_session()` (lines 1465-1478), after `wait_for_daemon_tracking()` returns 0 (success), the function returns 0 to `restore_online_sessions()`. At line 1760-1762, the caller only increments `total_restored` and sets `globally_spawned[$uuid]=1`. There is no call to `write_home_snapshot()` or `write_snapshot()` after a successful daemon tracking confirmation. The next `session_dirs.txt` update happens at the next watcher poll cycle (line 770, sleeping `POLL_INTERVAL=60s`).
- **Expected**: Immediately after `daemon_spawn_session()` confirms daemon tracking for a session (the daemon is now live-tracking it), that session should be durably written to `session_dirs.txt` so a crash in the next 60 seconds does not lose it.
- **Gap**: 60-second window (maximum `POLL_INTERVAL`) where a confirmed-tracked session is absent from `session_dirs.txt`.

### GAP 3 — MemoryMax defaults are infinity

- **Observed**: "Current defaults: HAPPY_RECOVERY_MEMORY_MAX=infinity ... OOM protection is never enforced by default."
- **Measured**: Lines 1418-1420: `local recovery_memory_max="${HAPPY_RECOVERY_MEMORY_MAX:-infinity}"`, `local recovery_memory_high="${HAPPY_RECOVERY_MEMORY_HIGH:-infinity}"`, `local recovery_memory_swap_max="${HAPPY_RECOVERY_MEMORY_SWAP_MAX:-infinity}"`. Comment at line 1415-1416 states "Defaults intentionally do NOT impose tiny hard memory caps: this installation legitimately runs Happy workloads above 60G total." The actual server has 32GB RAM + 40GB swap (verified: `MemTotal: 32088160 kB`, `SwapTotal: 41943032 kB`).
- **Expected**: Per-session defaults should be reasonable upper bounds that protect against runaway sessions while permitting legitimate large workloads when explicitly configured. The 60GB justification in the comment is inaccurate for per-session limits (whole-server aggregate vs. per-session cap). Reasonable defaults: `MemoryMax=4G`, `MemoryHigh=3G`, `MemorySwapMax=2G`. Operators needing more set `HAPPY_RECOVERY_MEMORY_MAX` env var.
- **Gap**: No OOM protection on any recovery-spawned session. A single runaway session can exhaust all 32GB RAM + 40GB swap before the kernel OOM killer intervenes.

## Edge Cases & Risks

1. **GAP 1 — Premature tombstone on transient daemon blip**: If the daemon briefly glitches and a session disappears from `/list` for one poll cycle, the immediate tombstone would prevent future resurrection. Mitigation: The tombstone is only written when the session disappears from `scan_running_sessions()` (process-level scan) — not just from `/list`. Process disappearance is a stronger signal than webhook absence. Review the `record_stable_loss_tombstones()` function to verify it uses the process-level scan output (it does, via `current_raw` from `scan_running_sessions()`). This makes premature tombstones unlikely for legitimate blips.

2. **GAP 1 — Tombstone written twice for the same UUID**: If the watcher writes an early tombstone at first detection and again at the 5-minute mark (via `record_stable_loss_tombstones()`), the tombstone file will have duplicate entries. `is_tombstoned()` uses `sys.exit(0)` on first match, so duplicates are harmless — but dev should confirm the 5-minute call still works correctly after the early write (it compares against `session_dirs.txt` which has not been updated yet, so it would add another row for the same UUID — still harmless but noisy). Consider skipping the 5-minute tombstone call for UUIDs already tombstoned.

3. **GAP 2 — write_home_snapshot() clobbers an in-progress loss observation**: The immediate snapshot after recovery could write a higher session count into `session_dirs.txt`, which `update_snapshot_with_peak_protection()` would normally guard against during runtime. Since the recovery is writing from the restore path (not the watcher path), this is acceptable — the restored sessions should appear in `session_dirs.txt`. Dev must write only the sessions belonging to the home being restored, not a cross-home snapshot.

4. **GAP 2 — Concurrent watcher write race**: The watcher also calls `write_snapshot()` on every poll. If the restore path and the watcher path write simultaneously, `write_home_snapshot()` uses atomic tmp+mv so the last writer wins — neither is corrupted. Acceptable.

5. **GAP 3 — 4G limit may be too low for Codex sessions**: Codex sessions can use substantial memory. The env var override mechanism (`HAPPY_RECOVERY_MEMORY_MAX`) must be documented in the script comment. The default of 4G is a soft guard that can be raised. If a session regularly exceeds 4G, operators should increase the limit.

## Acceptance Criteria

### AC1: Early tombstone on first loss detection
- GIVEN the watcher is running and observes that a session UUID was present in the previous scan but is absent in the current scan
- WHEN `loss_observe_mode` transitions from 0 to 1 for that UUID
- THEN a tombstone entry is written to `session_tombstones.jsonl` for that UUID immediately (before the next sleep cycle), with `reason: "early_loss_detected"` (or equivalent distinguishing field)
- AND the tombstone is written even if the daemon crashes immediately after that detection

### AC2: Tombstone prevents resurrection after early detection
- GIVEN a session's UUID has an early tombstone written (AC1)
- WHEN `restore_online_sessions()` runs and finds that UUID in `session_dirs.txt`
- THEN `is_tombstoned()` returns true and the session is skipped (not spawned)

### AC3: session_dirs.txt updated after tracking confirmation
- GIVEN `restore_online_sessions()` has successfully spawned a session and `wait_for_daemon_tracking()` returned 0
- WHEN `daemon_spawn_session()` returns success to the caller
- THEN `write_home_snapshot()` (or equivalent) is called for the session's home with a session list that includes the newly-confirmed session
- AND the write uses the atomic tmp+mv pattern to avoid partial writes

### AC4: Recovered session survives immediate daemon crash
- GIVEN a session was recovered and passed AC3 (session_dirs.txt updated)
- WHEN the watcher is checked after the write (before the next 60-second poll)
- THEN the session UUID is present in `session_dirs.txt` for its home

### AC5: Finite per-session memory limits
- GIVEN `daemon_spawn_session()` is called without any `HAPPY_RECOVERY_MEMORY_*` env vars set
- WHEN a transient systemd unit is spawned via `systemd-run`
- THEN the unit is created with `MemoryMax=4G`, `MemoryHigh=3G`, `MemorySwapMax=2G` (not `infinity`)
- AND the log line at ~line 1455 confirms these values

### AC6: Operator override still works
- GIVEN `HAPPY_RECOVERY_MEMORY_MAX=8G` and `HAPPY_RECOVERY_MEMORY_HIGH=6G` are set in the environment
- WHEN `daemon_spawn_session()` is called
- THEN the unit is created with `MemoryMax=8G` and `MemoryHigh=6G` (env var override respected)
