## Edge Cases & Risks

1. **Token session_id collision**: If another session wrote a push-gate token for the same repo/branch, `agents/cleaner.md`'s token write must check session_id and warn+skip (matching changelog-analyst Phase 10 behavior) — dev must include this guard.
2. **Commit failure**: If the cleaner's inline `git commit` fails (nothing staged, or git error), the push-gate token MUST NOT be written. Token write is conditional on `git commit` exit code 0.
3. **Nested repo**: `agents/cleaner.md` commits only in the target project repo (not the dot-claude nested repo). The token write applies to whichever repo was committed. If cleaner commits in both repos, a token per repo is needed — but in practice cleaner operates on a single target project. Dev should write token for the repo that was committed.
4. **Path B fallback**: If `agents/cleaner.md` cannot be edited due to tool-policy restrictions (Trap 3 from spec Section 8), dev must fall back to Path B: change `clean.md:589` to document "Changes already committed — to publish to remote: use `git push` directly (the push-gate token is not available after `/clean`; `/push` requires it)."
