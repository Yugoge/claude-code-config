# BA Specification: Layered prevention of tmp/ramdisk pressure recurrence

## Goal

Implement the codex-debated, user-confirmed (`实施全部`) 4-layer additive plan that prevents
`/tmp` (4G tmpfs) and `/dev/shm` (16G tmpfs) from filling up again, without changing existing
architecture or any subagent runtime behavior. The four layers are: (L1) session-start visibility
banner, (L1.5) mid-session UserPromptSubmit warning with rate-limit, (L2) extended scheduled GC
with EXCLUSION-FIRST ordering, (L3) `*-scratch-<ts>` convention documented.

## Context

Trigger event: `/tmp` reached 85% and `/dev/shm` reached 69% utilization on this Hetzner vServer;
manual cleanup recovered ~2.8G. Real disk `/` was only 27% used — this is a RAM-pressure problem
specific to tmpfs mounts. The existing daily cron `/etc/cron.d/tmp-cleanup-daily` invoking
`/usr/local/sbin/tmp-cleanup.sh` only recovered 28K on 2026-05-19 because its pattern list is too
narrow and it does NOT distinguish hard-exclusion paths from broad wildcards that could match
them. The fix is purely additive; no existing functionality is removed.

## Edge Cases & Risks

- **`-prune` semantic correctness (BLOCKER per codex review)**: a `find` expression of the
  form `find /tmp \( EXCL \) -prune -o BODY` with the action appended at the end (e.g.,
  `-print` or `-exec rm -rf {} +` after `BODY`) will short-circuit the excluded paths AND
  print them too — the prune branch evaluates true and the trailing action applies to both
  branches. The action MUST be inside the right-hand non-prune branch. Correct shape:
  `find /tmp -maxdepth 1 \( -path EXCL1 -o -path EXCL2 -o … -o -path '*debug-profile*' \) -prune
  -o \( <age/type/name predicates> -print \)`. For real delete, EITHER (a) end the right-hand
  branch with `-print0` and pipe to `xargs -0r rm -rf --`, OR (b) put `-exec rm -rf {} +`
  INSIDE the right-hand branch.
- **Pattern conflicts (TWO sites — both must be removed)**: complete grep on
  `/usr/local/sbin/tmp-cleanup.sh` confirms exactly two prune-target literals collide with the
  hard-exclusion set:
  1. `tmp-cleanup.sh:61` — `-o -name 'chrome-debug-profile' \)` in the Playwright-profiles >1d
     block. Once the shared `EXCLUDE_PRUNE` clause is applied, this literal becomes
     unreachable (chrome-debug-profile is pruned first) — dev MUST remove it to avoid dead
     code and reader confusion.
  2. `tmp-cleanup.sh:86` — `-o -name 'happy-attachments' \` in the dev/test scratch >3d block.
     Structurally identical conflict to (1); the shared `EXCLUDE_PRUNE` clause makes this
     literal unreachable, so dev MUST remove it as well.
  Both paths remain ACTIVELY PROTECTED via the shared `EXCLUDE_PRUNE` clause — these are
  prune-target predicate removals only, NOT removals from the protection set.
- **Wildcard `*debug-profile*` must be globally applied**: requirement's
  "Any path matching `*debug-profile*`" is hard-excluded. If this is only added near the
  Playwright-profile predicates, a future renamed debug profile could still be matched by a
  different cleanup tier (e.g., `*-deploy`, `*-app-check`, broad dev/qa patterns). The
  `-path '*debug-profile*'` exclusion MUST appear in the shared hard-exclusion clause used by
  EVERY `find /tmp ...` cleanup category — not only the Playwright group. Implementation must
  NOT add `*-debug-*` as a prune pattern (would match unrelated scratch like `dev-debug-foo`).
- **Session-ID token derivation (M2-token-priority)**: UserPromptSubmit hooks receive the
  session JSON on stdin — that's the authoritative source. Token priority for the counter
  filename: (1) `session_id` from stdin JSON (preferred — guaranteed unique per session),
  (2) `CLAUDE_SESSION_ID` env var, (3) hashed PPID+process-start-time fallback. Do NOT use
  `stat -c %Y /tmp/claude-0` — that mtime is global/mutable and would share counters across
  sessions or reset within a session. Sanitize the token to `[A-Za-z0-9_-]+` before using as
  a filename (length-cap at 64 chars; `sha256sum | cut -c1-32` works).
- **Counter race + leak (flock is the SOLE acceptable lock primitive — OBJ-3 resolution)**:
  simple read/increment/write can race under concurrent UserPromptSubmit invocations and
  exceed 3 warnings. Use `flock -x <fd>` on a dedicated `<counter_file>.lock` file (a
  regular FILE — NOT a directory). **Atomic-`mkdir` locks are FORBIDDEN** because the M3
  hook-state cleanup tier uses `-type f` (preserved from the existing script per AC7), so
  lock DIRECTORIES would never match the `-type f` predicate and would accumulate
  indefinitely in /tmp — defeating the very purpose of the rate-limit. With `flock`, the
  lock file is a regular file matched by `claude-pressure-warn-*` at the >7d hook-state
  tier (M3 pattern extension covers BOTH the counter file `claude-pressure-warn-<token>`
  AND the lock file `claude-pressure-warn-<token>.lock` because the wildcard matches both).
  **Parent-shell short-circuit (OBJ-1 resolution)**: a bare `exit 0` inside the
  `( ... ) 9>lockfile` subshell only exits the subshell — the parent hook keeps running
  and would still emit. The lock-protected critical section MUST be a command-substitution
  subshell that prints its decision (`emit` or `skip`) to stdout, and the parent shell
  branches on that decision (`[ "$decision" = "emit" ] || exit 0`). See Technical Hints
  M2 for the exact shape. This closes the loop: the feature meant to protect /tmp does
  not itself accumulate, and the rate-limit is empirically functional (the parent hook
  observes the "already at limit" signal and skips the warning print).
- **`*-scratch-<ts>` convention must actually be swept (codex finding #7)**: documenting the
  convention but not adding it to the cleanup pattern list would make AC8 misleading. Add
  `-name '*-scratch-[0-9]*'` to the >3d tier (bounded by `[0-9]*` so it matches only
  timestamp-suffixed names). This is the link between Layer 2 and Layer 3.
- **Top-5 largest dirs (`du` bug)**: `du -sh --max-depth=1` is broken on GNU du — `-s`
  conflicts with `--max-depth=1` and emits a warning instead of data. Use
  `timeout 5s du -xh --max-depth=1 "$mount" 2>/dev/null | awk -v m="$mount" '$2 != m' | sort -hr | head -5 || true`
  (no `-s`, exclude the root line by filtering on the path column, `-x` to skip cross-fs
  descents, 5s timeout for non-blocking guarantee).
- **`.output` file path depth (codex finding #3)**: real `.output` files live deeper than
  initially verified — at `/tmp/claude-0/<project>/<session>/tasks/*.output` (depth 6). Dev
  query MUST be `find /tmp/claude-0 -mindepth 2 -type f -name '*.output' -mtime +1` (note
  `-type f` to never remove directories named `*.output`; `-mindepth 2` to skip the
  `/tmp/claude-0` root). Re-verified at 2026-05-19T16:30Z: 26 files older than 1d exist,
  `lsof +D /tmp/claude-0` shows 17 currently-open `.output` files, intersection (open AND
  >1d) is 0 — threshold is empirically safe.
- **find -mtime granularity (corrected per codex F1)**: GNU `find -mtime +1` does NOT mean
  "older than 24h" — `find` rounds mtime to whole days and `+1` means "more than 1 day after
  the daily rounding", i.e., the file's age must exceed 2×24h = ~48h before it matches. A
  25-hour-old file does NOT match `-mtime +1`. This is the user-required exact command
  (`docs/dev/user-requirement-dev-20260519-161035.md` line 50 codified `-mtime +1` and the
  open_older_count=0 verification was taken at this exact threshold) — dev MUST use `-mtime +1`
  verbatim; do not "fix" it to `-mmin +1440` or any equivalent. The empirical safety result
  (lsof intersection = 0) was measured against this exact GNU rounding behavior.
- **PATH-shim validation (OBJ-5)**: the M2 hook (`userprompt-tmpfs-pressure.sh`) MUST
  resolve `df` and `du` via PATH lookup — i.e., the script body uses bare `df` / `du`
  invocations (the shell resolves them against `$PATH` at invocation time). Absolute
  paths (`/usr/bin/df` / `/usr/bin/du`) are REJECTED because they bypass the shim
  layer the QA test driver relies on for synthetic >75% scenarios. This is purely a
  M2 hook concern; M3 cleanup script (cron-invoked, root-owned, OBJ-5 not applicable
  there) is governed by the separate "Cron user" rule below. The QA test driver
  pattern (per OBJ-5): place a temp dir at the front of `$PATH` containing
  executable `df` and `du` shims that emit synthetic Use%/du output for a chosen
  scenario (no real tmpfs fill, no risk of triggering the ENOSPC failure mode codex
  F9 forbade). Each test iteration MUST use a DISTINCT `session_id` so the per-
  session counter resets — reusing one session_id would saturate the counter and
  mask shim-output variation across iterations.
- **Cron user (clarified per codex F5)**: cron runs as root. The narrow rule for the M3
  cron-invoked script is: NEW additions to the script MUST NOT depend on shell aliases,
  shell functions, sourced-rc state, or user-specific environment that wouldn't exist
  under cron's minimal PATH. This does NOT forbid bare invocations of standard system
  commands (`find`, `df`, `numfmt`, `xargs`, `du`) that already resolve under cron's
  PATH on this server — the existing tmp-cleanup.sh uses bare `find` / `df` and
  works fine. Dev may keep that style for M3. **The OBJ-5 bare-`df`/`du` PATH-shim
  requirement applies ONLY to the M2 UserPromptSubmit hook** — M3 is a separate
  cron context with a different threat model (root, no test-shim layer required).
  Earlier "absolute paths only" wording was over-broad; the canonical position is
  "no aliases / no sourced-rc state" for M3, and PATH-resolution for M2.
- **Log volume**: extended pattern list will increase per-run log size in
  `/var/log/tmp-cleanup.log`. Acceptable — log rotation is system-managed.
- **Dry-run output channel decision (codex finding #8 / QA objection #4 / OBJ-2 resolution)**:
  current `tmp-cleanup.sh:28` has unconditional `exec >> "$LOG" 2>&1` which silences ALL
  stdout, including `--dry-run` candidate enumeration. Dev MUST condition this redirect on
  `DRY_RUN=0` (real run: append to log only — current behavior; **dry-run MUST tee to BOTH
  stdout AND log** — `exec > >(tee -a "$LOG") 2>&1` shape). Any other dry-run output shape
  (stdout-only without log, log-only, alternating, dev-choice) is explicitly REJECTED per
  OBJ-2: the spec is now single-canonical on this point — tee to BOTH channels for dry-run.

  post-hoc log record (`/var/log/tmp-cleanup.log`) so that QA, dev, and future maintainers

- **Both-mounts AC3 layout (QA objection #5)**: when both `/tmp` AND `/dev/shm` exceed 75%

- **SessionStart per-mount layout (QA objection #2)**: AC1 emits exactly TWO lines (one per

### AC3: UserPromptSubmit pressure warning + rate limit
- GIVEN a Claude Code session is active with `/tmp` and/or `/dev/shm` >75%
- WHEN the user submits a prompt
- THEN a non-blocking warning is printed showing `df -h` for the two mounts AND a separate
  top-5 largest-directories block **for EACH mount whose usage exceeds 75%** (so: ONE top-5
  block when only one mount is over threshold; TWO top-5 blocks — one labeled `/tmp` and
  one labeled `/dev/shm` — when both mounts are over threshold). Each top-5 block is
  computed via
  `timeout 5s du -xh --max-depth=1 "$mount" 2>/dev/null | awk -v m="$mount" '$2 != m' | sort -hr | head -5`
  (NOT `du -sh --max-depth=1` which conflicts on GNU du). Concrete example shape (both
  mounts over threshold):
  ```
  /dev/sda1  4.0G  3.4G  600M  85% /tmp
  tmpfs       16G   13G  3.0G  82% /dev/shm

  500M  /tmp/claude-0
  200M  /tmp/playwright-artifacts-...
  ...
  --- top-5 dirs under /dev/shm ---

  ...
  ```
- AND the warning does NOT block the prompt from proceeding (the hook exits 0 unconditionally)
- AND across the lifetime of one session, the warning appears AT MOST 3 times (4th and later
  prompts at >75% emit nothing). The session identity is keyed off the **stdin JSON
  `session_id`** field (the same field used by `prompt-workflow.py`), with fallbacks to
  `CLAUDE_SESSION_ID` env var and then a hashed PPID-rooted token. The token must be
  sanitized/hashed before forming the counter filename
- AND counter updates are serialized via `flock` (atomic-`mkdir` is FORBIDDEN per OBJ-3
  because the resulting lock directory would not be swept by the M3 `-type f` hook-state
  predicate) so concurrent prompt submissions cannot exceed 3 warnings
- AND the rate-limit decision is propagated from the lock-protected critical section to
  the parent shell via command substitution. The lock file descriptor (fd 9) MUST be
  opened INSIDE the command substitution via a redirected brace-group
  (`decision=$( { flock -x 9 || { printf skip; exit 0; }; ...; } 9> "$lock_file" )`) —
  the trailing-redirection form `decision=$( ... ) 9> "$lock_file"` is REJECTED (codex
  F1) because fd 9 is not reliably visible to commands inside the command substitution
  in that form. Parent decision branch: `[ "$decision" = "emit" ] || exit 0` — a bare
  `exit 0` inside the subshell does NOT short-circuit the parent and the warning would
  still print (OBJ-1 resolution)
- AND in the saturated-counter branch (`n >= 3`), the counter file's mtime MUST be
  refreshed (e.g., by rewriting the same value: `printf '%s\n' "$n" > "$counter_file"`)
  so the >7d hook-state sweep cannot delete an actively-saturated counter and reset
  the ≤3/session guarantee for long-lived sessions (codex F7 — sessions lasting longer
  than 7 days would otherwise see their counter swept and start emitting from 0 again).
  Refreshing the saturated counter's mtime is NOT slot consumption; it only prevents
  active saturated counters from being garbage-collected
- AND when BOTH mounts are ≤75% no warning is printed regardless of prompt count (the
  rate-limit counter is consumed only on actual emission, NOT on every prompt; a session
  may submit 100 prompts at ≤75% and still receive 3 future warnings when pressure rises)
- AND the hook MUST resolve `df` and `du` via PATH lookup (NOT via absolute paths like
  `/usr/bin/df` or `/usr/bin/du`) — this is REQUIRED for AC3's PATH-shim validation
  approach (per OBJ-5): the QA test driver prepends a temp dir to PATH containing
  executable shims that emulate df/du with synthetic high-usage output; if the hook
  hard-codes absolute paths the shim cannot replace them and the test cannot verify
  the >75% threshold logic without risking real tmpfs fill (which is forbidden per
  codex F9). Concretely: the hook MUST use bare `df`/`du` (resolved against
  `$PATH`) — `/usr/bin/df` / `/usr/bin/du` / `command df` / `command du` are
  REJECTED. The test driver MUST also use a DISTINCT `session_id` for each of the
  4 test iterations (e.g., `ba-qa-iter1`, `ba-qa-iter2`, `ba-qa-iter3`,
  `ba-qa-iter4`) so the counter file is per-iteration — reusing one session_id
  across iterations would saturate the SAME counter from iteration 4 onward and
  mask shim output variation across iterations (the test would conflate "rate-limit
  worked" with "the test driver happened to share state"). The expected behavior
  with distinct session_ids: each iteration starts with a fresh counter at 0, so
  each iteration emits the warning exactly once (warning emitted at every iteration
  — the rate-limit is per-session, and four distinct sessions = four warnings).
  To verify the rate-limit itself, an additional FIFTH test uses the SAME
  session_id for 4 consecutive submissions: iterations 1-3 emit, iteration 4 is

     env var → `printf '%s-%s' "$PPID" "$(stat -c %Y /proc/$PPID 2>/dev/null || echo 0)" |
     sha256sum | cut -c1-16`. Sanitize the token to `[A-Za-z0-9_-]+` (length-cap 64) before
     using as a filename component.
   - Compute usage % for /tmp and /dev/shm by parsing `df --output=pcent /tmp /dev/shm | tail -n +2`.
     **`df` MUST be resolved via PATH lookup (bare `df` invocation — NOT `/usr/bin/df`)

     rate-limit slot — the slot is consumed only on actual emission)
   - Else, atomically read+increment the counter file
     `/tmp/claude-pressure-warn-<sanitized_token>` under a `flock` lock. **The lock-protected
     critical section MUST be a subshell that returns its decision via stdout to the parent
     shell** — `exit 0` inside the `( ... ) 9>lockfile` form only exits the SUBSHELL, leaving
     the parent hook free to keep executing and print the warning anyway (silently breaking
     the rate-limit). The correct shape captures the decision via command substitution and
     branches in the parent. **fd 9 MUST be opened INSIDE the command substitution via a
     redirected brace-group (per iteration-2 codex F1)** — a trailing `9> "$lock_file"`
     AFTER `$( ... )` does not reliably attach fd 9 to commands running inside the
     command substitution, so `flock -x 9` can fail with `Bad file descriptor` and leave
     the critical section unsynchronized. The correct shape:
     ```bash
     # counter_file=/tmp/claude-pressure-warn-<sanitized_token>
     # lock_file="${counter_file}.lock"
     decision=$(
       {
         flock -x 9 || { printf 'skip'; exit 0; }
         n=$(cat "$counter_file" 2>/dev/null || echo 0)
         if [ "$n" -ge 3 ]; then
           # F7: refresh mtime on the saturated counter so the >7d hook-state sweep
           # cannot delete an actively-saturated counter mid-session and reset the
           # ≤3/session guarantee. Rewriting the same value updates mtime without
           # changing the emit-count semantics.
           printf '%s\n' "$n" > "$counter_file"
           printf 'skip'
         else
           echo $((n + 1)) > "$counter_file"
           printf 'emit'
         fi
       } 9> "$lock_file"
     )
     [ "$decision" = "emit" ] || exit 0
     # parent shell continues only when decision==emit; prints df + top-5 below
     ```
     This pattern is mandatory: the rate-limit check MUST return a status usable by the
     parent shell (the parent's `[ "$decision" = "emit" ] || exit 0` is what actually
     short-circuits the warning emission). A bare `exit 0` inside the subshell does NOT
     short-circuit the parent. The redirected brace-group `{ ... } 9> "$lock_file"`
     inside `$( ... )` is the CANONICAL shape — equivalent shapes (e.g., a redirected
     subshell `( ... ) 9>"$lock_file"` placed INSIDE `$(...)`) are acceptable as long
     as fd 9 is opened in the same shell scope where `flock -x 9` runs. The trailing-
     redirection form `decision=$( ... ) 9> "$lock_file"` is REJECTED (codex F1). The
     flock-failure short-circuit (`flock -x 9 || { printf 'skip'; exit 0; }`) returns
     `skip` to the parent on any lock-acquisition failure (e.g. ENOSPC during lock
     file create), so the warning is suppressed rather than emitted unsynchronized.
     **`flock` is the ONLY acceptable lock primitive** for this hook (atomic-`mkdir`
     alternative has been retracted per OBJ-3 — see Edge Cases & Risks below; lock
     directories would not match the `-type f` predicate of the Layer-2 hook-state
     sweep and would leak indefinitely)
   - Print `df -h /tmp /dev/shm` once (again, bare `df` — PATH-resolved per OBJ-5),
     followed by ONE `--- top-5 dirs under <mount> ---` block PER pressured mount in
     the array (per AC3 — two blocks when both mounts are pressured, one block when
     only one is). Each block computed via
     `timeout 5s du -xh --max-depth=1 "$mount" 2>/dev/null | awk -v m="$mount" '$2 != m' |
     sort -hr | head -5 || true`. **`du` MUST also be resolved via PATH (bare `du` —
     NOT `/usr/bin/du`) per OBJ-5** so the QA shim can replace it for synthetic
