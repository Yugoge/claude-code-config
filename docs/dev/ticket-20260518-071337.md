# BA Specification: /allow Universal Bypass — systemctl and git push

**Request ID**: 20260518-071337
**Created**: 2026-05-18T07:13:37Z
**Tier**: STANDARD

## Goal

Two hooks fail to honour the `/allow` break-glass grant. Fix `pretool-bash-safety.sh` so the non-`happy-daemon` systemctl block (lines 802-810) is reachable only AFTER the `/allow` short-circuit, and add `_check_git_allowlist()` to `_evaluate_push()` in `pretool-git-privilege-guard.py` so git push is bypassable via `/allow` for the main agent — matching the pattern already applied to `_evaluate_commit()`, `_evaluate_merge()`, `_evaluate_reset_hard()`, `_evaluate_direct_ref_mutation()`, and `_evaluate_forbidden_plumbing()`.

## Context

Commit `c6653c0` relocated the `/allow` short-circuit in `pretool-bash-safety.sh` to run before Layer 1.A (`happy-daemon` restart prohibition). However the non-`happy-daemon` systemctl block (Layer "general systemctl", lines 802-810) IS preceded by the relocated short-circuit at line 615, so the immediate blocking of `systemctl restart serve-trip.service` when `/allow` is active is attributable to one of two causes (both investigated):

**Cause A** (confirmed by code trace, inferred): The short-circuit at line 615 should fire before lines 802-810. If the user's reported block persists, the most likely explanation is that the command is being issued by a subagent (agent_id set), which causes `check_and_consume_allowlist()` to return 1 at the IS_SUBAGENT early-exit (lines 141-147). In that case, the systemctl block at line 804 is reached and exits 2.

**Cause B** (confirmed by code structure, inferred): Even if the short-circuit fires correctly for the main agent, the ABSOLUTE BAN blocks at lines 698-720 (session_dirs, happy-session-recovery, happy-restart) are NOT preceded by any `/allow` check after line 615. If a future command matched those, `/allow` would not bypass them. These are intentional permanent bans, not a bug, but they establish precedent that some blocks are after the short-circuit.

The REAL missing case for the user's stated command (`systemctl restart serve-trip.service`) when issued by the MAIN agent: the short-circuit at line 615 IS already before line 802. The gap is that the user may have verified this still blocks because `DEV_SYSTEMD=""` (empty whitelist) causes `check_systemctl_targets_all_dev()` to return 1 for ANY service target, making the block fire unconditionally when the `/allow` grant is absent or the pattern did not match.

For `pretool-git-privilege-guard.py`: `_evaluate_push()` (lines 571-595) has no `_check_git_allowlist()` call. All five other evaluation functions (`_evaluate_commit`, `_evaluate_merge`, `_evaluate_reset_hard`, `_evaluate_direct_ref_mutation`, `_evaluate_forbidden_plumbing`) DO have this call. This was explicitly noted as "Won't Have" (W2) in BA spec for task 20260517-222440, which the user has now overridden.

## Edge Cases & Risks

- **Subagent IS_SUBAGENT firewall**: `check_and_consume_allowlist()` returns 1 for subagent calls regardless of where it's called from. Adding it inside the non-happy systemctl block at line 804 does not change this — subagents still cannot bypass the block. This is correct and intentional.
- **Inline-env injection for push**: `_check_git_allowlist()` is added AFTER the `_inline_env_present()` check, so the attack class (agent injects `CLAUDE_PUSH_COMMAND_ACTIVE=1` inline) remains blocked. The allowlist check only fires after the two absolute bans pass.
- **Single-use semantics**: The `/allow` grant is consumed once by the first matching check. If the main-agent short-circuit at line 615 fires, the grant is marked for PostToolUse deletion; the second callsite inside the non-happy block would not be reached anyway. No double-consumption risk.
- **Force-push remains blocked**: `_push_has_forbidden_ref_mutation()` check at line 575 runs before `_check_git_allowlist()`, so `git push --force` and `git push -f` are still blocked even with `/allow` active.
