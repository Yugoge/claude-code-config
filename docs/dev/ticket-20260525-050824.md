## Goal

Replace the direct score-write model in `scripts/score-update.sh` with an append-only JSONL audit log (`logs/lifecycle.jsonl`) where each score event is a new line entry and reads use the latest entry per agent. Writes use Compare-And-Swap on `prev_score` (under exclusive flock) to detect and reject stale-state concurrent writes that caused the observed `dev 81→73` score drift.

---

## Context

The current score persistence model (`agent-scores.json`) serializes the entire JSON on every write, relying on `flock` for mutual exclusion but not detecting stale-reader drift between sequential calls. During a close cycle, 3-6 sequential `score-update.sh` invocations run with independent lock acquisitions — a concurrent out-of-band write between any two calls produces an undetected inconsistency with no audit trail.

`logs/lifecycle.jsonl` was created and tracked in git by Cycle 20260522-000000 (commit `af96c61`) specifically to receive R9 entries. It is currently empty (0 bytes).

**Scope constraint**: This is a mandatory dedicated cycle. Touch ONLY the score persistence layer and its direct consumers/tests. Do NOT bundle W3/W5 events, JSONL rotation policy, or any other backlog item.

## JSONL Entry Schema

The user requirement (spec-20260520-221059.md Section 5 R9) specifies exactly 8 mandatory fields:

### Canonical score_update entry (8 mandatory fields verbatim from R9)

```json
{
  "ts": "2026-05-25T05:08:24Z",
  "agent": "dev",
  "event": "close_success_qa_pass",
  "prev_score": 15,
  "new_score": 17,
  "delta": 2,
  "actor": "orchestrator",
  "reason": "20260525-050824"
}
```

### Canonical score_baseline_import entry (8 mandatory fields)

```json
{
  "ts": "2026-05-25T05:08:24Z",
  "agent": "ba",
  "event": "score_baseline_import",
  "prev_score": 67,
  "new_score": 67,
  "delta": 0,
  "actor": "migration",
  "reason": "baseline import from agent-scores.json"
}
```

**Mandatory field notes**:
- `ts`: ISO-8601 timestamp of the event
- `agent`: canonical agent name (ba/dev/qa/...)
- `event`: event key (e.g., `close_success_qa_pass`, `score_baseline_import`)
- `prev_score`: score value before this event (for CAS comparison and audit)
- `new_score`: score value after this event (post-clamp)
- `delta`: signed delta applied (may differ from `new_score - prev_score` when clamped)
- `actor`: who triggered the event (e.g., `orchestrator`, `migration`)
- `reason`: the primary audit/traceability field required by R9 verbatim. The existing `--note` CLI parameter in score-update.sh maps to this field. Dev MUST rename the emitted JSON field from `"note"` (current) to `"reason"` (required). The CLI parameter name `--note` may remain unchanged to avoid breaking callers.

**Advisory implementation-only fields (NOT verified by ACs, NOT mandatory JSONL)**:
- `kind`: discriminator (e.g., `"score_update"`, `"score_baseline_import"`) — recommended for stream filtering but not part of the R9 8-field schema; dev may add if desired
- `event_id` (UUID4), `seq` (monotone int per agent), `prev_event_id` (UUID4): ABA-prevention chain fields; optional; not required for CAS comparison in this cycle
- `unclamped_score`: pre-clamp arithmetic target; useful for audit but not in the mandatory schema

## CAS Protocol

1. Acquire exclusive flock on `logs/lifecycle.jsonl.lock` (same 10s timeout as existing lock in score-update.sh)
2. Open `logs/lifecycle.jsonl` and scan from beginning to find the LAST valid JSON line for this agent (by `agent` field match). Skip unparseable lines only if they are the final line (partial write detection). Invalid non-final lines: exit 2.
3. If `--expected-prev-score` was passed by caller:
   - If no prior entry exists for agent: CAS passes (first write after baseline import — accept any prev_score)
   - If latest entry exists and `latest.new_score != expected_prev_score`: exit 3 (CAS conflict, write nothing)
4. Compute new_score = max(0, min(100, prev_score + delta))
5. Append one complete JSON line (terminated by `\n`) and fsync
6. Release lock

**--expected-prev-score is OPTIONAL**. When omitted, step 3 is skipped (serialized-append mode). Normal callers (commands/close.md, commands/dev.md) do not pass it in this cycle — they use serialized-append. The CAS mode is available for future callers that need explicit drift detection.

---

## Migration Protocol (Option B-prime)

`scripts/lifecycle-baseline-import.sh` must be run ONCE before any new score events are written. It is safe to run repeatedly (idempotent).

**Idempotency rule**: For each canonical agent, skip if lifecycle.jsonl contains any valid line whose `agent` field matches this canonical agent — regardless of event type. This prevents appending a baseline that would become the "latest" entry after existing score events.

**Atomicity**: All 21 canonical agents are processed under a single exclusive flock acquisition. If the script is interrupted mid-run, re-running it will skip agents already imported and complete the remainder.

## Edge Cases & Risks

1. **lifecycle.jsonl empty / agent absent**: score-inject.sh falls back to neutral mid-tier (identical to current missing-file behavior). score-update.sh proceeds with prev_score = default 50 for CAS purposes.
2. **Partial final JSONL line** (mid-write crash): reader skips the unterminated final line silently. Writer uses flock to prevent torn writes in practice; the skip handles crash recovery.
3. **lifecycle.jsonl unreadable / malformed non-final line**: score-inject.sh exits 1 (not neutral fallback — audit corruption must surface). score-update.sh exits 2.
4. **test_AC4 pre-existing failure**: The test asserts `delta == 8` and `uncapped_delta == 108` for ba/close_success_qa_pass — both wrong since task 20260524-205206 changed the delta to +1. This must be fixed as part of test maintenance in M8 REGARDLESS of R9 changes.
5. **score-inject.sh shared flock**: readers acquire shared flock (LOCK_SH) on `lifecycle.jsonl.lock`, compatible with other readers but blocking exclusive writer flock. This prevents torn-read of a partially-appended line.
6. **Baseline import after first score event**: If a score event was written to lifecycle.jsonl for agent X before the baseline import ran, the baseline import correctly skips X (idempotency rule). Agent X has a score event but no explicit baseline; reader uses that score event as the anchor.
