## Goal

Integrate the sentinel-grant bypass mechanism into `hooks/pretool-write-guard.sh` so that legitimate orchestrated Write-overwrites of existing files can be authorized via a structured sentinel grant — eliminating the hacky `_recheck.json` fallback paths and manual `cp -f` workarounds that accumulated across prior dev cycles.

## Context

The sentinel-grant system was introduced in task 20260519-211515 R2 for Bash commands. It writes a structured grant at `/tmp/claude-grants/<task_id>.json` containing `{task_id, session_id, allowed_operations[], created_at, expires_at}`. The Bash path in `pretool-bash-safety.sh` calls `match_sentinel_grant_for_bash_command()` and if a sentinel exists-but-doesn't-match, suppresses the legacy `/allow` fallback (the CF-1 invariant).

`pretool-write-guard.sh` has no equivalent: it calls `read_grant("Write", sid)` for the legacy pattern-string grant but has no sentinel path at all. `posttool-allowlist-consume.py` restricts sentinel consumption to `tool_name == "Bash"` only (line 105). As a result, even if a `/allow`-style grant wrote a Write-targeted sentinel, the write-guard would never check it and posttool would never consume it.

This gap has caused documented write-guard blocks across multiple prior cycles (CF2-10 first documented in task 20260519-211515 context.json OOS observation #4; recurred in dev-report-d1e94e.json, dev-report-20260519-211515-iter3.json, qa-output-ba-validation-20260519-211515.json, and completion-20260519-211515-redev9items.md).

The user-requirement document explicitly cites codex's top-priority verdict from the OOS triage (priority B — "major, days") for this item.

## Requirements (MoSCoW)

- M2: `hooks/pretool-write-guard.sh` extracts `FILE_PATH`, `_WG_SID`, and resolves task-id (with fallback: input `task_id` field → `CLAUDE_TASK_ID` env → session_id) BEFORE both the legacy `/allow` block and the new sentinel check. The sentinel check fires when a sentinel exists for the task: if it matches → `exit 0`; if it exists-but-doesn't-match → suppress legacy `/allow` fallback (CF-1 invariant parity with bash-safety). ## Edge Cases & Risks

- **Risk 1 (CF-1 parity)**: If a sentinel exists for the task but does NOT match the Write target path, the legacy `read_grant("Write", sid)` must be suppressed — exactly as `pretool-bash-safety.sh` suppresses legacy grants when a bash sentinel exists-but-doesn't-match (lines 516-526). Failing to implement this allows a broad legacy `/allow Write` to bypass the strict target-path check.
- **Risk 2 (field name)**: The `allowed_operations[]` entry schema uses `"target"` (not `"target_path"`) per the module docstring at `allowlist.py:22`. Dev MUST use `entry.get("target")` not `entry.get("target_path")`. Codex finding #2 identified this gap.
- **Risk 3 (parse ordering)**: `pretool-write-guard.sh` currently extracts `FILE_PATH` at line 50 — AFTER the legacy `/allow` block at lines 31-46. The sentinel check needs `FILE_PATH`, so both file-path extraction and task-id resolution must be hoisted before the sentinel block. Codex finding #3.
- **Risk 4 (posttool path)**: `posttool-allowlist-consume.py` reuses `command` (Bash-specific) for sentinel matching. The Write path must use `file_path` from `tool_input`, not `command`. Codex finding #4.
- **Risk 5 (session_id fallback)**: Resolving task_id in write-guard.sh must use the same fallback chain as posttool: `data.task_id → CLAUDE_TASK_ID env → session_id`. Using only `CLAUDE_TASK_ID` misses grants keyed by session-id fallback. Codex finding #3.
- **Risk 6 (subagent exemption)**: The sentinel check in write-guard.sh must only fire for the main agent (inside the `_WG_IS_SUB != "1"` block). Subagents are exempt from the write-guard sentinel path per existing subagent-bypass behavior (line 18-20 of write-guard.sh).

## Acceptance Criteria

### AC1: match_sentinel_grant_for_write present and correct

- GIVEN `hooks/lib/allowlist.py` contains `match_sentinel_grant_for_write(task_id, session_id, target_path)`
- WHEN called with a valid unexpired sentinel that has `allowed_operations = [{"op": "Write", "target": "/path/to/file.json"}]` and `grant["session_id"] == session_id`
- THEN it returns the matched dict entry (not None)
- AND when called with mismatched `session_id`, returns None
- AND when called with mismatched `target_path`, returns None
- AND when called with an expired sentinel, returns None

### AC2: write-guard.sh sentinel check fires before existing-file block

- GIVEN `pretool-write-guard.sh` is invoked with a Write tool call targeting an existing file
- WHEN a valid sentinel grant for the current task-id exists with `{"op": "Write", "target": "<file_path>"}`
- THEN the hook exits 0 (allows the write)
- AND when a sentinel exists for the task but does NOT match the target file, the legacy `read_grant("Write")` is NOT consulted (CF-1 parity), and the hook exits 2 (blocks)

### AC3: posttool consumes Write sentinel on terminal result

- GIVEN `posttool-allowlist-consume.py` fires for a `tool_name == "Write"` event
- WHEN a sentinel grant for the current task-id exists with a Write operation matching the `tool_input.file_path`
- THEN `consume_sentinel_grant_on_terminal_result()` is called
- AND the sentinel file is unlinked

### AC4: scope integrity

- GIVEN the three files modified: `hooks/lib/allowlist.py`, `hooks/pretool-write-guard.sh`, `hooks/posttool-allowlist-consume.py`
- WHEN `git diff HEAD` is examined
- THEN only those three files appear in the diff
- AND `hooks/userprompt-consent-allowlist.sh` (grant creation) is NOT modified
