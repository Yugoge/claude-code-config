## Round 1: Initial Codex Review

**Codex verdict**: DO NOT CLOSE

**Findings**:

### F1 — BLOCKER: Cross-SID Leak Not Fixed (Codex reproduced)

Codex ran actual test code against the installed hook:

- Legacy grant written at `/tmp/claude-bash-allowlist-{MAIN_SID}.json` (orchestrator SID)
- Sentinel written at `/tmp/claude-grants/{TASK_ID}.json` with `session_id=MAIN_SID`
- PostToolUse fires with subagent `session_id=SUB_SID` (different from MAIN_SID)
- Hook: sentinel consumed; `consume_grant_for_posttool(SUB_SID, ...)` called; Edit 4 unlinks `/tmp/claude-bash-allowlist-{SUB_SID}.json` (nonexistent)
- Result from Codex test run: `sentinel_exists=no`, `legacy_main_exists=yes` — **LEAK CONFIRMED**

**PROPOSED_FIX** (Codex R1): Load the sentinel grant before consuming it, capture `grant["session_id"]`, unlink legacy grants for both payload SID and sentinel owner SID.

### F2 — MAJOR: Edit 4 direct unlink is unlocked (bypasses flock+logging)

The direct `Path.unlink()` in Edit 4 bypasses `consume_grant_for_posttool()`'s flock, grant-matching, and stderr logging. In normal same-SID execution, the double-unlink is safe (second attempt gets FileNotFoundError). The concern is correctness for the cross-SID case — same root as F1.

**PROPOSED_FIX** (Codex R1): Move into a library helper that uses the sentinel's stored writer SID.

### F3 — QA Evidence Gap

QA's AC1 test used `session_id=sid` for both the sentinel writer and the subagent PostToolUse payload — only testing the same-SID path. Cross-SID was never exercised.

**PROPOSED_FIX** (Codex R1): Verdict FAIL until cross-SID test added.

## Round 2: Codex Scoping Analysis

**Question posed**: Is cross-SID in-scope for this requirement, or is it a separate pre-existing limitation?

**Codex verdict**: BLOCKER — cross-SID IS in-scope.

**Codex confirmed the attack chain**:
1. `/allow` writes legacy grant at `MAIN_SID`
2. Subagent (different `SUB_SID`) executes authorized Bash command via sentinel
3. Posttool hook: sentinel consumed, `consume_grant_for_posttool(SUB_SID, ...)` unlinks nothing meaningful
4. Legacy grant at `MAIN_SID` survives
5. Main agent (not a subagent) makes a second Bash call using `MAIN_SID`
6. `pretool-bash-safety.sh:541-557`: main-agent legacy fallback runs (IS_SUBAGENT=0), reads legacy grant, emits `permissionDecision: allow`
7. **Main agent reuses the grant** — direct violation of user requirement

**Codex on the scope argument**: "BA spec 4-edit limit is an implementation constraint, not the user-facing acceptance boundary. The stated requirement is specifically to prevent main-agent reuse after subagent sentinel execution; cross-SID is the documented production path."
