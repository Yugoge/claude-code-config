### Item A — raw-`$COMMAND` rule classification (the heart of this cycle) — CORRECTED after QA pass1

**The criterion (corrected).** A conversion to `COMMAND_CONTEXT_STRIPPED` is safe ONLY IF the rule's match result is IDENTICAL between the raw view and the stripped view for every command — i.e. converting can remove false positives but can NEVER silently remove a true block (false negative). Given the verified stripper semantics (`bash_context_strip.py:305-330`):

- The **command-word token** (`i <= cmd_idx`) is preserved verbatim in both the DANGER and non-DANGER paths. A rule whose ENTIRE danger signal is the command word is therefore identical raw-vs-stripped → **SAFE to convert**.
- For a **NON-DANGER** command word, any **argument / option / subcommand / path** token that is single- or double-quoted is ERASED in the stripped view (`single → ""`, `double → "<only $()/backtick>"`). So `git reset "--hard"` (a REAL hard reset — bash strips the quotes at exec) becomes `git reset ` in the stripped view and the `--hard` predicate no longer fires → **silent false negative**. Any rule whose danger token lives in such a position is **NOT convertible**.
- For a command word that is ALREADY in `DANGER_COMMANDS`, quoted args are UNQUOTED (exposed) rather than erased, so an argument-anchored rule on that command word is also identical raw-vs-stripped → **SAFE to convert** (this is the `kill` exception below).

This corrects pass1, which classified the git argument-anchored rules (`reset --hard`, `push --force/--delete`, `checkout … -- .`, `restore --source … -- .`, `revert <hash>`) as convertible "verb" rules. Their danger token is an ARGUMENT, not the bare verb; under the stripper they behave exactly like the path/value rules pass1 correctly forbade. They are moved to DO-NOT-CONVERT below.

**Class 1 — SAFE to convert (signal survives stripping identically to raw):**
- `dd | mkfs | fdisk | shred` (`:1045`) — the COMMAND WORD itself is the entire signal (`^\s*(dd |mkfs|fdisk|shred )`). Preserved verbatim in the stripped view; converting removes false positives like `echo "dd if=/dev/zero of=/dev/sda"` (the quoted arg to `echo` is erased) with zero false-negative risk. **No `DANGER_COMMANDS` entry is needed or wanted** — there is no dangerous quoted arg to expose; the signal is the command word, which is never stripped.
- `kill <PID>` (`:1311`) — argument-anchored (`kill\s+[0-9]`), BUT `kill ∈ DANGER_COMMANDS` already, so the stripper UNQUOTES its args (`kill "1234"` → `kill 1234`). Match is therefore identical raw-vs-stripped, and converting removes false positives like `echo "kill 1234"`. **SAFE — and requires NO new `DANGER_COMMANDS` entry** (kill is already a danger command; this is exactly why the sibling `kill -sig` rule at `:1119` already reads the stripped view). This is the principled exception: an argument-anchored rule is convertible iff its command word is already a DANGER command.

**Class 2 — DO NOT convert (danger token is a quotable argument/option/subcommand/path → quoting it erases the match in the stripped view → false negative):**

*Git rules — argument-anchored (MOVED here from pass1's Class-1; this is the core QA correction):*
- `git reset --hard` (`:1405`) — danger is the `--hard` ARGUMENT. `git reset "--hard"` is a real hard reset but strips to `git reset `. **DO NOT convert.**
- `git push --force/-f/--force-with-lease/--delete/-d/--mirror`, `+ref`, `:branch` (`:1414-1415`) — danger is the flag/refspec ARGUMENT. **DO NOT convert.**
- `git checkout <ref> -- . / -- * / -- dir/` (`:1372`) — danger is the wide-path ARGUMENT (`-- .` etc.). **DO NOT convert.**
- `git restore --source … -- . / -- * / -- dir/` (`:1389-1391`) — danger is the `--source` flag + wide-path ARGUMENT. **DO NOT convert.**
- `git revert <hash>/<ref>^/<ref>~N/<ref>@{N}` (`:1461-1472`) — danger is the ref-modifier ARGUMENT. **DO NOT convert.**
- `git update-ref / branch -D/-d/--delete/--force/--move / symbolic-ref … HEAD refs/` (`:1421`) — danger is the subcommand/flag/ref ARGUMENT. **DO NOT convert.**
- `git revert/cherry-pick/rebase` subagent-history block (`:1441`) — danger is the subcommand ARGUMENT word after `git`. **DO NOT convert.**
- `git stash push/save/create/store/-u/--all` and bare `git stash` (`:1350-1367`) — danger is the `stash` + push/save subcommand ARGUMENT (`git stash "push"` strips to `git stash `). pass1 called this "debatable"; verified argument-anchored → **DO NOT convert.**

*Docker / systemctl rules — argument/subcommand/target-anchored (CORRECTED from pass1, which over-included these as safe "verbs"):*

## Edge Cases & Risks

- **A — false negatives (highest risk)**: converting a rule whose danger token is a quoted argument will let the stripper remove it, silently disabling the block. Mitigated by Class-1/Class-2 split + mandatory should-block test per rule.
- **A — `DANGER_COMMANDS` omission**: converting a verb rule without adding its command word to `DANGER_COMMANDS` can strip the verb's own dangerous flags (e.g. `--hard`, `--force`). The should-block test catches this.
- **A — fail-closed preservation**: the existing `:1096-1106` setup falls back to raw `$COMMAND` on any stripper failure (timeout/ulimit/exception). New rules MUST read `COMMAND_CONTEXT_STRIPPED` (which equals raw on failure), preserving fail-closed behavior. Do not bypass the timeout/ulimit wrapper.
- **B — candidate order**: writer keys by `CLAUDE_TASK_ID:-SID`; pretool (`write-guard.sh:79-85`) and posttool (`consume.py:230-238`) must enumerate `[writer_key, data.task_id, session_id]` in the same de-duped order. Divergence = grant found at write but not consumed (leak) or vice versa.
- **C**: a literal-task-id grep could hide in a non-obvious fallback branch; dev must grep the whole step-7 path, not just the three cited lines.
