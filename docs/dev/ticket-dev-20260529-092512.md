### Item A — raw-`$COMMAND` rule classification (the heart of this cycle)

The stripper is purpose-built around `DANGER_COMMANDS` (`bash_context_strip.py:26`): it strips quoted-string arguments EXCEPT for the args of the listed danger commands. Therefore rules fall into two classes:

**Class 1 — VERB rules (convertible; arg is not the signal, the command word is):**
Candidates whose match should fire on the executable-context view so that a quoted-string mention does not false-positive. For each verb converted, its command word MUST first be added to `DANGER_COMMANDS` (or proven already covered) so its own dangerous arguments are not stripped:
- `git stash push/save/...` (`:1350-1361`)
- `git checkout <ref> -- <path>` (`:1372`)
- `git restore --source` (`:1389-1391`)
- `git reset --hard` (`:1405`)
- `git push --force/--delete/...` (`:1414-1415`)
- `git update-ref / branch -D / symbolic-ref` (`:1421`)
- `git revert/cherry-pick/rebase` (`:1441`, `:1461-1472`)
- `docker compose down/restart/stop` (`:1076`)
- `docker stop/restart/rm/kill ... happy` (`:1066`)
- `docker system prune -a` (`:1083`)
- `dd/mkfs/fdisk/shred` (`:1045`)
- `systemctl ... docker` (`:1052`), `systemctl <verb>` non-happy (`:1129`)

**Class 2 — PATH/VALUE rules (DO NOT convert; the quoted target IS the signal):**
Converting these would create FALSE NEGATIVES (suppressing a block that should fire), because the dangerous token legitimately appears as a quoted path/value that the stripper removes:
- `.env|credentials|secret|password` redirect rules (`:1031,1037`)
- `/etc/docker/daemon.json` writers (`:1059`)
- `/root/happy/packages/` source-contamination (`:1162-1165`)
- `Dockerfile.webapp` / `HAPPY_SERVER_URL=` build rules (`:1175-1208`)
- DB-mutation rules (`:1226,1232`), session-API curl rules (`:1270,1278`)
- sentinel-filename rules (`:669-670`), `session_dirs.txt` (`:1007`), daemon-restart-flag (`:624-625`)
- `/usr/bin/happy` symlink/exec rules (`:1301,1321,1330`)
- happy-daemon systemctl rules (`:564-586`) — already use multi-grep on raw command by design

> **Hard constraint for dev**: the existing line `:1143` deliberately uses the STRIPPED view for the verb portion (`rm|mv`) and the RAW view for the path portion (`workflow-*.json|.claude/todos/`). This split-grep pattern is the canonical template for any Class-1 rule whose block depends on BOTH a verb AND a path. Follow it; do not collapse to a single stripped grep when a path predicate is present.

## Edge Cases & Risks

- **A — false negatives (highest risk)**: converting a rule whose danger token is a quoted argument will let the stripper remove it, silently disabling the block. Mitigated by Class-1/Class-2 split + mandatory should-block test per rule.
- **A — `DANGER_COMMANDS` omission**: converting a verb rule without adding its command word to `DANGER_COMMANDS` can strip the verb's own dangerous flags (e.g. `--hard`, `--force`). The should-block test catches this.
- **A — fail-closed preservation**: the existing `:1096-1106` setup falls back to raw `$COMMAND` on any stripper failure (timeout/ulimit/exception). New rules MUST read `COMMAND_CONTEXT_STRIPPED` (which equals raw on failure), preserving fail-closed behavior. Do not bypass the timeout/ulimit wrapper.
- **B — candidate order**: writer keys by `CLAUDE_TASK_ID:-SID`; pretool (`write-guard.sh:79-85`) and posttool (`consume.py:230-238`) must enumerate `[writer_key, data.task_id, session_id]` in the same de-duped order. Divergence = grant found at write but not consumed (leak) or vice versa.
- **C**: a literal-task-id grep could hide in a non-obvious fallback branch; dev must grep the whole step-7 path, not just the three cited lines.
