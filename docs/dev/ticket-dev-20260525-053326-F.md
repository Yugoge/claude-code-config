## Edge Cases & Risks

- `tests/test-lock-detection.sh` uses `cd "$TEST_REPO"` at line 20, then later `cd /` at line 100. The trap fires at EXIT so the working dir may still be inside `$TEST_REPO` when cleanup runs — this is safe because `TEST_REPO` is an absolute path and bash `trap` uses the variable's value at trap-define time (the variable is defined before the cd). Dev should verify the trap is placed BEFORE the `cd` to ensure the variable is captured.
- `tests/verify-stop-spec-session-isolation.sh` already has `rm -rf "$WORK"` at the end of the script. Adding a trap means double-cleanup on normal exit (trap fires after the explicit rm) — this is harmless since `rm -rf` on a missing directory exits 0.
- `hooks/pre_slashcommand_validate.sh` is listed in hooks/INDEX.md and hooks/README.md but is NOT registered in `settings.json`. Removing the placeholder comments (while keeping the actual code and exit 0) does not affect behavior.
- The stale-comment lines in `hooks/smart-checkpoint.sh` and `hooks/ensure-git-repo.sh` contain a `rm` command in a comment (not code). The bash-safety hook does not block the Edit tool writing such comments; this is edit-only work.
