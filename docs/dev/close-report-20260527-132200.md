## Round 1b: Style Inspector Findings Assessment

The style-inspector found 10 critical violations, all introduced_in_diff: true.

**Group A — use-source-venv (4 violations)**:
- agents/graphify.md:37, commands/dev.md:182-184, commands/dev.md:697, commands/pull.md:110

Analysis: Standard 3 of agents/style-inspector.md explicitly says "Scan all .sh and .md files for
python3 calls". The command/agent Markdown files are explicitly in scope. The project's established
convention at commands/dev.md:802 uses "source venv/bin/activate && python3". The new graphify
invocations inconsistently use bare python3 without venv activation.

Verdict on Group A: REAL violations. The style-inspector correctly flags these. The 3 invocations
in commands/dev.md and commands/pull.md are operator-context Bash blocks that will be literally
executed by the orchestrator — bare python3 can fail silently if the project venv is not active at
that point. The invocation in agents/graphify.md:37 is a prose instruction to a subagent, but it
still constitutes a bare python3 reference in a .md file, and the project convention is venv-first.

**Group B — integer-step-numbering (3 violations — PROSE TEXT)**:
- agents/graphify.md:5, agents/dev.md:116,121, agents/qa.md:96

Analysis: AC6's grep recipe says "WHEN grepping for decimal step headings ('Step 1.5', 'Step 7.5'
as headings with ## or ###)". QA correctly passed AC6 because no HEADINGS with decimals exist.
However, Standard 4 of agents/style-inspector.md says "Scan .md files in .claude/ for decimal step
patterns (Step N.M)". The standard is broader than AC6's verification recipe — it covers prose
references, not only headings.

This is a spec_text_vs_execution_drift issue: AC6's WHEN clause specifies "as headings with ## or
###" but Standard 4 covers prose references too. The style violations are real under Standard 4
even though AC6 passes under its own recipe.

Verdict on Group B: REAL style violations. The prose references "Step 7.5" and "Step 1.5" in
agents/graphify.md, agents/dev.md, and agents/qa.md violate Standard 4. These are fixable:
replace with "between Step 7 and Step 8" and "between Step 1 and Step 2".

The AC6 pass is technically correct (no decimal headings with ## or ###), but the style violations
stand as separate findings. This is also a spec_text_vs_execution_drift objection: AC6's literal
grep recipe is narrower than Standard 4's actual scope.

**Group C — english-only (3 violations — CJK characters)**:
- agents/ba.md:640,646, agents/qa.md:100, scripts/graphify-query.py:47

Analysis: AC2 explicitly requires agents/ba.md to contain "the list of implicit reference trigger
words (之前/已有/现有/原来的/previous/existing/original)". The spec was written with full knowledge
of these CJK characters. Standard 6 of agents/style-inspector.md allows non-English verbatim in
docs/dev/ticket-*.md files but not in agents/*.md or scripts/*.py.

The functional requirement is that ba.md and qa.md contain the actual trigger words so that BA and
QA agents can detect them when scanning user requirements. The style-inspector proposes Unicode
escapes (之前 etc.) in scripts/graphify-query.py as a compliant alternative that preserves
runtime matching behavior. For agents/*.md, pointer to ticket file or Unicode escape form.

Verdict on Group C: REAL violations under Standard 6. However, the fix is minimal — Unicode escapes
in graphify-query.py preserve runtime behavior identically. For agents/ba.md and agents/qa.md, the
style-inspector's own recommendation provides a compliant path: reference the Chinese equivalents
via pointer to the ticket, or use Unicode escape forms.

Critically: the underlying feature requirement (trigger word detection) is not affected by the fix.
These are fixable-in-place violations that do not require architectural rework.
