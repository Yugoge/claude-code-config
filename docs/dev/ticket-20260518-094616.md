## Goal

Refactor the `/allow` bypass system to address 6 architectural issues identified by the architect, while keeping the set of bypassable operations exactly identical to today's. No new bypass surface must be opened; no currently-bypassable surface must be closed.

## Context

The `/allow` bypass system consists of three interacting layers:
1. `hooks/userprompt-consent-allowlist.sh` — writes the grant file on `/allow <pattern>` user input
2. `hooks/pretool-bash-safety.sh` — bash hook that reads the grant at the global short-circuit (line 615) and at 6 per-rule call sites (lines 805, 984, 995, 1007, 1027, 1110)
3. `hooks/pretool-git-privilege-guard.py` — Python hook with `_check_git_allowlist()` called in 6 per-function sites inside evaluate-* functions
4. `hooks/lib/allowlist.py` — Python library with `read_grant()` (used by `posttool-allowlist-consume.py`)
5. `hooks/posttool-allowlist-consume.py` — deletes the grant after successful tool execution

The grant file lives at `/tmp/claude-bash-allowlist-<sid>.json`. The bash hook, the Python guard, and `posttool-allowlist-consume.py` all read it inline (not via `hooks/lib/allowlist.py`), creating 4 independent implementations of grant-read/match logic (allow-6).

Task 20260518-071337 added the defense-in-depth call at line 805 as part of adding `/allow` bypass to the non-happy systemctl block. This spec addresses the subsequent architectural cleanup the architect identified.

### allow-2 — Unstructured per-function allowlist enforcement in `pretool-git-privilege-guard.py`

- **Observed**: architect flagged 6 independent `_check_git_allowlist(command, data)` calls with no structural enforcement preventing a new evaluate-* function from missing the call
- **Measured**: calls exist at lines 455, 480, 583, 602, 615, 648; `_evaluate_command` dispatcher at line 657 has no central call
- **Expected**: either (a) a single centralised call in the dispatcher that covers all operations except push (which has ordering requirements), or (b) a documented convention that makes the obligation explicit to future authors
- **Gap**: silent omission risk when new evaluate-* functions are added

Critical constraint (scope boundary): `_evaluate_push` (lines 571-598) places `_check_git_allowlist` AFTER `_push_has_forbidden_ref_mutation`. This ordering is intentional — it ensures `git push --force` is blocked even when `/allow` is active. Any fix must preserve this ordering; moving the check before `_evaluate_push` in the dispatcher would allow force-push via `/allow`, violating the non-goals.

BA evaluation of options:
- Option (a) — move to dispatcher for all ops EXCEPT push: achieves structural enforcement for 5 of 6 functions; `_evaluate_push` keeps its own correctly-positioned call. This is the stronger fix because it is mechanically enforced. The one remaining per-function call in `_evaluate_push` is self-documenting (it appears after the absolute ban, clearly intentional).
- Option (b) — keep 6 per-function calls + add dispatcher docstring: zero structural change; relies entirely on future authors reading and following the convention. Convention-only is weaker than mechanical enforcement.

**Recommended**: Option (a). Move `_check_git_allowlist` to `_evaluate_command` dispatcher as the default path, called before the per-operation dispatches. Remove the 5 per-function calls from `_evaluate_commit`, `_evaluate_merge`, `_evaluate_reset_hard`, `_evaluate_direct_ref_mutation`, and `_evaluate_forbidden_plumbing`. Leave `_evaluate_push`'s call in its current position (after `_push_has_forbidden_ref_mutation`). Add a comment in `_evaluate_command` explaining why push is exempt from the central call and keeps its own.

If the central call returns True (grant matched), `_evaluate_command` returns immediately — none of the evaluate-* functions are entered. This preserves the absolute ban in `_evaluate_push` because when the central call matches, the push path is never entered at all; and if the central call does NOT match (no grant or no pattern match), execution falls through to `_evaluate_push` which then hits its own correctly-ordered `_check_git_allowlist` (which also won't match, but the absolute ban fires first regardless).

Wait — there is a subtle ordering problem with the above: if the central call is checked first for ALL commands including push, then for a push command with a valid `/allow` grant, the central call would match and return early, bypassing `_push_has_forbidden_ref_mutation`. This would allow `git push --force` with a broad `/allow` grant. This is the exact scope regression the architect warned about.

Correct implementation of Option (a): the central call in `_evaluate_command` must EXCLUDE push commands. Implementation:

```python
def _evaluate_command(command, data):
    if _looks_like_git_forbidden_plumbing(command):
        _evaluate_forbidden_plumbing(command, data)
    if _looks_like_git_reset_hard(command):
        _evaluate_reset_hard(command, data)
    if _looks_like_git_direct_ref_mutation(command):
        _evaluate_direct_ref_mutation(command, data)
    if _looks_like_git_push(command):
        _evaluate_push(command, data)  # push keeps own _check_git_allowlist after absolute ban
        return
    # Central allowlist check for all non-push operations.
    # Push is excluded because _evaluate_push must check allowlist AFTER
    # _push_has_forbidden_ref_mutation to keep git push --force blocked.
    if _check_git_allowlist(command, data):
        return
    if _looks_like_git_merge(command):
        _evaluate_merge(command, data)
    if _looks_like_git_commit(command):
        _evaluate_commit(command, data)
```

But this restructuring only works for merge and commit. It does NOT help with reset_hard, direct_ref_mutation, forbidden_plumbing — which are already dispatched before the central check. A cleaner split: call `_check_git_allowlist` centrally for only the operations that currently have it as their first action. Those are: commit (L455), merge (L480), reset_hard (L602), direct_ref_mutation (L615), forbidden_plumbing (L648). Push (L583) is after the absolute ban — keep it there.

Cleanest implementation: move the central call to be the FIRST check in `_evaluate_command`, but only for non-push operations. Check `_looks_like_git_push` first; if true, route to `_evaluate_push` (which handles its own allowlist internally) and return. For all other operations, check allowlist centrally, then dispatch.

This eliminates the 5 per-function calls. See Technical Hints for the precise implementation.

### allow-5 — Duplicate IS_SUBAGENT inline Python parse in `pretool-bash-safety.sh`

- **Observed**: two independent IS_SUBAGENT inline Python parses: one at lines 141-147 inside `check_and_consume_allowlist()`, one at line 1068 as the outer `IS_SUBAGENT` variable
- **Measured**: inner parse at lines 142-147 inside `check_and_consume_allowlist()`. Outer `IS_SUBAGENT` variable assignment at line 1068. The comment at lines 137-143 documents the reason: the outer variable was not yet set when the function was written (outer was at ~line 499; now at line 1068 — still AFTER line 615 where the function is first called).
- **Expected**: ideally a single parse
- **Gap**: 1 duplicate Python subprocess invocation per function call (performance), plus maintenance risk if the two parses diverge

BA investigation of whether the outer IS_SUBAGENT can be moved earlier:

The outer `IS_SUBAGENT` at line 1068 guards the subagent-initiated git history mutation block (lines 1069-1090). If we moved it to, say, line 10 (after `INPUT=$(cat)`), the function `check_and_consume_allowlist` at line 134 could reference the outer variable, eliminating the inner parse.

Risk of moving `IS_SUBAGENT` earlier: the variable is a shell-global (not local); moving it to the top of the script would not break any logic — it reads the same `$INPUT` value regardless of position. The only risk is if some earlier code modifies `$INPUT`, which does not happen (INPUT is assigned once at line 6 and never reassigned). Moving IS_SUBAGENT to line ~17 (after PYTHON_BIN resolution) is safe.

However, `check_and_consume_allowlist` declares `is_sub` as `local` — it is a function-local variable. If we move `IS_SUBAGENT` assignment earlier, the function can reference the outer `$IS_SUBAGENT` directly instead of spawning a Python subprocess. This requires changing the function body from the inline parse to a reference to the outer variable.

**Conclusion**: This IS safe to refactor. Move `IS_SUBAGENT` assignment to the top of the script (after PYTHON_BIN resolution, ~line 17). Remove the inline parse from `check_and_consume_allowlist`. Replace `local is_sub; is_sub=$(...)` with a direct check on `$IS_SUBAGENT`. This is a SMALL refactor within pretool-bash-safety.sh.

### allow-6 — 4 independent grant-read implementations

- **Observed**: `hooks/lib/allowlist.py::read_grant`, `_check_git_allowlist` in `pretool-git-privilege-guard.py`, a heredoc in `pretool-bash-safety.sh`, and independent logic in `posttool-allowlist-consume.py` are four independent implementations
- **Measured**:
  - `hooks/lib/allowlist.py:read_grant` — Python, takes `(tool_name, sid)`, does literal/regex match on `tool_name` against pattern
  - `pretool-git-privilege-guard.py:_check_git_allowlist` — Python, takes `(command, data)`, extracts sid internally, does literal/regex match on `command`
  - `pretool-bash-safety.sh` heredoc at lines 163-322 — bash launches Python subprocess, does split-subcommand match + SIGALRM timeout + flock
- **Gap**: drift risk; a fix to one implementation may not propagate to others

BA evaluation of consolidation feasibility:

Full consolidation within a single cycle is not feasible without risk:

1. **bash → Python**: the bash heredoc in `pretool-bash-safety.sh` is a complete V1/V2/V3 implementation with SIGALRM timeout, flock serialization, and compound-command splitting. Replacing it with a subprocess call to a Python CLI entry point (`hooks/lib/allowlist.py`) is architecturally desirable but requires: (a) adding a CLI entry point to `allowlist.py`, (b) changing the bash shell result-parsing from the `CONSUMED\t...\t...\t...` protocol to whatever the new CLI emits, (c) end-to-end testing of the new path. This is non-trivial and carries regression risk to the /allow critical path.

2. **Python guard → lib**: `_check_git_allowlist` in `pretool-git-privilege-guard.py` uses `fcntl` flock and does a command-string match (not tool-name match). `hooks/lib/allowlist.py:read_grant` takes a `tool_name` and matches it literally/regex — semantically different from command-string matching. A direct drop-in replacement would change matching semantics.

3. **posttool-allowlist-consume.py**: does NOT use `hooks/lib/allowlist.py`. It imports only `from lib.subagent import is_subagent_context` (for the subagent exemption check) and implements its own independent grant-read and consume logic at lines 38-104: it reads the grant file directly with `fcntl.flock`, does its own compound-command splitting via `_split_bash_command`, matches pattern against subcommands and full command, and calls `os.unlink(grant_path)` on match. This makes posttool a FOURTH independent implementation of grant-read/match logic, not a consumer of `lib/allowlist.py`.

**Conclusion for this cycle**: Do NOT consolidate the bash heredoc — too much refactoring, too much regression risk to a security-critical path. DO align `_check_git_allowlist` in `pretool-git-privilege-guard.py` with `hooks/lib/allowlist.py` if the matching semantics can be unified; however, since `_check_git_allowlist` matches against the full command string (not just the tool name), the two functions serve different interfaces and cannot be drop-in replaced. Leave `_check_git_allowlist` as-is. The 4-implementation state (bash heredoc, Python guard, posttool independent impl, plus lib/allowlist.py itself) is documented in `out_of_scope_observations` as technical debt for a future dedicated consolidation cycle.

**Decision**: allow-6 is `Won't Have` for this cycle. No code change. Document the technical debt observation.

## Edge Cases & Risks

1. **Push scope regression (critical)**: If M2 is implemented incorrectly (allowlist check placed before `_looks_like_git_push` dispatch in the dispatcher), a broad `/allow git push` grant would bypass `_push_has_forbidden_ref_mutation` and allow force-push. Dev must verify that the central allowlist check is placed AFTER the `_looks_like_git_push` branch exits — or equivalently, that the check is only reached when `_looks_like_git_push` is false. See Technical Hints for the correct dispatcher structure.

2. **Early IS_SUBAGENT, late assignment of IS_SUBAGENT for the git-subagent block (allow-5)**: The outer `IS_SUBAGENT` at line 1068 is also used by the git history mutation block at lines 1069-1090. Moving IS_SUBAGENT to line ~17 means it is set earlier — this is safe because `$INPUT` is set at line 6 and unchanged throughout. The git-subagent block at line 1068 simply uses the already-set variable instead of re-computing it.

3. **Line 615 global short-circuit remains unconditional**: M3 removes the 6 secondary calls but must NOT touch line 615. The global short-circuit must remain the first command after the /do bypass block.

4. **`check_and_consume_allowlist` IS_SUBAGENT reference after M4**: after removing the inline parse, the function references `$IS_SUBAGENT` (outer script variable). This is valid in bash — a function can read script-global variables. The `local is_sub` variable is simply removed; the conditional `if [ "$is_sub" = "1" ]` becomes `if [ "$IS_SUBAGENT" = "1" ]`.

5. **Compound-command behavior change introduced by M2 `return` after push dispatch — OBJ-7 rebuttal**: The current dispatcher at `pretool-git-privilege-guard.py:657-670` has NO `return` after the `_evaluate_push(command, data)` call (line 666). M2 adds `return` after the push dispatch. QA raised OBJ-7: "the `return` causes `_evaluate_commit` to NEVER be called for compound commands like `git push origin main; git commit -m malicious`, skipping commit grant check and default-deny."

   **Evidence (measured — BA read hook source this session)**: The `command` variable entering `_evaluate_command` is set at `main()` line 684: `command = (data.get('tool_input', {}) or {}).get('command', '') or ''`. This is the raw bash string from the PreToolUse payload — the **full compound string**, not split. The Python git privilege guard performs NO compound-command splitting anywhere in the file. (Contrast: `pretool-bash-safety.sh` lines 50-57 explicitly splits on `&&`, `||`, `;` before matching — the Python guard does not have an equivalent step.)

   **What `_check_git_allowlist` actually receives** (measured — `_check_git_allowlist` at lines 141-177): the function takes `command: str` and performs `pattern in command` (literal) or `re.search(pattern, command)` (regex). For input `git push origin main; git commit -m "malicious"`, the full string is what the function sees — not a subcommand slice.

   **Why OBJ-7's model is incorrect**: OBJ-7 assumes per-function checks provide independent per-subcommand protection. They do not. For the compound string `git push origin main; git commit -m "malicious"`:
   - `_looks_like_git_push` at line 330 (`re.search(GIT_COMMAND_RE + r'push\b', command)`) — matches because "git push" appears in the full string
   - `_looks_like_git_commit` at line 323 (`re.search(GIT_COMMAND_RE + r'commit\b', command)`) — also matches because "git commit" appears in the full string
   - Under current code (no `return`): BOTH `_evaluate_push` and `_evaluate_commit` are called, each passing the same full string to `_check_git_allowlist`. The grant pattern (e.g. `"git push"`) matches the full string in the push check. The commit check then runs `_check_git_allowlist` with the same pattern against the same full string — the pattern `"git push"` also matches `"git push origin main; git commit -m 'malicious'"` in the commit check, because it is a substring of the same full string. The commit default-deny block is ALSO bypassed in the current code for this compound command.
   - Under M2 (with `return`): `_evaluate_commit` is skipped. But it would have been bypassed anyway by `_check_git_allowlist` — there is no independent per-subcommand commit check that the `return` destroys. The security outcome is identical.

   **Conclusion**: OBJ-7's concern is based on a wrong model of how `_check_git_allowlist` operates. It is not a per-subcommand parser; it matches against the full command string. The per-function calls in `_evaluate_commit` and `_evaluate_merge` provide no additional protection against compound commands containing both a push and a commit, because any grant pattern that matches the full string in the push call also matches the same full string in the commit call. M2 does not introduce a compound-command security regression. The `return` after push is correct.

   Note: this analysis also confirms that the `pretool-bash-safety.sh` allowlist (lines 50-57) IS more robust against compound-command smuggling because it splits before matching — the Python guard does not. This is a pre-existing architectural gap documented in `out_of_scope_observations` (allow-6 technical debt), not a regression introduced by M2.

### M2 — Correct `_evaluate_command` restructuring

Rejected approach (not used): placing the central `_check_git_allowlist` call after the `forbidden_plumbing`, `reset_hard`, and `direct_ref_mutation` dispatches leaves those three paths unguarded when a grant is present.

The correct structure checks `_looks_like_git_push` first (routing push to its own handler that manages its allowlist internally), then runs the central check for all remaining operations:

```python
def _evaluate_command(command, data):
    if _looks_like_git_push(command):
        # push handles allowlist internally, after absolute ban check
        _evaluate_push(command, data)
        return
    # Central /allow check for all non-push git operations.
    # _evaluate_push is excluded because its allowlist check must
    # come AFTER _push_has_forbidden_ref_mutation (force-push
    # must stay blocked even with a broad /allow grant).
    if _check_git_allowlist(command, data):
        return
    if _looks_like_git_forbidden_plumbing(command):
        _evaluate_forbidden_plumbing(command, data)
    if _looks_like_git_reset_hard(command):
        _evaluate_reset_hard(command, data)
    if _looks_like_git_direct_ref_mutation(command):
        _evaluate_direct_ref_mutation(command, data)
    if _looks_like_git_merge(command):
        _evaluate_merge(command, data)
    if _looks_like_git_commit(command):
        _evaluate_commit(command, data)
```

This is the correct structure. The dead `sid` removal (M1) is simply removing the first line of the current `_evaluate_command` body.
