## Goal

## Context

The dev-series commands (dev.md, dev-command.md, dev-overnight.md) each specify what the user should do after the command finishes: dev.md explicitly directs the user to `/close → /commit → /push`; dev-overnight.md directs to `commit.sh → /merge → /push`. The user wants: (1) the adaptation added to clean.md, and (2) an enumeration of all other commands with the same gap.

## Ending-Workflow Pattern: What dev.md Does (Research Finding)

`commands/dev.md` Step 15 (lines 1056-1073) implements the ending workflow as follows:

1. **Unfinished work path**: if any follow-up work remains → `/update` (continuation-spec mode) → next command is `/dev --spec <path>`. Do NOT hand unfinished work to /close or /commit.
2. **Complete work path**: `/update --temp` (compact temp note, not written into repo) → content includes Task ID, artifact paths, QA status, iteration count → suggested next command: `/close <task-id>`.
3. **Full chain**: `/dev` → `/close` → `/commit <task-id>` → `/push` (or `/merge` for overnight worktree branches).

`commands/dev-command.md` Step 14 (lines 900-914) has a weaker form: "Offer git commit (if requested)" with no explicit /close chain reference. This is a gap relative to dev.md.

**Why /close does NOT apply to /clean**: The /close command's normal path (close.md lines 159-165) requires these same-task artifacts: `ticket-<task-id>.md`, `context-<task-id>.json`, `dev-report-<task-id>.json`, `qa-report-<task-id>.json`, `completion-<task-id>.md` — all under a matching `<task-id>`. /clean produces none of these. /close would fail immediately with "no close-report for task". The /clean architecture intentionally bypasses this chain: the cleaner subagent commits inline and reports `git_status.cleanup_commit` in its execution report.

**What /clean should adapt**: After Step 18 (completion report), add a "Next Steps" guidance block equivalent to the chain pointer in dev.md Step 15. For /clean the correct chain is: satisfied → `/push` to publish the cleanup commit to remote. Unsatisfied → checkpoint rollback (already documented). There is no /close gate.

## Edge Cases & Risks

- **clean.md Step 17 already mentions checkpoint rollback** — the /push guidance must be added alongside it, not replace it. Both branches (satisfied → /push; unsatisfied → rollback) should be present.
- **doc-sync.md may not always produce a staged commit** — it modifies files via Edit/Write tool calls in the orchestrator context. The ending-workflow note should be conditional: "if changes were made, run `/push` or commit manually."
- **quick-prototype.md is intentionally quick** — heavy workflow guidance (full chain reference) would be over-engineering. A single line noting `/push` after a commit is sufficient.
- **dev-command.md gap is higher severity** — it's positioned as a dev-cycle command (full BA→dev→QA pipeline) but its ending workflow is weaker than dev.md. The user might expect identical behavior.
