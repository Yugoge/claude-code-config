## Implementation Analysis: What Is Duplicated vs. Legitimately Different

### Shared logic (duplicated — should move to lib)

All four implementations share:
1. Grant-file path computation: `/tmp/claude-bash-allowlist-<sid>.json`
2. File open + `fcntl.flock(LOCK_EX)` acquisition
3. `json.load(fh)` with exception → return False/None
4. `grant.get('pattern', '')` + type/emptiness check
5. `grant.get('is_regex', False)` branch
6. `re.search(pattern, target)` for regex case

### Legitimately different (must stay in callers)

| Concern | Where it stays |
|---|---|
| IS_SUBAGENT firewall (`data.get('agent_id')`) | `_check_git_allowlist` caller |
| IS_SUBAGENT firewall (`$IS_SUBAGENT`) | `check_and_consume_allowlist` bash caller |
| IS_SUBAGENT firewall (`is_subagent_context(data)`) | `posttool-allowlist-consume.py:46` |
| Literal match target = `tool_name` (exact OR substring) | `read_grant()` — no change |
| Literal match target = full `command` string (substring only) | new `read_grant_for_git_command()` |
| Subcommand split on `&&\|\|;\|` + SIGALRM timeout | new `match_grant_for_bash_command()` |
| `CONSUMED\t...\t...\t...` stdout protocol | bash heredoc shim (calls lib) |
| Approval JSON emission (`hookSpecificOutput`) | bash shell wrapper — no change |
| Atomic match + `os.unlink` under single flock | new `consume_grant_for_posttool()` |
| Session_id fallback to env var | posttool caller keeps `os.environ.get("CLAUDE_SESSION_ID")` |

### Matching semantics matrix (must be preserved exactly)

| Hook | Input | Literal match | Regex match | Compound split |
|---|---|---|---|---|
| lib/allowlist.py `read_grant` | `tool_name` | `pattern == tool_name or pattern in tool_name` | `re.search(pattern, tool_name)` | No |
| pretool-git-privilege-guard | full `command` | `pattern in command` | `re.search(pattern, command)` | No |
| pretool-bash-safety.sh heredoc | split subcommands | `pattern in subcmd` | `re.search(pattern, subcmd)` with SIGALRM | Yes (`&&\|\|;\|`) |
| posttool-allowlist-consume | split + full + tool_name | varies per branch (see lines 68-91) | `re.search(pattern, part)` | Yes (`&&\|\|;\|`) |

These differences are intentional and must be preserved. The library consolidates the **file I/O and lock** only, not the match semantics.

## Edge Cases & Risks

1. **SIGALRM portability**: SIGALRM-based regex timeout in `match_grant_for_bash_command` uses `signal.alarm()` which is UNIX-only. This is already the case in the bash heredoc; the library must preserve this, not abstract it away.

2. **NB flock retry in bash hook**: The bash heredoc uses `LOCK_EX | LOCK_NB` with 3x 100ms retry. The `read_grant_for_git_command` uses `LOCK_EX` (blocking). These are correct as-is for their contexts — the bash hook runs under a tighter latency budget; the git guard's blocking lock is fine. Preserve this difference.

3. **posttool session_id fallback**: `posttool-allowlist-consume.py:50` falls back to `os.environ.get("CLAUDE_SESSION_ID", "default")` when `data.get("session_id")` is absent. The `consume_grant_for_posttool` library function must accept a pre-resolved `sid` from the caller, not resolve it internally.

4. **posttool compound-split `||` handling**: `posttool-allowlist-consume.py:34` replaces `||` with `\n\n` before `|` with `\n`, producing different split tokens than the bash heredoc which replaces `||` with `\n` before `|` with `\n`. These are behaviorally equivalent for the purposes of subcommand iteration but the posttool must preserve its own split exactly.

5. **posttool tool_name fallback path**: Lines 82-84 in posttool handle `pattern == tool_name` as a fallback when no subcommand matches. This is NOT present in pretool-bash-safety.sh. The `consume_grant_for_posttool` function must preserve this path.

6. **Force-push ordering invariant** (Codex issue 9): In `pretool-git-privilege-guard.py`, `_push_has_forbidden_ref_mutation` runs at line 576 BEFORE `_check_git_allowlist` at line 583. This ordering must be preserved after M4. The library function must not be called before the force-push check.

7. **Backward compatibility of `read_grant`**: `pretool-orchestrator-gate.py`, `pretool-subagent-enforce.py`, and `pretool-aggregate-check.py` all call `read_grant(tool_name, sid)`. This signature must not change.

8. **`_load_and_match` `literal_policy` parameter**: `read_grant` uses `pattern == tool_name or pattern in tool_name`; `read_grant_for_git_command` uses `pattern in command` only. The private helper must parametrize this (e.g., `literal_policy: str` = `"exact_or_substr"` vs `"substr_only"`).

**Bash hook sys.path note**: The bash heredoc already runs with `PYTHON_BIN` pointing to the venv Python. `sys.path.insert(0, <hooks_dir>)` or equivalent is needed so `lib.allowlist` is importable. Look at how `posttool-allowlist-consume.py` does it: `sys.path.insert(0, str(Path(__file__).parent))`. In the heredoc, `__file__` is unavailable; use an env variable or a resolved path such as `os.path.dirname(os.path.abspath(os.environ.get('BASH_SOURCE', __file__)))` — or pass the hooks dir as an env var from the bash wrapper.

**Suggested implementation order**:
1. M2 first — write `_load_and_match` (internal) and update existing `read_grant` to use it (no external change, low risk)
2. M1 — write `read_grant_for_git_command` using `_load_and_match`
3. S1 — write `match_grant_for_bash_command` using `_load_and_match`
4. M3 — write `consume_grant_for_posttool` (atomic consume)
5. M4 — update `_check_git_allowlist` to call library

**Force-push invariant** (Codex issue 9, AC3): `_evaluate_push` in `pretool-git-privilege-guard.py` must call `_push_has_forbidden_ref_mutation` before `_check_git_allowlist`/`read_grant_for_git_command`. The force-push block must never be reachable via `/allow` grant. Current code at lines 574-584 has this ordering; M4 must preserve it.

**`match_grant_for_bash_command` return type**: Return `None` on no match / no grant, return a named tuple `MatchResult(pattern, is_regex, matched_sub)` on match. The bash shim converts this to the `CONSUMED\t<pattern>\t<is_regex>\t<matched_sub>` (or `MATCHED\t...`) protocol.

**`consume_grant_for_posttool` signature**: `consume_grant_for_posttool(sid: str, candidates: list[str], is_regex: bool = False) -> bool` — or accept raw grant data from caller. Simpler: `consume_grant_for_posttool(grant_path: Path, candidates: list[str]) -> bool` where posttool resolves the path. Either is acceptable; the key is that unlink happens inside the flock, not after.
