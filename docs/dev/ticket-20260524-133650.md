# BA Specification: /allow sentinel bypass for git ref-mutation guard

**Request ID**: 20260524-133650
**Created**: 2026-05-24T13:36:50Z
**Tier**: STANDARD

## Goal

When the user runs `/allow <git-command>`, the structured sentinel grant must be honored by ALL PreToolUse hook layers — including `pretool-git-privilege-guard.py`, which currently ignores sentinel grants and blocks the command anyway.

## Context

The `/allow` command writes two grants:
1. A legacy pattern-string grant at `/tmp/claude-bash-allowlist-<sid>.json`
2. A structured sentinel grant at `/tmp/claude-grants/<task_id>.json` with `allowed_operations[]`

`pretool-bash-safety.sh` checks the sentinel at lines 496–538 and exits 0 (`SENTINEL_OK`) before reaching any block rule at line 1015+. That file is already correct.

`pretool-git-privilege-guard.py` runs as a SEPARATE, PARALLEL PreToolUse hook (settings.json line 592). It calls `_check_git_allowlist()` (line 144), which only reads the LEGACY grant via `read_grant_for_git_command()`. It never calls `match_sentinel_grant_for_bash_command()`. So a sentinel that bash-safety.sh honors is invisible to the privilege guard, which then blocks regardless.

The privilege guard also has a subagent firewall at line 150 (`if data.get('agent_id'): return False`) that returns False for any agent with an `agent_id`, preventing even legacy grants from working for subagents. Per design decision M2 (task 20260521-090200), structured sentinel grants are user-created via `/allow` and must be honored even for subagents — this policy is already implemented in bash-safety.sh (lines 484–486) and must be mirrored in the privilege guard's sentinel check path.

## Edge Cases & Risks

1. **Sentinel TTL**: Sentinels expire after 300s (userprompt-consent-allowlist.sh line 202). If the sentinel expires between the `/allow` and the agent command, the guard correctly blocks. This is expected behavior — not a bug.
2. **Import placement**: If `match_sentinel_grant_for_bash_command` is imported inside the function body, it is re-imported on every call. Acceptable for a hook (not a hot path), but prefer module-level import for consistency.
3. **Subagent SID mismatch**: Subagents carry a different `session_id` than the orchestrator. `match_sentinel_grant_for_bash_command()` uses `task_id = CLAUDE_TASK_ID or sid`. If `CLAUDE_TASK_ID` is not set in the subagent environment, the fallback to `sid` may mismatch the sentinel written by the orchestrator's SID. Dev must verify that the task_id resolution path in the privilege guard matches the writer path. The existing bash-safety.sh implementation at lines 487–494 uses the same fallback logic.
4. **`git worktree remove --force`**: grep of bash-safety.sh and pretool-git-privilege-guard.py found NO pattern matching `worktree remove`. This command is not currently blocked. It appears in the user's requirement as an example of "should be allowable", not as "currently blocked and broken". No fix needed for worktree. Logged in Out-of-Scope Observations.
5. **Force-push special case**: `_evaluate_push()` checks `_push_has_forbidden_ref_mutation()` BEFORE calling `_check_git_allowlist()`. This ordering is intentional (line 610 before 617) and MUST NOT be altered. The fix only touches `_check_git_allowlist()`.

## Acceptance Criteria

### AC1: Sentinel grant bypasses privilege guard for ref-mutation commands

- GIVEN a valid (non-expired) sentinel grant at `/tmp/claude-grants/<task_id>.json` with `allowed_operations: [{"op": "git", "args_contain": ["branch -D foo"]}]`
- WHEN the privilege guard's `_check_git_allowlist("git branch -D foo", data)` is called with `data` containing no `agent_id` (main-agent context)
- THEN the function returns True and the command is allowed

Verifiable by: `pytest hooks/tests/` with a fixture that writes a sentinel grant and calls `_check_git_allowlist()` directly.

### AC2: Subagent sentinel bypass — subagent firewall does NOT apply to sentinel grants

- GIVEN a valid sentinel grant for `git branch -D foo`
- WHEN `_check_git_allowlist("git branch -D foo", {"agent_id": "some-subagent-id"})` is called
- THEN the function returns True (sentinel matched, subagent firewall skipped for sentinel path)

Verifiable by: pytest fixture with `agent_id` set.

### AC3: No sentinel — legacy behavior preserved

- GIVEN no sentinel grant file and no legacy grant file
- WHEN `_check_git_allowlist("git branch -D foo", data)` is called
- THEN the function returns False (existing behavior unchanged)

### AC4: Legacy grant still works for main-agent

- GIVEN a legacy grant file at `/tmp/claude-bash-allowlist-<sid>.json` with pattern `git branch -D foo`
- WHEN `_check_git_allowlist("git branch -D foo", data)` is called with no `agent_id`
- THEN the function returns True (legacy path unchanged)

### AC5: Subagent firewall preserved for legacy grant path

- GIVEN a legacy grant file only (no sentinel)
- WHEN `_check_git_allowlist("git branch -D foo", {"agent_id": "sub"})` is called
- THEN the function returns False (subagent firewall on legacy path unchanged)

    """Check /allow grant for non-push git operations. Read-only.

**After (proposed)**:
```python
def _check_git_allowlist(command: str, data: dict) -> bool:
    """Check /allow grant for non-push git operations. Read-only.

    Main-agent only for LEGACY grants. Sentinel grants (task 20260524-133650):
    extend to subagents — mirrors M2 decision in pretool-bash-safety.sh:484.
    IS_SUBAGENT check preserved for legacy path only.
    """
    sid = _get_session_id(data)
    if not sid:
        return False
    # Sentinel grant check (task 20260524-133650): NOT gated by subagent firewall.
    # User-granted sentinels must be honored in subagent context per M2 (20260521-090200).
    task_id = os.environ.get('CLAUDE_TASK_ID') or sid
    if match_sentinel_grant_for_bash_command(task_id, command) is not None:
        return True
    # Legacy grant check: subagent firewall preserved (original behavior).
    if data.get('agent_id'):
        return False
    return read_grant_for_git_command(command, sid)
```
