## Goal

Replace the bare task-id grep in `commands/commit.md` Step 7 stage-3 with a
machine-readable marker grep so that specs mentioning a task-id only in prose
are never false-positive continuation-spec candidates.

## Context

Step 7 stage-3 currently runs `grep -lF "${TASK_ID}" candidate.md` against
every spec file whose basename and mtime pass the stage-3 filter. Any spec that
cites the task-id in a cross-reference, historical note, or prior-cycle
description passes this predicate and becomes a candidate. In the
20260519-211515 cycle `spec-20260518-225715.md` and `spec-20260520-221059.md`
both matched because they contained prose references to `20260519-211515`,
triggering the fail-closed multi-candidate block (`exit non-zero, explicit
context.spec_path required`) even though neither file was the continuation spec
for that cycle.

The fail-closed policy itself is correct and must be preserved unchanged. Only
the candidate predicate is too broad.

Stage-4 outcome block has three branches (lines 160-162):
- Line 160: empty set — exit 0 (no action)
- Line 161: exactly one match — print `spec produced this cycle but not linked in context: <path>` and exit non-zero (fail-closed, single-candidate)
- Line 162: multiple matches — print `multiple specs produced this cycle without context linkage: <paths>; explicit context.spec_path required` and exit non-zero (fail-closed, multi-candidate)

Both fail-closed branches (161 and 162) are structurally affected by the predicate change: the predicate determines the cardinality of the candidate set, which in turn selects which branch fires. Both branches require verification ACs.

## Edge Cases & Risks

- **Stage 3 empty set for pre-existing specs**: after this change, any task-id
  that is not yet associated with a marker-bearing spec will produce an empty
  candidate set in stage 3. This is correct: stages 1 and 2 remain the primary
  resolution paths; stage 3 was always a fallback. The empty-set outcome
  (`print "No spec associated with task-id ${TASK_ID}" and exit 0`) is
  unchanged.
- **New BA/spec workflow must add marker**: future spec authors must place
  `<!-- spec-continuation-of: <task-id> -->` in newly created spec files for
  stage 3 to pick them up. This is intentional (machine-readable explicit
  intent). The orchestrator's prompt to BA should be updated in a future cycle
  to include this requirement — out of scope here.
- **grep -lF for literal string with `<` and `>`**: the marker contains angle
  brackets which are not special characters for `grep -lF`; no escaping needed.
- **Single-candidate branch (line 161) fires correctly**: when exactly one spec
  file contains the marker for a given task-id, the predicate returns count=1,
  which selects the line-161 branch (not line-162 multi-candidate). Both
  branches are fail-closed; only the error message differs.
